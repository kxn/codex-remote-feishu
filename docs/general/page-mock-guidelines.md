# 页面 Mock 生成规范

> Type: `general`
> Updated: `2026-04-24`
> Summary: 补充页面 Mock 到最终产品的落地一致性要求，约束最终实现除真实业务数据外必须与已确认 Mock 保持一致。

## 1. 适用范围

本规范适用于本仓库所有页面 Mock、页面原型、可交互 Demo、浏览器可运行预览页，包括但不限于：

- `docs/draft/*mock*.html`
- `web/src/**` 下的 mock / preview / prototype 页面
- `internal/app/daemon/adminui/**` 下用于预览或演示的页面
- setup / admin / onboarding / status / install 等用户可见页面的高保真原型

如果产物的目标是“让人直接在浏览器里体验页面”，就按本规范处理，不因为它仍被称为 mock 而降低要求。

## 2. 核心要求

页面 Mock 必须被当作“最终用户可直接体验的页面版本”来实现，而不是设计过程中的静态示意稿。

这意味着：

- 页面中只能出现最终用户真正关心的内容
- 页面必须能在浏览器里直接运行
- 页面必须可交互，不能只摆外观
- 可以使用假数据，但不能用“假交互”代替真实页面行为
- 如果某个页面后续要按 Mock 落成真实产品，则真实产品也必须遵守同一套用户可见边界

## 3. 用户可见内容规则

### 3.1 只展示最终用户关心的内容

页面默认只允许展示：

- 当前用户要做什么
- 当前状态如何
- 当前操作的可选项和结果
- 下一步去哪里

页面中不应默认展示：

- 设计动机
- 设计说明
- 架构意图
- 组件职责
- 占位说明
- “这里以后会接后端”
- “这里是 mock / demo / prototype / wireframe”

### 3.2 禁止暴露设计意图字样

以下内容禁止出现在用户可见页面中：

- `Mock`
- `Demo`
- `Prototype`
- `Wireframe`
- `设计目的`
- `设计说明`
- `内部备注`
- `占位`
- `TODO`

这条规则同样适用于：

- 浏览器标题
- 页面 header
- 卡片标题
- 空态说明
- 表单提示
- placeholder
- debug 面板
- toast / notice

如果这些说明仍然需要保留，应写到：

- issue
- 文档
- 注释
- skill / 设计说明文件

而不是渲染到页面里。

## 4. 数据规则

### 4.1 假数据允许，但必须完整覆盖交互

页面 Mock 可以使用假数据，但必须覆盖所有用户可交互的数据面：

- 可选择的数据
- 可编辑的数据
- 可筛选 / 排序 / 搜索的数据
- 会影响按钮、状态、提示或后续步骤的数据

### 4.2 只做“有数可点”的假数据，不做“无行为的占位数据”

允许使用本地状态、内存数据、固定 JSON 或前端 fake service。

但不允许：

- 页面上有列表，点任何项都没有后续效果
- 页面上有表单，提交只是无变化
- 页面上有分页、筛选、排序入口，但实际不生效
- 页面上有错误态、空态、完成态设计，却没有可达路径

如果真实后端尚未存在，必须通过前端本地状态把这些行为补齐，而不是留成空操作。

## 5. 交互规则

### 5.1 所有用户操作都必须是真的

页面上所有用户可执行的操作都必须产生真实页面行为，包括但不限于：

- 点击
- 输入
- 提交
- 切换 tab
- 展开 / 收起
- 打开 / 关闭弹层
- 筛选
- 搜索
- 排序
- 分页
- 向前 / 向后导航
- 多步流程切换
- 重新加载

“真的”指的是它必须驱动页面状态变化、流程推进、内容切换、验证反馈或导航结果，而不是纯视觉摆拍。

### 5.2 禁止静态页面冒充可交互原型

以下产物不满足页面 Mock 交付要求：

- 静态截图
- 只有排版没有交互的 HTML
- 所有按钮都是 dead button 的页面
- 只靠注释说明“这里以后会联动”的页面
- 只有一种默认态、没有任何可达变体的页面

## 6. 响应式与横竖屏要求

页面 Mock 的交互要求包含响应式与设备方向变化，不允许只做单一静态视口。

至少必须成立：

- 桌面端可用
- 移动端可用
- 浏览器宽度变化时布局和操作仍成立
- 横屏 / 竖屏切换后，主流程和主操作仍成立

如果页面在不同尺寸或方向下需要不同交互结构，例如：

- 侧栏变抽屉
- 双栏变单列
- 顶部操作变底部操作

那么这些行为也必须在 Mock 中真实实现，而不是只靠口头说明。

## 7. 允许的实现形式

页面 Mock 可以采用这些形式：

- 独立 HTML / CSS / JS 页面
- React / Next / Vite 页面
- 仓库现有 Web 应用中的临时预览路由
- admin UI 中可直接打开的预览页

前提始终是：

- 能在浏览器里运行
- 页面可交互
- 响应式成立
- 用户可见内容符合最终用户视角

## 8. 从 Mock 到最终产品的一致性要求

### 8.1 已确认 Mock 是最终产品的用户界面合同

如果某个页面已经存在被确认可用的页面 Mock，那么后续真实产品页面默认应把这个 Mock 视为用户界面合同。

这里的“合同”至少包括：

- 页面结构
- 页面信息层级
- 用户可见文案
- 状态切换
- 交互路径
- 响应式行为
- 横竖屏下的主要交互

### 8.2 允许变化的只有真实业务数据和技术接入层

从 Mock 落到真实产品时，允许替换的主要只有：

- 假数据替换为真实业务数据
- 本地状态替换为真实后端或真实服务接入
- 为实现需要增加的非用户可见代码、状态管理、埋点、容错和技术胶水

这些变化不应改变用户最终看到的页面合同。

### 8.3 禁止开发阶段擅自增加用户可见内容

如果已确认 Mock 中没有出现某段用户可见内容，那么真实产品实现阶段禁止自行新增：

- 新文案
- 新提示
- 新说明
- 新免责声明
- 新设计解释
- 新的用户可见状态标签
- 任何最终用户本不该看到的补充文字

特别禁止在真实产品里擅自加入：

- 设计说明
- 研发解释
- 调试提示
- 临时过渡文案
- “后端暂未完成”之类的实现中间态说明

### 8.4 如果要改用户可见合同，先改 Mock 或规范，再改产品

如果开发过程中发现 Mock 需要变化，正确顺序应是：

1. 先更新 Mock 或对应规范
2. 明确新的用户可见合同
3. 再按新的合同更新真实产品

不允许反过来：

- 先在真实产品里加文案或改交互
- 再事后说这是实现方便或临时补充

## 9. 交付检查表

交付前至少检查以下问题：

1. 页面里是否还有 `mock / demo / prototype / 设计说明 / TODO` 这类用户可见字样？
2. 这个页面是否可以在浏览器里直接运行，而不是只能看静态图？
3. 所有用户可见按钮、表单、列表项、筛选、导航是否都是真实可用的？
4. 假数据是否已经覆盖全部用户可交互的数据面？
5. 错误态、空态、完成态、切换态是否都是可达的，而不是只画出来？
6. 桌面、移动、宽度变化、横竖屏切换后，页面是否仍可完成主任务？
7. 如果真实产品是按现有 Mock 落地，除真实业务数据外，用户可见结构、交互和文案是否与 Mock 一致？
8. 是否出现了 Mock 中没有、且未经规范或 Mock 本身更新就被开发自行加入的用户可见文案？

如果以上任一问题回答为“否”，该页面 Mock 默认不应视为完成。
