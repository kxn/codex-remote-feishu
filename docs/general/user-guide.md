# 使用说明书

> Type: `general`
> Updated: `2026-04-30`
> Summary: 按当前 mode 术语基线重写 headless/vscode 与 `codex|claude|vscode` 的用户说明，保留 `/mode normal` 兼容说明。

## 1. 这是什么

`codex-remote-feishu` 用来把你正在使用的 Codex 工作现场带到飞书里。

它适合的不是“专门坐在电脑前高强度连续操作”的时段，而是这些更常见的场景：

- 出门以后，想在手机上继续刚才的工作
- 不方便开电脑时，想快速看一下 Codex 当前做到了哪一步
- 想直接在飞书里继续某个项目、补一句 steer、发一张截图、或者让它继续往下做

当前最推荐的使用方式其实很简单：

- 默认直接用 `codex` 模式
- 在飞书里按工作区和会话继续工作
- 只有你明确需要“跟着 VS Code 当前焦点走”时，再切到 `vscode` 模式

虽然 VS Code 联动看起来更酷，但真正长时间用下来，通常还是 headless 下的 `codex` 模式更稳、更顺手。

## 2. 为什么是这样设计

已经有不少“远程操控 AI”的工具了，但这个项目一开始想解决的问题并不是“把 Codex 在聊天软件里生搬硬套一份”。

它更在意的是两件事：

1. 尽量让碎片时间也能真正干活，而不是只能看一眼状态
2. 尽量利用飞书现成的交互能力，把常用动作做得足够顺手

所以它的默认方向一直是：

- 优先把工作区、会话、继续对话、发图、停止、切模式这些动作做顺
- 优先服务飞书里的真实使用体验
- 不为了兼容交互能力弱的入口而把主体验做差

## 3. 特色功能

### 3.1 在飞书里继续同一个工作现场

你可以在飞书里：

- 接管一个工作区
- 继续一个已有会话
- 在同一个工作区里直接新建会话
- 看最终回复、过程消息和状态提示
- 在需要时切到 VS Code 跟随模式

### 3.2 尽量保留原有上下文

这个系统的目标不是另起一个“聊天机器人上下文”，而是尽量继续你原本的 Codex 工作语义：

- 继续原来的工作目录
- 继续原来的会话
- 尽量保留原本的模型、推理强度和线程上下文

### 3.3 深度集成飞书交互

当前已经支持的飞书侧体验包括：

- 统一的工作区 / 会话选择卡
- 在选择卡里直接添加工作区
- 当前会话的历史查看卡
- 菜单命令和参数卡
- 最终回复直接回到触发它的原消息下方
- 老卡片、旧按钮、过期动作的明确提示
- 文档预览链接替换

### 3.4 过程信息和最终结果都更透明

现在飞书里不只是“看到一段最终答复”。

你还能看到：

- 计划更新卡
- 命令执行 / 工具调用 / Web 搜索等共享过程卡
- 最终答复底部的 token usage、thread usage 和 context left 摘要
- 长结果自动拆分后的 continuation 卡

### 3.5 多机器人并行使用

一台机器可以同时接入多个飞书机器人。

这在真实使用里非常有价值，因为你很容易天然地把“不同机器人”当成“不同工作上下文”。相比在一个机器人里来回切多个项目，多机器人并行通常更清楚。

### 3.6 图片暂存、文件发送与文档预览

- 图片不会因为你单独发出就立刻触发一轮请求
- 它会先暂存，等你下一条文字一起发给 Codex
- 也可以用 `/sendfile` 从当前工作区里挑一个文件，直接发回飞书聊天
- 如果开通了飞书云空间相关权限，最终回复里的本地 `.md` 和单文件 `.html` 链接还能自动替换成飞书可预览链接

### 3.7 可以直接在飞书里配置定时任务

如果你给飞书应用开通了 `bitable:app`，就可以在飞书里直接用 `/cron` 管理当前 daemon 实例的后台任务。

它适合：

- 定时跑某个本地工作区
- 定时从 Git 仓库拉一个临时目录再执行
- 在飞书里直接看任务配置、运行记录和最近一次 reload 结果

## 4. 安装前准备

开始前建议先确认这些事情：

1. 目标机器上的真实 `codex` 已经能正常运行
2. 至少已经实际用过一次 Codex，最好已经有过可恢复的工作区或会话
3. 如果你想接管 VS Code，会用到的那台机器上也已经安装了 VS Code 的 ChatGPT / Codex 扩展，并且也至少说过一次话
4. 如果你打算用文档预览，飞书应用需要额外开通 `drive:drive`

飞书应用模板和权限说明可参考：

- [deploy/feishu/README.md](../../deploy/feishu/README.md)
- [deploy/feishu/app-template.json](../../deploy/feishu/app-template.json)

## 5. 安装

### 5.1 安装到哪台机器

先记住一个最容易混淆的规则：

- 这个工具要装在**真正运行 Codex 的那台机器**上
- 如果你用的是 VS Code Remote SSH，那么实际跑 Codex 的通常是远端 SSH 机器，不是你手边的本地电脑

所以：

- 普通本机使用，就装在你的工作机上
- VS Code Remote SSH 使用，就装在那台 SSH 目标机器上
- 如果两边都会各自运行 Codex，两边都可以各装一套，但它们是两套独立环境

### 5.2 一条命令安装

macOS / Linux：

```bash
curl -fsSL https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.sh | bash
```

Windows PowerShell：

```powershell
irm https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.ps1 | iex
```

安装器会自动：

1. 识别当前平台
2. 下载对应 release 包
3. 安装 `codex-remote`
4. 启动本地后台服务
5. 打开或打印 WebSetup 地址

如果你要装某个 prerelease track，例如 `beta`：

```bash
curl -fsSL https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.sh | bash -s -- --track beta
```

```powershell
& ([scriptblock]::Create((irm https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.ps1))) -Track beta
```

如果你要固定某个版本：

```bash
curl -fsSL https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.sh | bash -s -- --version v1.0.0
```

```powershell
& ([scriptblock]::Create((irm https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.ps1))) -Version <version>
```

### 5.3 手动安装 release 包

macOS / Linux：

```bash
./codex-remote install -bootstrap-only -start-daemon
```

Windows PowerShell：

```powershell
.\codex-remote.exe install -bootstrap-only -start-daemon
```

安装完成后，打开输出里的 `/setup` 地址继续初始化。

## 6. 首次配置

首次配置主要都在 WebSetup 完成。

推荐顺序：

1. 先配置飞书应用
2. 再看运行环境检查
3. 按需处理自动启动
4. 最后再决定是否接入 VS Code

### 6.1 配置飞书机器人

当前有两条入口：

1. 扫码新建飞书应用
2. 接入已有飞书应用

默认更推荐第一条，因为它会自动帮你处理掉绝大部分容易出错的东西。

扫码新建路径的优点是：

- 会自动完成大部分基础配置
- 通常能省掉很多权限、回调、事件订阅的手工排查

但有两件事仍然建议你额外关注：

- 如果你想让最终回复里的本地文档自动变成飞书可预览链接，通常还要补开 `drive:drive`
- 机器人菜单一般还是建议按自己的使用习惯手动配一下

如果你经常在手机上使用，至少建议配这两个菜单：

- `menu`
- `stop`

常用以后，再按需加这些：

- `steerall`
- `new`
- `model`
- `reasoning`
- `access`

如果你走的是“已有应用”路径，就按 WebSetup 页面的提示补齐权限、事件订阅、回调和发布步骤。

### 6.2 运行环境检查

安装完成后，页面会先检查当前机器的基础运行条件。

这一步主要会告诉你：

- 当前 daemon 会用哪个 `codex-remote` 作为 headless 启动器
- wrapper 实际会调用哪个真实 `codex`
- 这些路径在当前环境里是否真的可用
- 有没有明显风险，比如只靠 `PATH` 找到、或者错误地回指到 `codex-remote` 自己

注意它检查的是“能不能启动起来”，不负责帮你验证 Codex 登录状态、账号配置或 provider 凭据。

### 6.3 自动启动

当前最成熟的是 Linux 下的 `systemd --user` 路径。

这一步不是必须的，但如果你希望机器重启后自己恢复服务，会比较有用。

### 6.4 VS Code 集成

只有在你确实想接管 VS Code 当前对话时，才需要做这一步。

有一个非常重要的操作顺序：

- 先退出 VS Code
- 再在页面里执行接入 / 迁移 / 重装动作
- 完成后重新打开 VS Code

如果你升级了 VS Code 扩展，页面也可能提示重新安装扩展入口。按提示处理即可。

## 7. 开始工作

先记住两个概念：

- `工作区`
  - 你当前工作的目录
- `会话`
  - 你在 Codex 里能继续恢复的那条对话

当前可以先按两层来理解：

- 运行形态
  - `headless`
  - `vscode`
- 用户可见 mode
  - `codex`
  - `claude`
  - `vscode`

其中：

- `codex`
  - 默认也是最推荐的 headless mode
- `claude`
  - 也是 headless，只是 backend 换成 Claude
- `vscode`
  - 专门给“飞书跟随 VS Code 当前焦点”这个场景准备

切换方式：

```text
/mode codex
/mode claude
/mode vscode
/mode normal
```

`/mode normal` 仍兼容，但它现在等价于 `/mode codex`。

### 7.1 Headless 模式怎么用

在 headless 下，最常见的入口是：

```text
/list
```

现在 `/list`、`/use`、`/useall` 都已经收敛成同一套目标选择卡：

- 上面选工作区
- 下面选会话
- 最后点击确认

也就是说，你现在可以直接在同一张卡里完成这些事：

- 切到某个工作区
- 继续那个工作区里的已有会话
- 直接选择 `新建会话`
- 直接切到“添加工作区”

如果你点的是“添加工作区”，当前支持两种常见来源：

- 选择一个已经存在的本地目录
- 填 Git 仓库地址，把仓库拉到本地后接成新工作区

更实用的理解方式是：

- `/list`
  - 更偏向“我要换工作区 / 找工作目录”
- `/use`
  - 更偏向“我要快速继续最近会话”
- `/useall`
  - 更偏向“我要把所有可见会话都翻出来找”

如果你只是想继续当前项目，通常这样就够了：

1. `/list`
2. 选中工作区
3. 选中会话，或者直接选 `新建会话`
4. 开始说话

如果你已经在某个工作区里，也可以直接：

```text
/new
```

它会把当前工作区切到“准备新会话”的状态，下一条普通文本就会真正创建并进入新会话。

如果你已经接管了某个会话，也可以直接：

```text
/history
```

它会打开当前输入目标 thread 的历史卡：

- 列表页按最新一轮在前分页展示
- 可以在卡片里直接选某一轮看详情
- 详情页可以继续看前后相邻的轮次，再返回列表

如果当前还没有真正进入某个会话，例如你只是刚接管工作区、还没开始第一轮，它会明确提示你先 `/use` 或直接发一条文本开始新会话。

### 7.2 VS Code 模式怎么用

如果你想让飞书接管 VS Code 当前正在做的活：

```text
/mode vscode
/list
```

这时 `/list` 列出来的是当前在线的 VS Code 实例，而不是普通工作区。

接管以后，推荐做法通常是：

- 回到 VS Code 那边继续说一句话，或者切到你正在用的对话
- 系统会自动观察并跟过去
- 然后你再回飞书继续操作

如果你想手动回到“跟随当前 VS Code 焦点”的状态，可以用：

```text
/follow
```

如果你只是想在当前已接管的 VS Code 实例里手动挑一个会话，也可以继续使用 `/use` 或 `/useall`。

## 8. 日常使用技巧

### 8.1 运行过程中还能继续发消息

即使机器人正在执行上一条请求，你仍然可以继续发消息。

这些消息会先进入队列，当前一轮结束后再继续发送给 Codex。

### 8.2 点赞排队消息，或直接 reply，都可以用来 steer

如果某条排队中的文字消息，你不想等它排到后面再单独开新一轮，而是想让它并入当前正在执行的过程：

- 给这条消息点 `ThumbsUp`
- 系统会尝试把它升级成当前执行中的 follow-up / steer
- 成功后，机器人也会给这条消息点一个赞作为确认

这对于“实时纠偏”特别有用。

如果你是直接回复当前正在执行的那条用户消息，也可以这样做：

- 直接 reply 那条消息，补一句文本说明，或发图再补文字
- 系统会优先把这条 reply 当成对当前 running turn 的 steer
- 如果当前不满足 steer 条件，才会回退到普通排队语义

如果你已经连续发了好几条排队补充，也可以直接用：

```text
/steerall
```

它会把当前队列里可并入本轮执行的输入一次性并进去。

### 8.3 手机发图片也不用担心会立刻触发

在手机端飞书里，图片通常会直接发送出去，没法像桌面端那样和文字自然合成一条消息。

但这里不用担心：

- 图片会先暂存
- 你可以连续发多张
- 等你下一条文字发出时，这些图片会和文字一起发送给 Codex

### 8.4 `/sendfile` 可以把当前工作区里的文件直接发回飞书

如果你想把本地结果文件、截图、日志或构建产物直接发回当前聊天，不一定非要自己去系统文件管理器里找。

现在可以直接发：

```text
/sendfile
```

然后在当前工作区里选文件。

更适合的场景通常是：

- 想把某个生成文件直接发到当前群聊或单聊
- 想在手机上快速转发一个工作区里的现成文件
- 你已经接管了目标工作区，不想再离开当前上下文去手工上传

### 8.5 过程卡和结果 footer 能帮你更快判断当前进展

从 1.6 开始，飞书里会尽量把过程信息整理成更容易扫读的结构，而不只是不断往下刷文本。

常见可见内容包括：

- 计划更新
- 命令执行摘要
- 工具调用 / Web 搜索 / 探索过程
- 最终答复底部的 token usage、thread usage、context left

如果你想少看一点中间过程，可以发送：

```text
/verbose quiet
```

如果你保持默认的 `normal`，会看到计划、最终答复，以及会影响当前状态的共享过程项，例如：

- 文件修改
- 上下文压缩
- MCP 调用

`normal` 当前不会显示 reasoning 的 `思考中...` 占位。

如果你想看更完整的共享进行中卡，可以发送：

```text
/verbose verbose
```

这时仍会显示执行、探索、工具调用等共享过程，但 reasoning 进行中只会显示尾部占位 `思考中...`。

如果你想把完整 reasoning / thinking 明细也一起打开，可以发送：

```text
/verbose chatty
```

### 8.6 文档预览会更方便

如果飞书应用开通了 `drive:drive`，最终回复里的本地文档链接会尽量自动替换成飞书可预览链接。

当前重点支持：

- `.md`
- 单文件 `.html`

这样你在飞书里一点就能直接看结果，不用再回机器上找文件。

### 8.7 `/cron` 可以把常见后台任务收敛到飞书里

如果你希望某些工作定时执行，而不是每次都手动发命令，可以直接在飞书里发：

```text
/cron
```

这会先打开 Cron 根页。常见流程是：

1. `/cron edit`
2. 打开并编辑任务配置表
3. 按需配置本地工作区任务，或 Git 仓库来源任务
4. `/cron reload`
5. 用 `/cron status` 或 `/cron list` 看是否已经生效

### 8.8 多机器人通常比一个机器人来回切更舒服

虽然系统支持在一个机器人里来回切不同工作区和会话，但长期使用时，更推荐开多个机器人并行用。

原因很简单：

- 人会天然把“不同机器人”理解成“不同上下文”
- 而在一个机器人里频繁切多个项目，人脑很容易乱

所以比较推荐的方式是：

- 一个机器人盯一个主要项目或一类工作
- 在 Web 管理页里按需新增多个飞书应用

这里还要特别注意一个限制：

- 当前不允许多个机器人共享同一个工作区

这是故意设计成这样的，因为历史经验表明：

- 两个 Codex 同时在同一工作目录、尤其还可能碰到同一会话时，非常容易把上下文搞乱
- VS Code 模式下也很难完全控制另一边是不是碰到了同一会话

所以最安全的做法就是：

- 一个工作区同一时刻只交给一个机器人 / surface
- 也不要在机器人正在工作的同一目录上，手动再起一个容易碰到同会话的 Codex CLI 去搅在一起

### 8.9 机器人菜单支持带参数的快捷入口

所有 slash 命令本质上都可以映射成机器人菜单项。

如果某个命令需要参数，可以用 `_` 把参数直接拼进菜单 key，例如：

- `/model gpt-5.4`
  - 可以写成 `model_gpt-5.4`

如果你只配置了不带参数的按钮，例如 `model`，点击后系统会再给你一张后续卡片，让你选择或手动输入参数。

## 9. 常用命令

如果忘了命令，先用：

```text
/help
/menu
```

常用命令可以先记这些：

- `/list`
  - 查看并选择当前可接管的目标
  - headless 下主要是工作区
  - `vscode` 模式下主要是在线 VS Code 实例
- `/use`
  - 快速继续最近可见会话
- `/useall`
  - 查看全部可见会话
- `/new`
  - 在当前工作区准备一个新会话
- `/history`
  - 查看当前会话最近历史，并在卡片里继续翻页或看单轮详情
- `/status`
  - 查看当前状态、队列、模式、模型配置和当前分支等摘要
- `/stop`
  - 中断当前 turn，并清掉尚未发出的飞书队列
- `/compact`
  - 对当前已绑定 thread 主动发起一次手动上下文整理
- `/steerall`
  - 把当前队列里可并入本轮执行的输入一次性并入当前 running turn
- `/detach`
  - 断开当前接管
- `/mode vscode`
  - 切到 VS Code 跟随模式
- `/mode codex`
  - 切回默认的 headless Codex 模式
- `/mode claude`
  - 切到 headless Claude 模式
- `/mode normal`
  - 兼容旧写法，等价于 `/mode codex`
- `/follow`
  - 在 `vscode` 模式下重新跟随当前 VS Code 焦点
- `/sendfile`
  - 打开文件选择卡，从当前工作区挑一个文件直接发到当前聊天
- `/cron`
  - 打开当前实例的 Cron 根页；状态、任务列表和配置入口都在子页里
- `/cron reload`
  - 重新加载当前 Cron 配置表里的任务
- `/upgrade latest`
  - 检查并升级到当前 track 的最新版本
- `/upgrade track`
  - 查看或切换当前升级渠道
- `/verbose`
  - 控制飞书前端显示过程消息的详细程度
- `/autowhip on`
  - 打开自动鞭打；`off` 关闭

## 10. 关于自动鞭打

这是一个偏娱乐化但确实有人会用的功能：

```text
/autowhip on
/autowhip off
```

打开后，如果系统判断 Codex 中途有点像是在找理由停下，它会继续推它往下做，直到它自己判断事情确实做完了为止。

这个功能更适合当作辅助工具来看，不建议把它理解成“绝对可靠的强控制开关”。如果你准备开着长期用，先自己熟悉一下它的风格会更稳妥。

## 11. 需要记住的几个注意点

- 旧卡片、旧按钮、旧菜单动作如果已经过期，会收到明确提示；这时候直接重发对应命令就行
- `测试连接` 只说明你当前保存的飞书凭据是可用的，不代表旧聊天窗口会自动切换到新机器人身份
- 如果你把 `App ID` 换成了另一个飞书应用，需要从新机器人的聊天入口重新开始会话
- 如果你平时就是普通远程使用，优先坚持 headless 下的 `codex` 模式，通常体验最稳定
