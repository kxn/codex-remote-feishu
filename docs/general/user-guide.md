# 使用说明书

> Type: `general`
> Updated: `2026-04-09`
> Summary: 同步 VS Code 场景化接入路径，并保留 release track、命令帮助、旧卡片、后台恢复与文档预览 fixed-root inventory 的当前用户语义。

## 1. 这是什么

`codex-remote-feishu` 用来把你正在使用的 Codex 工作现场带到飞书里。

它的核心价值不是“在飞书里另外开一个机器人”，而是：

- 继续使用同一台机器上的 Codex instance
- 继续使用已有对话
- 尽量保留原本的工作目录、模型和线程语义

如果你平时主要在 VS Code 里工作，但经常需要在飞书里继续发指令、看结果、发图片或临时接管会话，这个工具就是为这个场景设计的。

## 2. 适合哪些场景

- 你在 VS Code 里开着 Codex，但离开电脑后想继续从飞书接着做
- 你想在飞书里切到某个已有对话，继续当前项目工作
- 你想在飞书里跟随 VS Code 当前正在看的对话
- 你想把截图或参考图片先发到飞书，再和下一条文字一起交给 Codex
- 你希望把最终回复里的本地 Markdown 文档直接变成飞书可预览链接
- 你需要一台机器同时接入多个飞书机器人，给不同团队或不同入口使用

## 3. 主要功能

### 3.1 在飞书里继续同一个工作现场

你可以在飞书里：

- 查看当前在线 VS Code 实例
- 显式接管某个实例
- 从最近会话列表直接继续已有对话
- 切换到某个已有对话
- 跟随 VS Code 当前对话
- 直接准备一个新对话

### 3.2 飞书和 VS Code 共用一套上下文

这个系统优先保留原本的工作语义：

- 当前对话仍然是 Codex 原生对话
- 工作目录沿用原对话可继承的目录
- 飞书侧的设置覆盖只影响飞书发出的后续消息，不会改掉已经排队的请求

### 3.3 图片暂存

在飞书里发图片时，图片不会单独触发一次请求。

当前行为是：

- 图片先暂存
- 下一条文字消息发出时，图片会和这条文字一起发送给 Codex

这很适合“先发截图，再补一句说明”的工作方式。

### 3.4 文档预览

如果最终回复里包含本地 `.md` Markdown 链接，系统可以在发送前把它上传到飞书云空间，并把链接替换成可预览地址。

这样你在飞书里直接点开就能看文档，不需要自己再去机器上找文件。

### 3.5 多机器人管理

一台机器可以配置多个飞书机器人。

你可以在本地管理页里：

- 新增机器人
- 修改名称和凭据
- 启用或停用
- 重连
- 继续完成首次配置

## 4. 开始前准备

开始前建议先确认：

1. 目标机器上的真实 `codex` 已经能正常运行
2. 已安装 VS Code 的 ChatGPT / Codex 扩展
3. 已准备飞书自建应用的 `App ID` 和 `App Secret`
4. 如果你要用文档预览，飞书应用还需要开通 `drive:drive`

飞书应用模板和权限说明可参考：

- [deploy/feishu/README.md](../../deploy/feishu/README.md)
- [deploy/feishu/app-template.json](../../deploy/feishu/app-template.json)

## 5. 安装

### 5.1 macOS / Linux 一条命令安装

```bash
curl -fsSL https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.sh | bash
```

这条命令会：

1. 下载适合当前平台的 release 包
2. 安装 `codex-remote`
3. 启动本地后台服务
4. 打印 WebSetup 地址

如果你想安装某个 prerelease track 的最新版本，例如 `beta`：

```bash
curl -fsSL https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.sh | bash -s -- --track beta
```

如果你想固定到某个版本：

```bash
curl -fsSL https://raw.githubusercontent.com/kxn/codex-remote-feishu/master/install-release.sh | bash -s -- --version v1.0.0
```

### 5.2 手动安装 release 包

macOS / Linux：

```bash
./codex-remote install -bootstrap-only -start-daemon
```

Windows PowerShell：

```powershell
.\codex-remote.exe install -bootstrap-only -start-daemon
```

安装完成后，打开命令输出里的 `/setup` 地址。

## 6. 首次配置

首次配置主要在 WebSetup 完成。

推荐顺序：

1. 添加飞书机器人名称、`App ID` 和 `App Secret`
2. 测试连接
3. 按页面指引完成飞书平台里的权限、事件订阅、菜单和发布
4. 让页面检测当前 VS Code 环境
5. 按你的实际使用场景完成 VS Code 接入

当前 setup 不再直接让你面对 `editor_settings` / `managed_shim` 这些内部模式名，而是先按场景引导：

- 如果当前就是远程 SSH 机器：
  - 页面只会提供“在这台远程机器上启用 VS Code”
  - 这条路径只接管扩展入口，不去写 host 机器的 `settings.json`
- 如果当前不是 SSH 机器：
  - 页面会先问你以后主要怎么使用 VS Code 里的 Codex
  - `只在这台机器本地使用`
    - 会写当前机器的 `settings.json`
  - `主要去别的 SSH 机器上使用`
    - 当前机器先跳过，不做接入
    - 等你在目标 SSH 机器上安装 `codex-remote` 后，再去那台机器完成这一步
  - `这台机器本地要用，也会 SSH 到别的机器`
    - 当前机器只接管扩展入口
    - 不写当前机器的 `settings.json`，避免 host 设置影响后续 Remote SSH

如果你后续升级了 VS Code 扩展，页面可能会提示重新安装扩展入口。按提示处理即可。

管理页里的 VS Code 区域也沿用同一套场景心智；只有在 `高级处理` 里，才会再看到更底层的接入方式。

首次在飞书里验证时，建议先发一次 `/help` 或 `menu`，先确认当前可用命令和快捷入口。

## 7. 日常使用

如果你一时忘了命令名，先发 `/help` 或 `menu` 看当前帮助，再决定是 `/list`、`/use` 还是 `/new`。

### 7.1 接管 VS Code 实例

在飞书里发送：

```text
/list
```

机器人会列出当前可手工接管的在线 VS Code 实例。

当前主交互是卡片按钮：

- 点击卡片里的“接管”按钮
- 如果你看到的是旧卡片，重新发送 `/list`

如果你只是想继续某个已有会话，而不关心先 attach 哪个实例，通常直接用 `/use` 更快。

### 7.2 选择当前对话

可以发送：

```text
/threads
```

或者：

```text
/use
```

`/threads` 和 `/use` 当前是同一条主入口，都会列出最近可见的会话，而不只限于当前实例。

这意味着：

- 如果你已经接管了某个实例，可以从这里切到别的会话
- 如果你当前还没有显式接管实例，也可以直接从这里继续已有会话
- 选择会话后，系统会自动把后续文字路由到目标会话
- 如有需要，系统会自动接管对应在线实例，或在后台恢复这个会话

当前主交互同样是卡片按钮；如果你看到旧卡片，重新发送 `/use` 或 `/useall`。

如果你想切到“查看所有可见对话”，也可以使用：

```text
/useall
```

`/useall` 会展示全部可见会话；默认的 `/use` / `/threads` 只展示最近一部分。

### 7.3 跟随 VS Code 当前对话

如果你和 VS Code 共用同一个实例，希望飞书始终跟着 VS Code 当前正在看的对话走，可以发送：

```text
/follow
```

这时飞书不会固定绑死某一个对话，而是尽量跟随 VS Code 当前活动对话。

如果当前还没有可跟随的对话，飞书会提示你等待，或者改用 `/use` 手动指定。

### 7.4 新建一个对话

如果当前已经绑定到一个可继承工作目录的对话，可以发送：

```text
/new
```

这不会立刻发出第一条请求，而是进入“新建对话待命”状态：

- 下一条文本会创建一个新对话
- 新对话会继承当前对话的工作目录

如果当前没有已接管的对话，或者当前对话没有可继承的工作目录，`/new` 会被拒绝。

### 7.5 发送文字和图片

普通文字会按当前目标对话进入队列。

图片的规则是：

1. 先发图片
2. 图片进入暂存
3. 下一条文字发送时，图片和文字一起交给 Codex

如果你发了图片但又改变了目标对话，例如切换 `/use`、执行 `/follow` 或 `/new`，这些尚未绑定到文字的图片会被丢弃。

如果有一条文字还在排队，而当前已经有一条回复在执行，你也可以给这条排队文字点 `ThumbsUp`，把它升级成对当前执行回复的跟进或纠偏。成功后，这条文字和它绑定的图片会一起被带过去。

### 7.6 查看当前状态

发送：

```text
/status
```

可以看到当前接管状态、目标对话、队列情况以及当前生效的模型设置。

### 7.7 停止当前请求

发送：

```text
/stop
```

会执行三件事：

1. 尝试中断当前正在运行的请求
2. 清空还没发出的排队输入
3. 丢弃尚未绑定到文字的图片

### 7.8 断开接管

发送：

```text
/detach
```

会断开当前实例接管，并放弃当前 surface 上尚未发送的队列与图片暂存。

如果当前还在进行后台恢复流程，`/detach` 也会一并取消这次恢复。

## 8. 飞书侧临时设置

你可以只对“之后从飞书发出的消息”临时覆盖参数。

### 8.1 模型

```text
/model
/model gpt-5.4
/model clear
```

### 8.2 推理强度

```text
/reasoning
/reasoning high
/effort xhigh
/reasoning clear
```

### 8.3 执行权限

```text
/access
/access confirm
/access full
/access clear
```

`/approval` 是 `/access` 的别名。

这些覆盖只影响飞书侧后续消息，不会回头改掉已经进入队列的请求。

## 9. 文档预览

文档预览只处理一种情况：

- assistant 最终回复里的本地 `.md` Markdown 链接

例如正文里出现：

```md
[README](docs/README.md)
```

系统会尝试：

1. 读取本地文件
2. 上传到飞书云空间
3. 把链接替换成飞书可打开的预览地址

几点注意：

- 不会改写普通纯文本路径
- 不会改写代码块里的路径
- 预览失败不会阻塞主回复
- 不开 `drive:drive` 时，聊天功能仍可正常使用，只是不会替换成预览链接

默认预览目录是：

- `Codex Remote Previews/`
- 管理页里的文档预览清理只会作用于这个固定目录内的内容

权限规则：

- 单聊会给当前用户授权
- 群聊会给当前用户和当前群授权

## 10. 多机器人和管理页

完成首次配置后，后续主要在本地管理页进行日常管理。

管理页适合做这些事：

- 查看当前是否有机器人异常
- 新增第二个或更多机器人
- 修改机器人名称和凭据
- 暂时停用某个机器人
- 查看工作实例
- 清理图片暂存和文档预览
- 检查 VS Code 接入是否还正常

如果你只是想继续工作，平时主要还是在飞书里使用；管理页更像“本机后台的控制台”。

## 11. 使用细节与建议

### 11.1 队列是按飞书会话分别维护的

每个飞书聊天都有自己独立的队列。

这意味着：

- 你在 A 聊天里排的消息，不会跑到 B 聊天里
- 已经排队的消息会保留它入队时的目标对话和设置
- 你后面再切换对话，只影响之后的新消息

### 11.2 本地 VS Code 正在工作时，飞书可能会等待

如果本地 VS Code 已经先发起交互，飞书侧可能会先暂停自动发送，等本地这轮工作结束后再继续。

这是为了减少两个入口同时争用同一个工作现场造成的混乱。

### 11.3 撤回消息可以取消尚未发送的输入

如果你的飞书应用已经订阅了消息撤回事件，那么：

- 撤回尚未发送的文本，可以取消这条输入
- 撤回尚未绑定的图片，可以取消这张图片

如果没订阅这个事件，基础聊天仍然能用，只是撤回不会等价于“取消发送”。

### 11.4 待确认卡片优先级更高

如果当前有确认卡片还没处理：

- 普通文字不会继续入队
- 图片也不会继续暂存

你需要先在卡片上做决定，或者选择“告诉 Codex 怎么改”。

### 11.5 最终回复会回在原始消息下面

当前最终回复默认会直接回在你触发它的那条消息下方。

这带来的好处是：

- 在群聊里更容易看懂是谁发起了这条请求
- 同一段上下文会更自然地聚在一起
- 多个人同时看群聊时，不容易把结果看串

### 11.6 旧卡片过期时，直接重发命令

如果 daemon 刚重启过，或者你点到的是很早之前留下来的旧卡片，系统可能会直接提示这是一张过期卡片或旧动作。

这时不要继续在旧卡片上尝试操作，直接重发对应命令即可，例如：

- `/list`
- `/use`
- `/useall`
- `menu`

### 11.7 `/new` 最适合在旧上下文已经不需要时使用

`/new` 的目标就是快速清空远端上下文，但保留项目目录。

适合这些情况：

- 当前任务已经做完，准备开始下一件事
- 你想保留项目目录，但不想继续带着旧对话历史
- 你希望从飞书里快速进入一个干净的新对话

## 12. 常见问题

### 12.1 `/follow` 后没有立刻跟上

先确认：

1. VS Code 端是否真的已经接入本机 relay
2. 当前实例里是否存在可观察到的活动对话

如果暂时没有，等待即可；或者直接用 `/use` 手动指定。

### 12.2 `/new` 不能用

通常是以下原因之一：

- 还没接管任何实例
- 还没绑定任何对话
- 当前对话没有可继承的工作目录
- 当前有请求正在运行，或者有确认卡片还没处理

### 12.3 文档预览没生效

先检查：

1. 飞书应用是否开了 `drive:drive`
2. 链接是不是 assistant 最终回复里的 `.md` Markdown 链接
3. 本地文件路径是否在当前工作目录可访问范围内

### 12.4 飞书里看不到实例

先去本地管理页确认：

- 服务是否已启动
- VS Code 是否已经正确接入
- 目标实例是否在线

### 12.5 系统提示“旧卡片已过期”

这通常表示你点到的是 daemon 上一个生命周期留下来的旧卡片，或者一张已经失效的历史按钮。

处理方式很简单：

1. 不要继续点这张旧卡片
2. 重新发送对应命令
3. 在新卡片上继续操作
