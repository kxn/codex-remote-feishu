# 飞书业务卡片交互设计原则

> Type: `general`
> Updated: `2026-04-18`
> Summary: 从 2026-04 的菜单 / slash command / picker 交互审计中提炼出稳定的业务卡片设计原则，统一 owner card、后台例外、取消语义与封卡规则。

## 1. 文档定位

这份文档不是单个功能的设计稿，也不是某一次重构的执行计划。

它记录的是当前仓库在飞书业务卡片交互上的通用产品原则，用来回答下面这些长期问题：

- 一条业务流程应该由几张卡承接
- 什么时候应该继续 patch 原卡
- 什么时候允许切到别的卡
- 取消、返回、执行中、完成态应该如何分层
- 在飞书卡片空间有限的前提下，如何控制信息密度和主关注点

它是后续各类业务卡片设计与重构的共同基线。

## 2. 核心目标

飞书卡片交互的核心目标不是“尽量多复用一张卡”，也不是“每一步都拆成一张新卡”，而是：

**让用户在任意时刻都能清楚知道：当前哪一张卡拥有这条业务流、我现在在这条业务流的哪一步、这张卡还能不能继续操作。**

围绕这个目标，当前更明确的产品判断是：

- 单一业务流默认应该由一张 `owner card` 走到底
- 只要用户还在等待结果、还想看进度、还可能取消，这条业务就不应被拆散到多张卡上
- 真正允许脱离当前 `owner card` 的，只能是产品上显式允许后台执行的例外任务

## 3. 卡片角色

后续飞书交互里的卡片，尽量只落在下面四类之一：

### 3.1 菜单卡（Launcher Card）

职责：

- 只负责分组和功能分流
- 不承担具体业务的执行状态

典型特征：

- 同卡内切组、返回可以 patch
- 菜单分组之间不是独占流程，不需要额外定义“取消”

### 3.2 业务 owner card（Business Owner Card）

职责：

- 承担一条具体业务流的完整交互
- 从开始、编辑、执行，到最终完成，都优先在这张卡里完成

典型特征：

- 内部可以有多个步骤页面
- 也可以有执行中页面、失败页、成功页、最终封卡页
- 只要业务还没结束，这张卡就仍然是当前 owner

### 3.3 后台任务卡（Background Task Card）

职责：

- 承担那些产品上明确允许“开始后不用继续盯结果”的任务

典型特征：

- 不是默认路径，而是显式例外
- 只有任务真的允许后台化时，才把执行和结果移交出去

### 3.4 结果卡（Result Card）

职责：

- 承载最终广播、总结、失败提示、下一步提示

典型特征：

- append-only 为主
- 更适合做附加广播，而不是单一业务流的主编辑界面

## 4. 主设计原则

## 4.1 单一业务默认一张 owner card 走到底

默认规则：

- 如果这是同一条业务流
- 用户还在等待结果
- 或者执行中还需要看进度、看输出、决定是否取消

那么这条业务就应继续停留在同一张 `owner card` 内。

也就是说，这张卡可以依次经历：

- 步骤页
- 编辑页
- 执行中页
- 成功 / 失败 / 已取消封卡页

但原则上仍然是同一张卡。

## 4.2 菜单进入业务时，只做一次 ownership handoff

从菜单进入具体业务时：

- 菜单卡只负责把用户送进业务 `owner card`
- 不应再插一张“命令已提交”之类的中间锚点卡
- 一旦进入业务卡，用户注意力就不应再回到菜单卡

也就是说，菜单只做分流，不做后续执行承接。

## 4.3 编辑态结束，不等于必须换卡

任何会触发副作用的确认动作，都必须让“编辑态”立刻结束。

但这里的“结束”指的是：

- 不能再保留一张看上去还能编辑、实际上已经失效的活卡

它不等于：

- 必须切到另一张新卡

更推荐的做法是：

- 让同一张 `owner card` 从编辑页切到执行中页
- 再从执行中页切到成功 / 失败 / 已取消封卡页

## 4.4 默认不后台化，后台执行必须是显式例外

只有满足下面条件，才适合脱离当前 `owner card`：

- 产品上明确允许“开始后不用继续盯结果”
- 用户暂时离开，不会影响他理解流程是否完成
- 成功 / 失败结果不需要立刻成为当前注意力中心

反过来说，只要任务满足下面任一条件，就不应该后台化：

- 用户还在等待结果
- 用户可能想取消
- 用户需要观察执行进度或输出
- 完成与否会直接影响用户下一步是否能继续工作

## 4.5 业务完成后，优先直接封卡

一条业务完成后，优先让 `owner card` 直接进入最终封卡态。

最终封卡通常应包含：

- 最终结果
- 当前状态摘要
- 用户现在可以做什么

额外的结果卡或 notice，只应作为补充广播，而不是默认主承载。

## 4.6 取消语义必须按层级区分

### 菜单层

- 菜单分组切换只是自由导航
- 不需要强定义“取消”

### 子页面层

例如：路径选择、局部表单、局部确认。

规则：

- `取消` / `返回` 只意味着放弃当前子动作
- 回到上一个父页面继续编辑

### 业务 owner card 层

规则：

- `取消` 意味着放弃整个业务流程
- 不再回到菜单
- 当前卡直接终态化为“已取消”

### 执行中层

如果当前任务支持取消执行：

- 这个取消动作仍然应该留在同一张 `owner card` 里完成
- 取消的是“当前正在执行的业务”，而不是只取消一段展示

## 4.7 一张卡只允许一个主问题

一张卡可以有多个控件，但只能有一个主问题。

判断方式不是数控件，而是看用户此刻是不是在回答同一个问题：

- 如果多个控件都在帮助用户回答同一个问题，可以同页
- 如果一张卡开始同时要求用户回答两个不同阶段的问题，就该拆成不同页面

这条原则尤其适用于：

- 高频联动比较区可以同页保留
- 重分支、重表单、重副作用区必须拆到下一步页面

## 5. 展示层原则

飞书卡片空间很有限，因此在展示层还需要额外约束：

### 5.1 阶段头要短、单行、弱化

- 阶段头优先用简短斜线分隔，例如：`模式/目标`、`模式/来源/Git`
- 一旦会换行，就先缩文案，不继续堆字
- 阶段头应用次级小字样式，不用 code 字体

### 5.2 当前上下文默认不展示

当前工作区 / 当前会话这类上下文：

- 默认不展示
- 如果确实有价值，也只能保留成一行次级小字

### 5.3 单一主问题优先纵向布局

如果页面当前只有一个主问题：

- 二选一就纵向排布
- 不为了“看起来像 tab”硬做横排

### 5.4 横向布局只在被迫并存多个问题时使用

只有在同一页里确实并存多个问题、必须压缩空间，并且横排能明显减少换行和空间浪费时，才考虑使用原生分栏。

## 6. 常见判断题

后续做业务卡片设计时，可以先用下面几条快速自检：

1. 这是不是同一条业务流？
2. 用户是不是还在等待结果？
3. 用户执行中是不是还可能想取消？
4. 这张卡是不是同时在让用户回答多个阶段的问题？
5. 当前这一步如果完成，用户下一步是否立刻受影响？
6. 这个任务是不是产品上明确允许后台执行？

如果前五条多数答案都是“是”，而第六条不是“是”，那就应优先留在同一张 `owner card` 里，不要中途拆散到多张卡。

## 7. 与结果广播的关系

这份原则文档并不否认 append-only 结果卡的价值。

当前判断是：

- append-only 的 final / notice / snapshot 依然有用
- 但它们更适合做广播、补充、留痕
- 不应继续承担“这条业务现在由谁承接”的职责

因此后续设计时，先回答：

- 当前业务 owner 是谁

再决定：

- 是否额外补一张广播卡

顺序不能反过来。

## 8. 适用范围与后续演进

这份原则当前优先约束：

- 菜单进入业务的 handoff
- picker 类业务流
- 配置类卡片
- request / approval / staged input
- 长任务执行中的卡片状态

后续如果有新的业务流与这里的原则冲突，应先回到这份文档补充或修正原则，再去实现具体卡片。
