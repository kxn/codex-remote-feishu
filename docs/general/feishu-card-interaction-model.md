# 飞书卡片交互模型

> Type: `general`
> Updated: `2026-04-21`
> Summary: 固化当前产品讨论收敛出的飞书交互模型，补充前台抓注意力状态、前台卡 notice 区、notice 的前后台投递规则，以及将 thread selection change 收敛进 notice 体系。

## 1. 文档定位

这份文档回答的是产品层的问题：

- 飞书侧到底应该存在几种交互模型
- 什么叫前台抓注意力，什么时候释放
- 哪些交互必须始终停留在同一张卡里
- notice 在前台、turn 运行中、完全后台三种状态下应该怎样投递
- turn 运行过程中，哪些内容属于持续 patch 的共享过程卡
- 哪些内容应该作为独立卡或独立消息插入时间线

它不负责描述具体 callback 字段、旧卡拒绝、daemon 生命周期 stamping 等实现细节。

这些内容分别以其他文档为准：

- 产品与业务卡片通用原则：
  - `docs/general/feishu-business-card-interaction-principles.md`
- 当前已实现的 Feishu UI 状态机：
  - `docs/general/feishu-card-ui-state-machine.md`
- 飞书卡片平台限制：
  - `docs/general/feishu-card-api-constraints.md`

## 2. 核心目标

这套模型的目标不是把所有内容都塞进一张大卡里，而是让用户始终清楚下面四件事：

1. 当前哪一张卡拥有这条交互
2. 当前是不是有一张卡正在抓前台注意力
3. 当前看到的是前台卡、过程卡、检查点卡，还是后台时间线输出
4. 这张卡接下来会继续更新，还是已经封口

## 3. 核心术语

### 3.1 前台卡

指当前正在抓用户注意力的 owner card。

它既可以是：

- 用户主动发起的前台流程卡
- 后台突然弹出的确认卡
- 单步但仍属于主动操作结果的状态卡，例如 `/status`

只要它仍在抓注意力，就应只认这一张卡，不应在流程中途逃逸成另一张新的主卡。

### 3.2 前台抓注意力

指用户当前应盯着某一张卡继续完成交互，这张卡还没有放开前台控制权。

进入前台抓注意力的典型来源：

- 用户主动发起菜单 / slash command 流程
- 后台主动弹出确认、审批、授权之类的卡

判断标准不是“谁发起”，而是：

- 用户当前是不是还必须盯着这张卡
- 这张卡是不是还在等待用户确认、输入、观察结果或取消

### 3.3 前台释放点

指一张前台卡从“还在抓注意力”切换为“已经放开前台”的那个产品时刻。

这个释放点必须由每条业务单独定义清楚。

典型例子：

- Git clone 这类长动作，用户点确认后，卡仍然可能继续抓前台，因为用户还在等这张卡内的过程和结果
- 升级这类动作，用户确认后，产品可以选择立即放开前台；之后的提示改走后台 notice 或其他后台交付

### 3.4 前台卡 notice 区

指前台卡内部专门承接 notice 的区域。

位置要求：

- 放在业务内容区与底部确认 / 取消 / 下一步按钮区之间
- 没有内容时完全隐藏
- 有内容时作为卡内提示区出现，不替换主业务内容

只要当前卡仍在抓前台注意力，相关 notice 就优先打进这个 notice 区，而不是另外发一张新卡或新 notice。

### 3.5 turn 时间线

指一次 turn 运行过程中，按时间向下追加的所有可见输出序列。

它不是单一一张卡，而是一条由多种可见项组成的时间线。

### 3.6 可见项

指最终真的渲染给用户的一条输出。

它可以是：

- 一张卡
- 一条文本
- 一张图片
- 一条后台 notice

如果某个内部事件最后没有被渲染出来，它就不算可见项。

### 3.7 打断

指当前活跃的共享过程卡被封口，不再继续 patch。

一旦发生打断，后续如果还有新的共享过程输出，必须新开一张共享过程卡。

## 4. 产品状态

在产品层，当前应先区分下面三种状态。

## 4.1 前台抓注意力状态

特征：

- 当前同一时刻只应有一张前台卡
- 这张卡正在吃用户注意力
- 用户还需要继续盯着它完成交互
- notice 不应打断它，也不应另发新卡覆盖用户注意力

这类前台卡包括：

- 用户主动发起的前台流程卡
- 后台突然弹出的确认卡
- 主动触发的单步状态卡，例如 `/status`

## 4.2 turn 运行中状态

特征：

- 当前存在某个 turn 的运行输出
- 运行中内容以共享过程卡、检查点卡、普通文本、图片等形式落在时间线里
- 这是一条时间线状态，不等于前台抓注意力

如果 turn 过程中突然弹出确认卡，则：

- 确认卡成为当前前台抓注意力状态
- turn 自己的时间线仍按其产品规则继续存在

## 4.3 完全后台状态

特征：

- 当前没有卡在抓用户注意力
- 用户可以继续自由输入
- 如果这时系统需要提示某件事，提示应以后台 notice 或其他后台时间线项出现

## 5. 主渲染模型

当前应收敛为下面四类主渲染模型，再加一个跨层的 `notice` 体系。

## 5.1 前台卡

适用场景：

- 菜单进入后的业务向导
- slash command 打开的业务卡
- 工作区 / 会话选择
- 各类 picker 与其子步骤
- approval / `request_user_input` / MCP 授权确认
- 主动触发的单步状态卡，例如 `/status`

产品要求：

- 整条前台交互只在同一张卡里完成
- 前进、返回、局部选择、校验失败、处理中、完成态都 patch 当前卡
- 不允许中途改成“原卡保留，结果另发主卡”
- 不允许插入“命令已提交”之类的过渡主卡
- 必须显式定义这条卡什么时候释放前台注意力

## 5.2 共享过程卡

适用场景：

- 执行过程中的共享 progress
- tool / command / file change / compaction 等过程类输出
- 其他需要把最近一段过程聚合在同一张卡里持续 patch 的场景

产品要求：

- 任意时刻最多只能有一张活跃的共享过程卡
- 活跃共享过程卡永远是当前时间线里最下面的一张共享过程卡
- 这张卡只承载一段连续的、最终实际渲染出来的共享过程输出
- 同一段连续过程里，后续共享过程内容继续 patch 这张卡
- 一旦中间插入任何非共享过程的独立可见项，这张卡立即封口
- 后面若再有共享过程输出，必须重新开一张新的共享过程卡

共享过程卡不是整轮 turn 的唯一主卡。

它只是 turn 时间线里一段“最近过程窗口”。

## 5.3 检查点卡

适用场景：

- `plan` 卡
- `final` 卡
- 未来其他需要作为阶段里程碑保留在时间线中的卡

产品要求：

- 检查点卡是 append 型的独立卡
- 它不是共享过程卡的一部分
- 它出现时，会打断当前共享过程卡
- 它发出后保留在时间线上，随后可被后续消息逐渐往上挤

其中：

- `plan` 卡代表过程中的阶段性里程碑
- `final` 卡代表这轮 turn 的最终收口

`plan` 不应简单并入共享过程卡，因为它不是“最近过程窗口”，而是“应被保留下来的中途检查点”。

## 5.4 独立时间线项

适用场景：

- 正常的 agent message 文本
- 图片输出
- 后台 notice
- 其他最终独立渲染出来、但不属于前台卡、共享过程卡或检查点卡的内容

产品要求：

- 这类内容按时间线独立追加
- 它们不是前台卡的一部分
- 它们只要真的渲染出来，就会打断当前共享过程卡

## 6. notice 体系

`notice` 不是单一的卡片模型，而是一套跨前台 / turn / 后台的提示体系。

它要同时回答两个问题：

1. 这条 notice 属于哪种语义
2. 当前应该投递到前台卡内部，还是作为后台独立项发出

## 6.1 notice 的投递规则

### 当前存在前台抓注意力卡

如果当前有前台卡在抓注意力，并且 notice 属于这张卡的当前业务上下文，则：

- 不应打断当前前台卡
- 不应另外新发一张 notice 卡
- 不应把原有业务内容整块替换成 notice 文本
- 应进入这张前台卡的 `notice 区`

也就是说：

- notice 是卡内提示区的更新
- 不是一条新的独立时间线项
- 也不应算作会打断共享过程卡的独立可见项

### 当前没有前台抓注意力卡

如果当前没有前台卡抓注意力，则：

- notice 可以作为后台独立时间线项发出
- 当前实现先按单发卡来定义

### 前台释放后的行为

一张前台卡一旦释放前台注意力：

- 后续 notice 不应继续回填进这张已经放开的卡
- 若还有新的提示，应改走后台独立 notice

例如：

- Git clone 可以定义为“确认后仍保持前台”，此时 notice 继续打进卡内 notice 区
- 升级可以定义为“确认后立即释放前台”，此时后续提示不再写回原卡

## 6.2 notice 的语义家族

按产品语义，当前 notice 至少应拆成下面几类。

### A. 流程内 notice

含义：

- 当前前台流程内部的校验、失效、上下文错误、局部失败

例子：

- 卡片过期
- picker 失效
- 请求字段无效
- owner-flow 缺少上下文

产品定义：

- 如果当前仍有对应前台卡抓注意力，应优先进入该卡的 notice 区
- 不应把整个业务页替换成一句话

### B. 动作结果 notice

含义：

- 用户主动操作后的结果反馈

例子：

- 已切换模式
- 已开启 / 关闭某设置
- 已 detach
- 已进入 follow
- 已准备新建会话

产品定义：

- 如果对应动作仍属于前台卡流程，就进入该卡 notice 区
- 如果动作在确认后已释放前台，则以后台 notice 发出

### C. 目标 / 路由 notice

这类 notice 吸收原先单独的 `thread selection change` 语义。

也就是说，`thread selection change` 不再需要作为独立大类理解，而应看作 notice 家族里的一个专门分支。

它的含义是：

- 当前输入目标或路由状态发生了变化

典型出现时机：

- `/use` 或其他显式接管动作切到了具体会话
- `/new` 进入了 `new_thread_ready`
- `/follow` 进入等待中
- follow 模式自动重定向到新的线程
- 队列或远端事件落地后，surface 被补绑到某个具体线程
- 强踢 / 释放 / 失去拥有权后，surface 退回未绑定或跟随等待态

产品定义：

- 用户主动触发的目标切换，通常值得显示
- 系统自动重定向导致的切换，应尽量弱化，避免无意义地打断当前交互
- 若当前有前台卡抓注意力，应优先进入前台卡 notice 区，而不是再插一张新卡

### D. 全局运行时 notice

含义：

- 不属于当前某个前台业务步骤，而是系统后台运行时自己冒出来的提示

例子：

- surface resume 失败
- VS Code resume 失败
- 需要先打开 VS Code
- transport degraded
- daemon shutting down
- gateway apply failure

产品定义：

- 这类 notice 通常是后台独立项
- 它们当前也已有单独的 delivery class / family / 节流逻辑
- 不应假装自己属于某张普通前台业务卡

### E. 问题 / 错误 notice

含义：

- 执行失败、链路错误、problem 转换出的错误提示

产品定义：

- 如果错误发生在某张仍抓注意力的前台卡内部，优先进入该卡 notice 区
- 如果错误来自后台运行时，则作为后台独立 notice

## 6.3 `thread selection change` 的收口结论

当前产品上，不再建议把 `thread selection change` 视为独立大类。

更合适的定义是：

- 它属于 `notice` 体系
- 更具体地说，它属于 `目标 / 路由 notice`

这样做的好处是：

- 它可以复用和其他 notice 一样的投递规则
- 当前台卡抓注意力时，它可以进入卡内 notice 区
- 当前台卡不在时，它也可以作为后台 notice 独立出现

## 7. 共享过程卡的严格规则

共享过程卡的关键不是“上游发来了什么事件”，而是“最终有没有渲染出别的独立可见项”。

因此它的分段依据必须按最终渲染结果来判断，而不能只按内部事件类型判断。

### 7.1 连续 patch 的条件

只有同时满足下面条件，当前共享过程卡才继续 patch：

1. 当前已有一张活跃共享过程卡
2. 新到达的内容最终会被渲染为共享过程内容
3. 从上一条共享过程内容到这次内容之间，没有插入任何其他非共享过程的独立可见项

### 7.2 打断条件

只要中间插入任意一条最终真的显示给用户的非共享过程独立可见项，当前共享过程卡就必须被打断并封口。

至少包括：

- 普通 agent message 文本
- 前台确认卡
- `plan` 卡
- `final` 卡
- 图片
- 后台独立 notice
- 其他独立卡或独立消息

### 7.3 不打断条件

如果某个中间事件因为产品可见性策略没有真正渲染出来，它就不应打断共享过程卡。

例如：

- 因 `verbose` 等级被压掉的过程事件
- 被聚合后没有单独显示的内部过程片段

如果某条 notice 只是进入前台卡内部的 notice 区，它也不属于独立时间线项，因此不应打断共享过程卡。

### 7.4 重开规则

当前共享过程卡被打断后，后面若再次出现可见的共享过程输出，则：

- 不能回头继续 patch 旧卡
- 必须新开一张新的共享过程卡
- 新卡继续成为当前时间线最下面、唯一活跃的共享过程卡

## 8. 典型示例

### 8.1 纯过程到最终结束

```text
共享过程卡 P1 -> final 卡 F1
```

### 8.2 过程中插入普通文本

```text
共享过程卡 P1 -> 文本 T1 -> 共享过程卡 P2 -> final 卡 F1
```

### 8.3 过程中插入确认流

```text
共享过程卡 P1 -> 前台确认卡 R1 -> 共享过程卡 P2 -> final 卡 F1
```

### 8.4 过程中插入 plan 卡

```text
共享过程卡 P1 -> plan 卡 L1 -> 共享过程卡 P2 -> final 卡 F1
```

### 8.5 前台卡内部追加 notice

```text
前台卡 F1(notice 区追加 N1) -> 仍然是同一张前台卡 F1
```

### 8.6 多次交错

```text
共享过程卡 P1 -> 文本 T1 -> 共享过程卡 P2 -> 前台确认卡 R1 -> 共享过程卡 P3 -> plan 卡 L1 -> 共享过程卡 P4 -> final 卡 F1
```

## 9. 当前功能到模型的映射

| 当前功能或输出 | 应归入的模型 |
| --- | --- |
| 菜单进入的业务向导、target picker、path picker、history 等 owner card 流程 | 前台卡 |
| approval、`request_user_input`、MCP 授权、升级确认 | 前台卡 |
| `/status` 这类主动触发的单步状态卡 | 前台卡 |
| exec/tool/file change/compaction 等最近过程窗口 | 共享过程卡 |
| `plan`、`final` | 检查点卡 |
| 普通文本、图片、后台独立 notice | 独立时间线项 |
| 原 `thread selection change` | `notice` 体系中的 `目标 / 路由 notice` |
| reaction | 辅助微交互，不属于上面四种主模型，也不属于 notice 主体 |

## 10. 收口方向

如果后续做结构统一，推荐按下面三层来收。

### 10.1 前台卡引擎

统一承接：

- 前台主动流程
- 后台触发但会抓注意力的确认流
- 单步状态卡

共享语义是：

- 始终只有一张 owner card
- 整轮交互在同一张卡里完成
- 结束前不拆出新的主交互卡
- 必须显式定义前台释放点
- notice 优先进入卡内 notice 区

### 10.2 turn 时间线渲染层

统一承接：

- 共享过程卡
- 检查点卡
- 独立时间线项

关键语义是：

- 共享过程卡只有一张活跃卡
- 检查点卡是 append 型里程碑
- 任何独立可见项都会打断共享过程卡
- 卡内 notice 区更新不属于独立可见项

### 10.3 notice 投递层

统一承接：

- 流程内 notice
- 动作结果 notice
- 目标 / 路由 notice
- 全局运行时 notice
- 问题 / 错误 notice

关键语义是：

- 先判断 notice 属于哪个语义家族
- 再根据当前是否存在前台抓注意力卡，决定打进卡内 notice 区还是后台独立发出

## 11. 不应误解的点

### 11.1 不是所有内容都应合进一张大卡

这套模型明确反对“一个 turn 永远只用一张卡承载所有内容”。

原因包括：

- `plan` 和 `final` 的语义不是滚动过程窗口
- 独立文本、图片、后台 notice 本来就更适合作为时间线项存在
- 飞书卡片本身也有体积、组件数和更新频率限制

### 11.2 `plan` 不是共享过程卡的一部分

`plan` 会打断共享过程卡。

它更像过程中的里程碑卡，而不是共享过程卡里的一个小 section。

### 11.3 共享过程卡的分段依据是最终渲染结果

决定它是否被打断的，不是内部产生了什么事件，而是用户最终是否真的看到了别的独立可见项。

### 11.4 notice 不是天然都该独立发出

如果当前有前台卡抓注意力，很多 notice 更适合进入卡内 notice 区，而不是额外插一张新卡或新提示。

### 11.5 `thread selection change` 不再建议当成独立大类

它更合适作为 notice 家族里的 `目标 / 路由 notice` 处理。

## 12. 后续使用方式

以后凡是设计或重构飞书交互时，先回答下面五个问题：

1. 当前是不是有一张前台卡正在抓注意力，它什么时候释放
2. 这件事属于前台卡、共享过程卡、检查点卡、独立时间线项，还是某类 notice
3. 这条 notice 应该进入前台卡 notice 区，还是作为后台独立项发出
4. 它是否会打断当前共享过程卡
5. 它完成之前，用户到底应该盯着哪一张卡

如果这几个问题答不清楚，就说明交互模型还没有定清楚，不应直接进入实现。
