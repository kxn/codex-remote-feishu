# Web 设计规范

> Type: `general`
> Updated: `2026-04-24`
> Summary: 补充页面 Mock 与按 Mock 落产品的一致性基线，要求用户可见结构、交互和文案保持受控。

## 1. 适用范围

这份规范适用于本仓库所有面向用户的 Web 页面与页面级改动，包括但不限于：

- `web/src/**`
- `internal/app/daemon/adminui/**`
- setup 向导、admin 管理页、安装页、引导页、状态页
- 页面信息架构、交互流程、文案和默认显隐策略
- 页面 Mock、页面原型和浏览器可运行预览页

如果某个页面同时包含产品化内容和排障能力，本规范默认约束产品化主界面；排障信息应收进明确的技术详情、折叠区或按需展开入口。

如果本次任务包含页面 Mock 或浏览器可运行原型，还必须同时遵守：

- [page-mock-guidelines.md](./page-mock-guidelines.md)

## 2. 目标

统一后续 Web 改动的最低设计基线，避免页面再次出现这些问题：

- 只照顾桌面端，移动端阅读和操作负担过重
- 直接用研发视角写文案，把普通用户不需要理解的技术术语摆在主界面
- 把单页面内容直接堆成长滚动页，让用户在手机上不断下翻才能找到当前操作
- 把设计说明、实现动机或内部备注直接写进用户可见页面

本规范不定义唯一视觉风格，也不规定必须使用某一种组件库。

## 3. 基础原则

### 3.1 双端默认成立

- 每个页面默认同时面向桌面端和移动端
- 不能先把桌面端做完，再把移动端当作事后补丁
- 在窄屏下，主操作、主状态和主 CTA 必须仍然可见、可点、可理解
- 桌面端允许更丰富的信息架构，但不能依赖“只有大屏才看得懂”

### 3.2 普通用户优先

- 默认假设用户只关心“现在要做什么、是否成功、下一步去哪”
- 页面主文案优先使用任务口径，而不是实现口径
- `gateway`、`surface`、`manifest`、`reconcile`、内部状态名等概念，不应默认出现在主界面
- 技术细节只有在这两种情况下才应该出现：
  - 当前步骤必须让用户处理它
  - 当前步骤失败，需要给出可执行的诊断信息

### 3.3 信息分层优先于信息堆叠

- 可以接受单页面应用
- 但不能把所有步骤、说明、状态和未来信息直接铺成一整页
- 页面必须明确区分：
  - 当前最需要用户处理的内容
  - 已完成但只需要摘要保留的内容
  - 尚未到达、不该提前暴露的内容
- 不重要的信息应折叠、延后或隐藏，而不是“先全部放出来再靠用户自己筛”

### 3.4 页面只服务当前任务

- 每个页面或每个步骤都应有一个清晰主目标
- 默认只保留一个主要 CTA；次要操作应降级处理
- 不要为了“信息完整”把后续步骤、未来能力或低频排障入口挤进当前主流程
- 当前步骤需要的说明可以充分，但必须直接服务于当前动作

### 3.5 内部设计说明不进入用户界面

- 页面中禁止直接出现“设计目的”“设计说明”“为什么这样设计”等内部视角文案
- 设计动机、实现边界、架构取舍应写在文档、issue、注释或技术详情中，而不是用户页面正文
- 用户看到的应该是结果导向说明，例如：
  - 现在要做什么
  - 这样做会完成什么
  - 失败时应该怎么办

### 3.6 页面 Mock 也按最终页面要求处理

- 页面 Mock 不是静态示意稿，而是浏览器中可运行的最终交互版本
- 页面 Mock 中只能出现最终用户关心的内容
- 页面 Mock 禁止出现 `mock`、`demo`、`prototype`、`设计说明`、`TODO` 等用户可见设计意图字样
- 页面 Mock 可以使用假数据，但所有用户可交互的数据面和交互操作都必须真实可用
- 页面 Mock 必须覆盖响应式变化与横竖屏切换，而不是只做单一静态视口
- 如果真实产品是按已确认 Mock 落地，除真实业务数据和技术接入外，最终页面的用户可见结构、交互与文案都应与 Mock 保持一致
- 开发实现阶段禁止自行添加 Mock 中没有、且最终用户不应看到的额外文案

## 4. 页面结构要求

### 4.1 桌面端

- 可以使用侧栏、分栏、卡片分区等信息架构
- 主要操作区应稳定，避免让用户在长页面中反复寻找按钮
- 技术详情不应与主流程混排

### 4.2 移动端

- 默认按单列阅读和单手操作设计
- 页面进入窄屏后，应优先保证：
  - 当前步骤标题与状态可见
  - 主操作按钮不需要长距离滚动才能找到
  - 导航、侧栏或辅助面板不会长期挤占正文
- 如果桌面端使用侧栏，移动端应收敛成顶部导航、抽屉或分段入口，而不是简单把双栏硬挤成窄列长页

### 4.3 单页与分步

- 单页 SPA 允许存在，但必须通过分步、分段、折叠或路由切分控制信息曝光
- 不允许把完整流程正文按时间顺序一次性全部展开
- 用户在当前步骤，不应轻易滚动看到后续步骤的大段正文
- 已完成步骤默认只保留摘要；未开始步骤默认只保留名称和状态

## 5. 内容显隐规则

### 5.1 默认应该直接展示的内容

- 当前要做的事情
- 当前状态是否正常
- 本步成功或失败的判断结果
- 明确下一步入口

### 5.2 默认不应直接展示的内容

- 底层配置路径
- 内部标识和内部状态名
- 协议、同步、重试、底层 runtime 细节
- 与当前步骤无关的未来功能说明
- 只对研发或排障有意义的设计背景

### 5.3 允许延后或折叠的内容

- 高级设置
- 诊断信息
- 原始错误详情
- 技术字段与系统路径
- 低频管理能力

如果这些内容会影响当前任务完成，应提供明确入口和上下文说明，而不是完全埋掉。

## 6. 文案规则

- 用“用户现在能完成什么”来命名区块和按钮
- 避免把实现术语直接当成功能名称
- 避免把内部对象模型直接暴露给用户
- 避免为了说明充分而写成长段设计自述
- 如果一句文案只有研发能看懂，就不应该出现在默认主界面

推荐方向：

- `继续完成首次配置`
- `打开飞书后台`
- `查看技术详情`
- `清理旧图片`

不推荐方向：

- `进入 reconcile`
- `查看内部标识`
- `设计目的`
- `当前 surface 状态`

## 7. 落地检查表

后续每次新增或修改 Web 页面时，至少检查以下内容：

1. 这页在桌面端和移动端都能完成主任务吗？
2. 主界面文案是否仍然站在普通用户视角，而不是研发视角？
3. 是否把不重要的信息折叠、延后或移到了技术详情，而不是默认全展开？
4. 如果是单页或向导，用户是否会在当前步骤轻易看到后续整段内容？
5. 页面里是否出现了“设计目的”“设计说明”或同类内部说明性文案？
6. 主 CTA 是否明确，页面是否围绕当前任务收敛？

如果其中任一问题回答为“否”或“是，存在问题”，该页面改动默认不应直接合入。
