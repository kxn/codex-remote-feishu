# Web 管理界面重设计

> Type: `implemented`
> Updated: `2026-04-11`
> Summary: 同步飞书配置未生效状态、admin 侧扫码新增机器人、扫码完成说明页、admin/setup 共用移动端 shell 与相对本地路径约束，管理页默认视图继续收敛到日常运维任务。

## 1. 文档定位

本文记录当前已经落地的 `web admin` 管理界面行为。

它回答三件事：

1. 管理页现在按什么信息架构组织。
2. 用户在默认视图里能完成哪些日常任务。
3. 哪些内容被明确收进了技术详情，而不再占据主界面。

相关文档：

- [web-setup-wizard-redesign.md](../draft/web-setup-wizard-redesign.md)
- [web-install-admin-prerequisites-design.md](../draft/web-install-admin-prerequisites-design.md)

## 2. 已实现的信息架构

当前管理页固定分为 6 个区域：

1. `总览`
2. `飞书机器人`
3. `工作实例`
4. `文档与图片`
5. `VS Code`
6. `技术详情`

其中前 5 个是日常操作区，`技术详情` 只用于排障和查看底层状态。

管理页和 setup 当前还共用两条基础约束：

- 桌面端保留侧栏信息架构；窄屏下侧栏会收起为顶部可展开导航，同一套 shell 同时服务 admin 与 setup
- 页面内本地跳转、静态资源和本地 API 入口统一按相对本地路径生成，避免 `/assets/...`、`/api/...` 这类根路径假设把反代访问弄坏

## 3. 当前实现

### 3.1 总览

总览卡片只保留这 4 类摘要：

1. 机器人数量与在线数
2. 需要处理的事项数
3. 工作实例数量与在线数
4. 文档与图片占用概览

下面提供一个 `需要关注` 列表，只列建议优先处理的项目，例如：

- 机器人连接失败
- 机器人还未完成连接测试
- 机器人首次配置尚未完成
- VS Code 尚未完全接入当前 relay
- 旧图片暂存文件可清理

每个项目都带直接入口，要么跳到对应管理分区，要么直接跳到 setup 继续处理。

### 3.2 飞书机器人

机器人区域采用 `左侧列表 + 右侧详情`。

左侧列表默认只展示：

- 机器人名称
- App ID
- 连接状态
- 启用状态
- 是否存在 `未生效 / 待重试 / 待移除`
- 是否需要继续完成首次配置
- 是否只读

右侧详情默认展示：

- 当前连接状态
- 最近验证时间
- 当前 runtime apply 状态
- 首次配置完成度
- 是否可编辑
- 机器人名称 / App ID / App Secret / 启用开关
- 保存、测试连接、重新连接、启停、删除
- 当配置已落盘但 runtime apply 失败时，明确显示 `未生效 / 待重试` 或 `待移除`，并提供重试入口

当前“新增机器人”已经不是单一路径表单，而是两条明确分流：

- `新建飞书应用`
  - 作为推荐路径
  - 管理页会创建 onboarding 会话，展示二维码，并把自动轮询状态明确展示出来
  - 扫码成功后，不会直接掉回编辑态；会先显示一张“扫码完成说明页”，告诉用户基础接入已自动完成了什么、还剩哪些可选动作
  - 用户点击“下一步”后，才回到新机器人的详情/编辑态
- `接入已有应用`
  - 继续保留手动填写 `App ID / App Secret` 的路径
  - 不会被强行改成“扫码选择已有 app”

扫码新建路径下：

- 机器人显示名称会尽量从飞书侧带回；如果当前拿不到，就先回退到 `App ID`
- 说明页会明确提示：如果需要 Markdown 预览上传到飞书云盘，还要额外申请 `drive:drive`，且通常需要管理员审批

如果机器人尚未完成首次配置，会出现明确的 `继续完成首次配置` 入口。

该入口会跳到 setup 页，并通过 `?app=<id>` 预选当前机器人，避免用户进入 setup 后再自己重新寻找目标机器人。

如果机器人由运行时参数接管，则管理页进入明确只读态：

- 字段不可编辑
- 保存和删除不可操作
- 页面显式说明只读原因

机器人内部标识、保存来源、完整首次配置检查项等内容仍然保留，但只放在 `查看技术详情` 的折叠区域里。

### 3.3 工作实例

工作实例区域现在聚焦一个任务：

1. 查看当前可接入的 VS Code 工作实例

默认文案使用用户任务口径：

- `当前实例`
- `当前策略`
- `后台恢复实例由系统自动管理`

实例卡片默认展示：

- 实例名称
- 在线 / 启动中 / 异常 / 离线
- 工作目录
- 实例来源
- 最近状态摘要

这里不再提供“新建实例”或“删除实例”操作。

当前策略是：

- 管理页只展示当前可见的 VS Code 实例
- 后台恢复实例由系统按需自动复用或启动
- 后台恢复实例不再作为普通实例列出来，也不再允许从 admin API 手工创建或删除

### 3.4 文档与图片

该区域分成两块：

1. `图片暂存`
2. `文档预览`

图片暂存默认只展示：

- 暂存文件数
- 占用空间
- 当前仍在使用的文件数

并提供 `清理旧图片`。

文档预览按机器人分别展示，每个机器人各自维护自己的预览目录。默认界面只展示：

- 是否已经启用预览
- 文件数量
- 估算占用
- 最近使用时间
- `清理旧预览`

预览目录链接、授权对象数、目录一致性检查等信息，统一收进卡片内的技术详情折叠区。

### 3.5 VS Code

VS Code 区域现在默认使用用户能理解的口径，而不是直接暴露内部模式名。

默认界面展示：

- 当前环境是本机还是远程 SSH 机器
- 当前是否已接入
- 当前建议路径
- 是否需要重新安装扩展入口

主交互按场景组织：

- 如果当前是远程 SSH 机器：
  - 只提供“在这台远程机器上启用 VS Code”这一路径
  - 若还没检测到远程扩展入口，会直接显示阻塞说明，而不是给出错误模式选择
- 如果当前不是 SSH 机器：
  - 先让用户选择使用场景：
    - `要在当前这台机器上使用`
    - `主要去别的 SSH 机器上使用`
  - 然后再给当前场景唯一合理的主按钮和结果提示
  - “要在当前这台机器上使用”统一只处理扩展入口，不再写本机 `settings.json`
  - “主要去别的 SSH 机器上使用”会在当前机器明确跳过，而不是误写本机 `settings.json`
  - 若检测到旧版 `settings.json` override 或 managed shim 已失效，会直接展示迁移/修复状态，而不是把它继续当成可选模式

底层模式型操作被折叠到 `高级处理`：

- `重新安装扩展入口`
- `查看技术详情`

执行文件路径、安装状态文件、记录中的扩展入口等底层字段，移入技术详情折叠区。

### 3.6 技术详情

技术详情是单独分区，不再与主流程混排。

当前集中收纳：

- 本机配置路径与地址
- 当前选中机器人的内部字段
- 权限导入 JSON
- 机器人连接原始状态

这样主界面保留产品化信息层，底层信息仍可在排障时直接查看。

## 4. 本轮收敛掉的默认暴露项

本轮实现中，这些词或概念已不再占据主界面默认文案：

- `gateway`
- `wizard progress`
- `scopes JSON`
- `managed shim`
- `reconcile`
- `surface`
- `remote queue`

其中确实仍有排障价值的内容，被移动到了技术详情或折叠区。

## 5. 实现边界

### 5.1 继续配置入口依赖 setup 预选

为了让管理页里的 `继续完成首次配置` 真正可用，setup 页现已支持通过 URL 查询参数预选机器人：

- `?app=<app-id>`

这只是预选当前机器人，不改变 setup 原有的步骤逻辑。

### 5.2 preview reconcile 已下线

本轮之后，预览目录 `reconcile` 不再作为管理页能力保留。

原因是 preview drive 已切到 fixed-root remote-first 语义：

- status 直接围绕远端 inventory 汇总
- cleanup 直接围绕远端 inventory 生效
- “本地对账”不再提供额外产品价值，反而会放大本地 state 仍是治理真源的误解

### 5.3 当前改版包含前后端一起收口

这轮改版主要收敛的是：

- 信息架构
- 默认文案
- 操作入口
- 管理路径的连贯性
- admin API 的默认暴露面

但并未借机重做 daemon 侧整体实例模型。

### 5.4 移动端与相对路径约束已经进入实现基线

当前落地约束如下：

- admin 与 setup 共用移动端 shell；窄屏下分区导航通过顶部入口展开，点击后会自动收起
- hero、panel header、按钮区、机器人编辑区、向导区和双栏布局在手机宽度下会优先收敛为单列与全宽按钮
- 构建产物里的静态资源引用、前端本地跳转和本地 API 请求都统一走相对本地路径

这意味着：

- `localhost` 下直接访问仍然可用
- 挂到带路径前缀的本地反代下时，也不需要额外依赖站点根路径兼容层
