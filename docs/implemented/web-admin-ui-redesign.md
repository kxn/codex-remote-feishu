# Web 管理界面重设计

> Type: `implemented`
> Updated: `2026-04-16`
> Summary: 补记 `#218` 已把 Web Admin 主入口固定到 `/admin/`，根路径只保留轻量说明页，setup / startup / install 完成后的默认返回地址统一回到 `/admin/`。

## 1. 文档定位

本文记录当前已经落地的 `web admin` 管理界面行为。

它回答三件事：

1. 管理页现在按什么信息架构组织。
2. 用户在默认视图里能完成哪些日常任务。
3. 哪些旧语义已经从主界面移走，哪些能力仍然依赖旧后端数据推导。

## 2. 已实现的信息架构

当前管理页固定分为 6 个区域：

1. `运行概览`
2. `飞书应用`
3. `工作实例`
4. `文档与图片`
5. `VS Code`
6. `技术详情`

其中前 5 个是日常操作区，`技术详情` 只用于排障和查看底层状态。

管理页和 setup 当前还共用两条基础约束：

- 桌面端保留侧栏信息架构；窄屏下侧栏会收起为顶部可展开导航，同一套 shell 同时服务 admin 与 setup
- 页面内所有引用的本地资源、本地跳转和本地 API 入口都按相对本地路径生成，避免 `/assets/...`、`/api/...`、`/setup` 这类根路径假设把反代访问弄坏

当前挂载路径也已经固定：

- `GET /admin/` 是唯一的主管理页入口，`/admin` 只负责补斜杠重定向
- 根路径 `/` 只保留轻量说明页，不再承担主管理页职责
- `setup` 继续保留独立的 `/setup` 入口，但 setup 完成、startup/install 回跳与本地管理链接都会统一落到 `/admin/`

## 3. 当前实现

### 3.1 运行概览

概览卡片只保留这 4 类摘要：

1. 飞书应用数量与当前可用数
2. 需要处理的事项数
3. 工作实例数量与在线数
4. 文档与图片占用概览

下面提供一个 `建议优先处理` 列表，只列当前最容易影响使用体验的项目，例如：

- 还没有接入任何飞书应用
- 某条飞书应用还没完成连接验证
- 某条飞书应用还有待补全项
- 某条飞书应用最近连接不稳定
- VS Code 尚未完全接入当前 relay
- 旧图片暂存文件可清理

这些入口现在都直接回到 admin 内对应区域处理，不再默认把用户送回 setup。

### 3.2 飞书应用

飞书应用区域采用 `左侧列表 + 右侧详情`。

左侧列表默认展示：

- 飞书应用名称
- App ID
- 当前主状态，例如 `可用`、`可用，待补全`、`待验证`、`待同步`、`外部接管`
- 启用状态
- 是否只读
- 是否仍有待补全项

右侧详情默认分成 4 层：

1. 头部状态卡
2. 接入信息
3. 能力摘要与待处理项
4. 接入说明与技术详情

#### 头部状态卡

默认展示：

- 当前主状态
- 启用状态
- 配置来源
- 是否外部接管
- 待补全项数量

如果最近保存后运行时应用失败，或运行时移除失败，会在这里直接显示 `待同步 / 待移除` 提示并提供重试入口。

#### 接入信息

默认提供：

- 飞书应用名称 / App ID / App Secret
- 启用开关
- 保存更改
- 测试连接
- 重新连接
- 启用 / 停用
- 删除

如果当前飞书应用由运行时参数或外部配置接管，则会进入明确只读态：

- 字段不可编辑
- 保存和删除不可操作
- 页面显式说明只读原因

#### 能力摘要

默认按用户能理解的能力口径收敛成 4 张摘要卡：

1. `当前状态`
2. `连接验证`
3. `基础对话与交互`
4. `Markdown 预览`

这里不再把旧 wizard 步数直接当主界面中心叙事。

#### 待处理项

如果当前飞书应用还有待补齐项，admin 会在同一页面内直接给出待处理列表，例如：

- 先完成连接验证
- 导入基础权限
- 补齐事件订阅
- 补齐卡片回调
- 补齐飞书应用菜单
- 做一次发布验收
- 按需开通 Markdown 预览权限

当前能直接在 admin 内复用的旧后端能力包括：

- `PATCH /api/admin/feishu/apps/{id}/wizard`
  - 用于记录基础权限、事件、卡片回调、菜单等手工确认项
- `POST /api/admin/feishu/apps/{id}/publish-check`
  - 用于重新检查发布验收

因此管理页里的“我已完成”“检查发布状态”都已经不再要求回 setup 继续。

#### 新增飞书应用

当前“新增飞书应用”已经不是单一路径表单，而是两条明确分流：

- `新建飞书应用`
  - 管理页创建 onboarding 会话
  - 展示二维码
  - 自动轮询扫码状态
  - 扫码完成后显示一张说明页，再回到新飞书应用详情
- `接入已有飞书应用`
  - 继续保留手动填写 `App ID / App Secret` 的路径
  - 不会被强行改成“扫码选择已有 app”

扫码新建路径下：

- 显示名称会尽量从飞书侧带回；如果当前拿不到，就先回退到 `App ID`
- 说明页会明确提示：如果需要 Markdown 预览上传到飞书云盘，还要额外申请 `drive:drive`

#### 接入说明与技术详情

普通用户默认不需要看这里，但在 admin 内已经直接提供：

- 基础权限导入 JSON
- 事件订阅清单
- 卡片回调清单
- 飞书应用菜单清单
- 内部标识、是否已保存 Secret、配置来源、记录中的接入时间

也就是说，当前 admin 已经能在一个页面里同时承载：

- 稳态运维
- 新增飞书应用
- 手工补齐现有后端支持的能力项

而不再依赖 `继续完成首次配置` 这条旧入口。

### 3.3 工作实例

工作实例区域聚焦一个任务：

1. 查看当前可接入的 VS Code 工作实例

默认文案使用用户任务口径：

- `当前实例`
- `当前策略`
- `后台恢复实例由系统自动管理`

实例卡片默认展示：

- 实例名称
- 在线 / 启动中 / 异常 / 离线
- 工作目录
- 实例来源
- 最近状态摘要

这里不再提供“新建实例”或“删除实例”操作。

### 3.4 文档与图片

该区域分成两块：

1. `图片暂存`
2. `文档预览`

图片暂存默认展示：

- 暂存文件数
- 占用空间
- 当前仍在使用的文件数

并提供 `清理旧图片`。

文档预览按飞书应用分别展示，每条飞书应用各自维护自己的预览目录。默认界面展示：

- 是否已经启用预览
- 文件数量
- 估算占用
- 最近使用时间
- `清理旧预览`

预览目录链接、授权对象数等信息统一收进卡片内的技术详情折叠区。

当后端返回旧版“机器人”口径的提示时，前端当前会在展示层改写为“飞书应用”，避免旧 wording 回流到新界面。

### 3.5 VS Code

VS Code 区域默认使用用户能理解的口径，而不是直接暴露内部模式名。

默认界面展示：

- 当前环境是本机还是远程 SSH 机器
- 当前是否已接入
- 当前建议路径
- 是否需要重新安装扩展入口

主交互按场景组织：

- 如果当前是远程 SSH 机器：
  - 只提供“在这台远程机器上启用 VS Code”这一路径
  - 若还没检测到远程扩展入口，会直接显示阻塞说明
- 如果当前不是 SSH 机器：
  - 先让用户选择使用场景：
    - `要在当前这台机器上使用`
    - `主要去别的 SSH 机器上使用`
  - 然后再给当前场景唯一合理的主按钮和结果提示
  - “要在当前这台机器上使用”统一只处理扩展入口，不再写本机 `settings.json`
  - “主要去别的 SSH 机器上使用”会在当前机器明确跳过，而不是误写本机 `settings.json`

底层模式型操作被折叠到 `高级处理`：

- `只处理扩展入口`
- `重新安装扩展入口`
- `查看技术详情`

### 3.6 技术详情

技术详情是单独分区，不再与主流程混排。

当前集中收纳：

- 本机配置路径与地址
- 当前选中飞书应用的内部字段
- 完整权限导入 JSON
- 飞书应用连接原始状态

这样主界面保留产品化信息层，底层信息仍可在排障时直接查看。

## 4. 本轮收敛掉的默认暴露项

本轮实现中，这些词或概念已不再占据主界面默认文案：

- `继续完成首次配置`
- `wizard progress`
- `gateway`
- `reconcile`
- `managed shim`
- `surface`
- `remote queue`
- `机器人`

其中确实仍有排障价值的内容，被移动到了技术详情或折叠区。

## 5. 实现边界

### 5.1 当前主状态仍是前端派生

当前页面上的这些用户态：

- `可用`
- `可用，待补全`
- `待验证`
- `待同步`
- `外部接管`

仍然主要基于现有 admin API 返回的旧字段派生，例如：

- `wizard.*`
- `status.state`
- `runtimeApply`
- `readOnly / runtimeOnly / runtimeOverride`
- preview drive 摘要

这意味着：

- 现在已经实现了新的 admin 外观
- 但最终统一 truth model 仍然要由后续 issue 继续补齐

### 5.2 preview reconcile 已下线

预览目录 `reconcile` 不再作为管理页能力保留。

原因是 preview drive 已切到 fixed-root remote-first 语义：

- status 直接围绕远端 inventory 汇总
- cleanup 直接围绕远端 inventory 生效
- “本地对账”不再提供额外产品价值

### 5.3 当前改版以“新外观接旧能力”为主

这轮改版主要收敛的是：

- 信息架构
- 默认文案
- 操作入口
- 在 admin 内复用现有 wizard / publish-check 能力
- 旧 wording 的默认消隐

但并未借机重做 daemon 侧整体状态结构或一次性补齐新的能力检测 API。

### 5.4 移动端与相对路径约束已经进入实现基线

当前落地约束如下：

- 同一套 shell 同时服务桌面端与移动端
- 本地资源、本地跳转和本地 API 都按相对路径生成
- admin 与 setup 都不再假设部署在根路径
