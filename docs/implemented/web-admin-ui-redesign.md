# Web 管理界面重设计

> Type: `implemented`
> Updated: `2026-04-29`
> Summary: 当前 admin 的用户可见合同以 `v1.7.0` 页面结构为基线，默认主界面只保留机器人管理、系统集成、Claude 配置、存储管理四块，不再承载共享 onboarding 叙事。

## 1. 文档定位

本文记录当前已经落地的 `web admin` 页面合同。

它只回答三件事：

1. 默认主界面固定显示哪些区域。
2. 每个区域服务什么用户任务。
3. 哪些后来出现的额外叙事和额外分区不应该再回到主界面。

## 2. 页面合同

### 2.1 最终用户

- 正在使用本机管理页的普通用户
- 需要维护飞书应用、机器集成和存储占用的日常使用者

### 2.2 当前页面只服务什么任务

- 查看和维护现有机器人
- 新增一个机器人
- 处理自动运行和 VS Code 集成
- 管理 Claude 配置
- 清理预览、图片暂存和日志文件

### 2.3 允许出现的信息类型

- 机器人名称、App ID、连接状态、验证状态
- 机器人是否有异常以及最短处理入口
- 自动运行与 VS Code 的当前状态
- Claude 配置列表与可编辑字段
- 存储占用摘要与清理动作

### 2.4 不允许默认出现的信息类型

- 共享 onboarding workflow 总览
- “当前推荐处理”“剩余建议项”这类全局工作流叙事
- 工作实例、运行概览、技术详情等额外分区
- 大段设计说明、调试说明、内部状态说明

## 3. 已实现的信息架构

当前管理页固定分为 4 个区域：

1. `机器人管理`
2. `系统集成`
3. `Claude 配置`
4. `存储管理`

页面顶部仍沿用 `v1.7.0` 的简短口径：

- 标题：`<产品名> 管理`
- 副标题：`管理机器人、系统集成与本地存储。`

`Claude 配置` 是在 `v1.7.0` 主页面形态上新增的唯一默认分区。除此之外，不再追加其他默认分区。

## 4. 各区块行为

### 4.1 机器人管理

- 结构保持 `左侧列表 + 右侧详情`。
- 左侧只显示现有机器人和 `新增机器人`。
- 现有机器人默认只在异常时打标，不在成功时展示额外流水。
- 右侧详情只显示：
  - App ID
  - 连接状态
  - 启用状态
  - 最近验证
  - 权限异常提示
  - `测试事件订阅`
  - `测试回调`
  - `删除机器人`

新增机器人时，右侧只展示 `扫码创建 / 手动输入` 两条连接路径。这个区域允许长得和 setup 里的“飞书连接”很像，但它仍然属于 `admin` 内的新增机器人动作，不扩展成共享 onboarding 主界面。

### 4.2 系统集成

- 保持 `v1.7.0` 的两张主卡：
  - `自动运行设置`
  - `VS Code 集成`
- 自动运行只显示当前状态，必要时提供 `启用自动运行`。
- VS Code 只显示当前状态，保留 `重新检查并修复`。

这里不再加入额外的机器决策工作流、阶段总结或全局提示面板。

### 4.3 Claude 配置

- 当前管理页直接提供独立的 `Claude 配置` 区域。
- 这块是 `v1.7.0` 之后新增的唯一主界面功能。
- 该区域负责：
  - 展示默认配置
  - 展示自定义配置列表
  - 创建、编辑、删除自定义配置
- 这块默认允许展示 `Claude` 相关配置字段，因为这就是该区域正在服务的当前任务。

### 4.4 存储管理

- 保持 `v1.7.0` 的三张卡：
  - `预览文件`
  - `图片暂存`
  - `日志文件`
- 每张卡只展示占用摘要和清理动作。
- 不在默认主界面再展开额外技术说明。

## 5. 明确不回归的内容

以下内容不应再回到默认 admin 主界面：

- 共享 onboarding 组件驱动的整页叙事
- 全局 workflow 概览
- “当前推荐处理”“剩余建议项”之类的附加文字
- 运行概览
- 工作实例
- 技术详情
- 任何默认展开的大段解释性文案

## 6. 维护约束

- 以后继续调整 admin 时，默认以 `v1.7.0` 页面结构作为基线。
- 如需新增默认主界面分区，必须先更新本文档和对应 mock，再改产品页面。
- `Claude 配置` 之外，不应再把低频或技术型能力直接抬回默认主界面。
