# Web 管理界面重设计

> Type: `draft`
> Updated: `2026-04-07`
> Summary: 重写本地管理页的产品方案，收敛信息层级、多飞书 App 管理流程，以及与 Web Setup 的复用边界。

## 1. 文档定位

这份文档只讨论 `web admin` 管理界面，不讨论 `web setup` 向导。

它回答四件事：

1. 管理页应该服务哪些日常任务。
2. 页面上哪些信息该展示，哪些不该默认展示。
3. 多个飞书机器人应该怎么增删改和观察状态。
4. 哪些逻辑应该与 `web setup` 复用，避免两套实现继续分叉。

相关文档：

- [web-setup-wizard-redesign.md](./web-setup-wizard-redesign.md)
- [web-install-admin-prerequisites-design.md](./web-install-admin-prerequisites-design.md)

## 2. 这次重设计要解决的问题

当前管理页已经能做很多事，但主要问题不是“能力不够”，而是“展示方式不对”。

### 2.1 当前页面的主要问题

1. 混入了太多内部术语。
2. 把 setup 期信息、运行期信息、排障期信息混在同一层。
3. 展示了大量用户不需要理解、但一眼看上去很重的技术字段。
4. 多飞书 App 虽然已经能管理，但交互仍像“配置面板”，不像“机器人工作台”。
5. 一些内容是工程实现细节，不应该直接作为用户文案。

### 2.2 当前页面里应避免直接暴露的内容

默认视图里不应突出这些词或结构：

- `gateway`
- `surface`
- `remote queue`
- `trusted loopback`
- `session cookie`
- `wizard progress`
- `scopes JSON`
- `reconcile`
- `managed shim`
- `bundle entrypoint`

这些词不是完全不能出现，而是不应该出现在主流程和默认文案里。若确实有排障价值，应收进“技术详情”或“高级信息”。

## 3. 管理页的目标

管理页的核心目标应该收敛成五件事：

1. 一眼看出系统现在是否健康。
2. 方便管理多个飞书机器人。
3. 方便管理本机工作实例。
4. 方便清理与观察文档/图片相关存储。
5. 在需要时查看 VS Code 集成是否正常。

它不是：

- 首次安装向导
- 全量技术状态 dump 页
- 协议调试台

## 4. 目标用户任务

### 4.1 高频任务

1. 查看哪些机器人在线、哪个异常。
2. 新增一个机器人。
3. 修改某个机器人的凭证或名称。
4. 临时停用某个机器人。
5. 重连某个机器人。
6. 创建或删除一个工作实例。
7. 清理图片临时文件或预览文档。

### 4.2 中频任务

1. 检查某个机器人是否还没完成首次配置。
2. 查看某个机器人最近一次连接失败原因。
3. 查看 VS Code 集成是否需要处理。

### 4.3 低频任务

1. 排查更深的连接问题。
2. 查看底层监听地址、路径、运行细节。

这类低频任务不该占据首页主视觉。

## 5. 信息架构

建议管理页收敛成 5 个分区，顺序按“最常用 -> 最少用”排列：

1. 总览
2. 飞书机器人
3. 工作实例
4. 文档与图片
5. VS Code

另加一个不默认展开的：

6. 技术详情

## 6. 页面设计

### 6.1 总览

总览只回答“现在需不需要处理什么”。

建议展示 4 个摘要卡片：

1. 机器人总数 / 在线数
2. 需要处理的异常数
3. 工作实例总数 / 运行中数量
4. 存储告警数量

下面只保留一个“需要关注”的列表，而不是技术状态流水账。

列表每行只展示：

- 名称
- 当前状态
- 最近问题摘要
- 直接操作入口

例如：

- `机器人 A 连接失败，点击重连`
- `机器人 B 尚未完成首次配置，点击继续配置`
- `VS Code 集成需要重新安装，点击处理`

### 6.2 飞书机器人

这是管理页的核心区域。

#### 6.2.1 结构

采用“左侧列表 + 右侧详情”。

左侧列表只展示：

- 机器人名称
- 启用/停用
- 连接状态
- 是否需要关注

右侧详情分成三个块：

1. 基本信息
2. 当前状态
3. 可执行操作

#### 6.2.2 基本信息

默认只允许维护这些字段：

- 名称
- App ID
- App Secret
- 是否启用

`Gateway ID` 不应作为普通主文案出现。

如果必须保留稳定标识，应放在“高级信息”里，文案改成：

- `内部标识`

#### 6.2.3 当前状态

当前状态建议只展示对用户有意义的结果，而不是 setup 过程时间线。

建议保留：

- 当前连接状态
- 最近验证时间
- 最近错误
- 是否已完成首次配置

不建议在管理页默认展示：

- `凭证已保存`
- `Scopes 已导出`
- `事件已确认`
- `回调长连接已确认`
- `菜单已确认`
- `机器人已发布`

这些更适合 setup 流程，或在“首次配置未完成”的专用说明里出现。

#### 6.2.4 操作区

对单个机器人建议保留这些主操作：

- 保存
- 测试连接
- 重连
- 停用 / 启用
- 删除

若机器人处于“首次配置未完成”状态，再额外给一个：

- `继续完成首次配置`

它跳去 setup 或打开对应指导面板，而不是在管理页继续展开整套向导。

#### 6.2.5 只读机器人

如果机器人来自运行时覆盖，管理页应进入明确只读态：

- 字段不可编辑
- 保存按钮不可点
- 页面显式提示“该机器人由当前启动参数接管，只能查看状态”

不能让用户填完再在提交时失败。

### 6.3 新增机器人流程

建议流程：

1. 点击 `新增机器人`
2. 填写名称、App ID、App Secret
3. 保存
4. 立即执行一次连接测试
5. 测试通过后回到详情页
6. 如果它是首次接入，再提示 `继续完成首次配置`

设计重点：

1. 新增流程只解决“先把机器人接进来”。
2. 首次发布、权限、菜单这些仍由 setup 引导，不在管理页强行展开。
3. 管理页负责“持续管理”，setup 负责“首次带通”。

### 6.4 工作实例

工作实例区域只处理实例层面的日常操作。

建议保留：

- 实例名称
- 工作目录
- 在线状态
- 是否托管
- 删除实例

不建议默认展示太多实现字段。

如果需要更细信息，可以放到“查看详情”里。

### 6.5 文档与图片

这个区域的主要价值是“知道有没有积压”和“能不能清理”。

建议分成两个小卡：

1. 图片临时文件
2. 预览文档

每张卡默认展示：

- 当前大小
- 文件数量
- 最近使用时间
- 清理操作

按机器人分别展示预览文档是合理的，但不应默认使用“对账 / reconcile”作为主按钮文案。

建议改成：

- `检查异常`

若用户点进去，再在二级说明里解释它会检查什么。

### 6.6 VS Code

管理页里只需要回答：

1. 当前集成是否可用。
2. 是否有推荐动作。

建议默认只展示：

- 当前状态
- 推荐动作
- 一键处理按钮

更底层的路径、bundle、shim 信息应折叠在“技术详情”里。

## 7. 交互原则

### 7.1 先结果，后原理

先告诉用户：

- 现在是否正常
- 需要做什么

再告诉用户底层原因。

### 7.2 先单对象操作，后全局细节

用户管理多个机器人时，最自然的心智模型是“我在看第几个机器人”，而不是“我在看一堆系统状态”。

### 7.3 管理页不复用 setup 的步骤心智

管理页可以复用 setup 的能力，但不应该复用 setup 的“步骤条”和“流程完成度”。

因为用户来管理页时，目标通常不是“把一个应用从 0 带到 1”，而是：

- 看状态
- 改配置
- 处理问题

### 7.4 高级信息默认折叠

需要排障的技术信息可以保留，但默认折叠，统一放到：

- `技术详情`

里。

## 8. 与 Web Setup 的复用策略

这里唯一需要明确的技术点是：不要让 setup 和 admin 分别维护两套机器人配置逻辑。

### 8.1 后端复用

应抽出统一的“机器人配置能力”，供两个页面共用：

- 读取机器人列表
- 创建 / 更新 / 删除
- 连接测试
- 启用 / 停用
- 重连
- 只读态判定
- 状态汇总

setup 和 admin 的差别只在“页面引导方式”和“调用顺序”，不在底层数据处理。

### 8.2 前端复用

应复用这些前端模块：

- 机器人基础表单
- 连接测试结果展示
- 状态 badge
- 只读态处理
- 错误提示与确认交互

setup 在这些共享能力外，再包一层“逐步引导”；admin 则包一层“列表 + 详情 + 日常操作”。

## 9. 不在本轮实现的内容

这轮不建议把管理页继续做成：

1. 全量调试台
2. 线程 / queue / surface 级别操作台
3. 首次安装向导的替代品

如果未来要补更深的排障能力，建议单独做：

- `诊断页`
- `技术详情抽屉`

而不是继续往主管理页堆字段。
