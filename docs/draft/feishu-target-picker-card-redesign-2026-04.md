# 飞书 Target Picker 卡片重设计图

> Type: `draft`
> Updated: `2026-04-19`
> Summary: 按飞书 business card interaction principles 第 5 节，重做“选择工作区与会话”owner card 的逐步卡片设计，覆盖已有工作区、本地目录、Git URL、子页面、执行态与终态，并把可预判失败前置拦截。

## 1. 参考基线

本次重设计直接以：

- `docs/general/feishu-business-card-interaction-principles.md`

的 **第 5 节“展示层原则”** 为主基线，尤其采用下面 4 条：

1. 阶段头要短、单行、弱化
2. 当前上下文默认不展示
3. 单一主问题优先纵向布局
4. 横向布局只在被迫并存多个问题时使用

同时对照当前实现：

- `internal/core/control/feishu_target_picker.go`
- `internal/core/orchestrator/service_target_picker.go`
- `internal/core/orchestrator/service_target_picker_add_workspace.go`
- `internal/core/orchestrator/service_target_picker_git_import.go`
- `internal/adapter/feishu/projector_target_picker.go`
- `internal/adapter/feishu/projector_path_picker.go`

## 2. 当前卡片里设计不对的地方

### 2.1 把多个阶段的问题同时塞在同一页

当前 editing 卡在多数情况下会同时出现：

- 模式切换
- 工作区 / 会话选择
- 或 来源切换
- 或 Git / 目录的具体配置

这导致一张卡片同时在问：

- “你要走哪个分支？”
- “你要选哪个目标？”
- “你要怎么创建目标？”

这直接违反“单一主问题优先纵向布局”。

### 2.2 默认展示当前上下文摘要，挤占首屏空间

当前 `targetPickerSummaryMarkdown(...)` 会在 editing 卡默认回显：

- 当前工作区
- 当前会话
- 已填的目录 / Git 信息

这会让真正需要用户回答的问题被推到更下面，违反“当前上下文默认不展示”。

### 2.3 单一问题仍使用横向按钮组

当前：

- `模式`
- `工作区来源`

都用横向按钮组表达。

但这两个阶段本质上都是单一选择题，更适合纵向列表，不应为了“像 tab”硬做横排。

### 2.4 阶段头和说明层级不稳定

当前卡片会混用：

- 大标题
- `**模式**` / `**工作区**` / `**工作区来源**`
- `AddModeSummary`
- `AddModeDetail`
- `Hint`
- 警告 / 错误块

结果是：

- 阶段头不够短
- 首屏层级不稳定
- 说明性文案太多
- 用户不容易一眼看出“这一步到底让我做什么”

### 2.5 子页面虽然技术上 inline，但体验上仍像跳出原流程

当前 path picker 已经是 owner-card 内联返回，但卡面本身仍然很像一张独立工具卡：

- `允许范围`
- `当前目录`
- `当前选择`

三块摘要堆在顶部，太像“文件系统浏览器”，而不像“这一步里的一个子页面”。

### 2.6 Git 页面的信息密度仍然偏乱

当前 Git 编辑态会同时堆：

- 落地目录摘要
- 选择目录动作
- Git URL 输入
- 本地目录名输入
- 最终路径
- Hint
- source messages
- 底部按钮

这些内容并非不能同页，但现在缺少清晰的主次顺序，所以视觉上还是显得“挤”。

### 2.7 已知会失败的情况没有尽量拦在前面

当前交互里，部分可以在编辑态就确定会失败的条件，仍然会被留到执行后才报错，例如：

- Git 最终目标目录已经存在
- 落地目录不可访问
- 本地目录不是有效目录
- 目标目录当前被其他飞书会话接管

这类情况如果还能让用户继续点下一步或确认，体验上会像“先答完题，再告诉你这题根本不能提交”。

## 3. 新版工作流拆法

这条业务流继续保持 **一张 owner card 贯穿整个流程**，但卡内拆成明确页面：

### 3.1 主流程

1. `模式`
2. `目标`
3. `来源`
4. `目录`
5. `Git`

### 3.2 子页面

1. `目录/选择目录`
2. `Git/选择目录`

### 3.3 执行态

1. `目标/处理中`
2. `目录/处理中`
3. `Git/处理中`

### 3.4 终态

1. `已完成`
2. `失败`
3. `已取消`

## 4. 新版卡片设计规则

### 4.1 固定卡面骨架

每一页都统一成：

1. 顶部业务标题：`选择工作区与会话`
2. 次级阶段头：例如 `模式/目标`
3. 单一主问题
4. 本步输入区
5. 本步校验 / 提示区
6. 本步按钮区

### 4.2 当前上下文默认移除

默认不展示：

- 当前工作区
- 当前会话
- 已 attach 状态摘要

只有在 processing / terminal 时，才回显最小必要对象摘要。

### 4.3 单题页面一律纵向

以下页面全部纵向布局：

- `模式`
- `来源`
- `目录`
- `Git`

`目标` 页面保留两个纵向字段：

- 工作区
- 会话

不再把它们和模式切换放在同一页。

### 4.4 按钮语义固定

- `返回`
  - 只用于子页面返回主页面
- `上一步`
  - 只用于业务步骤回退
- `取消`
  - 取消整个 target picker owner flow

### 4.5 可预判失败要前置拦截

凡是能在当前编辑页就判断出“继续一定会失败”的条件，都应该在当前页直接拦住，而不是放到 processing 或失败终态里再报。

典型例子：

- Git 最终目标目录已存在
- 落地父目录不可写或不可进入
- 选中的本地路径不存在 / 不是目录
- 目录当前正被其他飞书会话接管，无法接入

交互要求：

- 当前页直接显示阻塞错误
- 主按钮置灰，不允许继续
- 用户修正输入后，阻塞错误即时消失

### 4.6 其他错误继续走统一失败终态

对那些无法在编辑态提前确认的错误，例如：

- 运行时 Git clone 失败
- 执行过程中网络异常
- 接入工作区时底层服务返回错误

这次不额外扩展复杂错误分流，继续沿用统一的 processing -> 失败终态处理。

### 4.7 执行态进度用 emoji，不用文字括号态

处理中阶段列表不再使用：

- `[x]`
- `[>]`
- `[ ]`

统一改成更直观的状态符号：

- `✅` 已完成
- `🔄` 进行中
- `⚪` 未开始

## 5. 逐步卡片设计图

下面所有图都不是实现截图，而是用户最终看到的结构草图。

### 5.1 页面 1：模式

适用：

- 所有入口进入 owner card 的第一页

```text
[Card]
Title: 选择工作区与会话
Stage: 模式

这次要做什么？

(●) 进入已有工作区
    切到一个已有工作区里的某个会话

(○) 新建工作区
    接入目录或克隆仓库后，直接进入一个新会话

[取消]                              [下一步]
```

设计说明：

- 这里只有一个主问题，所以必须纵向。
- 不展示当前工作区 / 当前会话。
- 如果当前已经在某个工作区里，默认选中 `进入已有工作区`。

### 5.2 页面 2A：目标

适用：

- `模式 = 进入已有工作区`

```text
[Card]
Title: 选择工作区与会话
Stage: 模式/目标

切到哪个工作区 / 会话？

工作区
[ 最近活跃的工作区 A · 2 分钟前 v ]

会话
[ 修 CI 的那条会话 · 空闲 v ]

[取消]                [上一步]                [确认切换]
```

设计说明：

- `工作区` 和 `会话` 仍保留同页，因为它们共同回答“切到哪个目标”。
- 页面内不再出现 `模式` 切换控件。
- 当会话选择为“新会话”时，右下角按钮文案动态改成 `进入新会话`。

### 5.3 页面 2B：来源

适用：

- `模式 = 新建工作区`

```text
[Card]
Title: 选择工作区与会话
Stage: 模式/来源

新工作区从哪里来？

(●) 已有目录
    把本机上已经存在的目录接入成工作区

(○) 从 Git URL
    克隆仓库到本机，再接入成工作区

[取消]                [上一步]                [下一步]
```

设计说明：

- 这里只有一个主问题，所以继续纵向。
- 不在这里提前出现目录路径、Git URL、最终路径。
- 如果 `git` 不可用，这一页仍可展示 `从 Git URL`，但整项置灰并在下一行显示：
  - `当前机器未检测到 git，暂时不能使用这个来源`
- 这类可在当前页判断的阻塞条件，直接在这一页拦住，不进入下一步后再失败。

### 5.4 页面 3A：目录

适用：

- `模式 = 新建工作区`
- `来源 = 已有目录`

```text
[Card]
Title: 选择工作区与会话
Stage: 模式/来源/目录

要接入哪个本地目录？

目录
/workspace/projects/my-project

[选择目录]

提示
- 该目录已是已有工作区；继续后会复用该工作区并进入新会话

[取消]                [上一步]                [接入并继续]
```

阻塞错误态示例：

```text
提示
- 该目录对应的工作区当前正被其他飞书会话接管，暂时不能继续
```

设计说明：

- 顶部只保留一个字段：`目录`。
- 把“说明文案”收缩到提示区，不再额外放一块 AddModeSummary / Hint。
- `接入并继续` 未满足条件时置灰。
- `目录不存在`、`不是目录`、`当前不可访问`、`被其他飞书会话接管` 都属于前置阻塞，不进入 processing。

### 5.5 页面 3A-1：目录子页面

适用：

- 从页面 `目录` 点 `选择目录`

```text
[Card]
Title: 选择工作区与会话
Stage: 目录/选择目录

选择要接入的目录

范围：/workspace

当前位置
/workspace/projects

进入目录
[ .. / my-project / another-repo / tmp v ]

[返回]                              [使用这个目录]
```

设计说明：

- 这里明确是“目录步骤里的子页面”，不是另起一条业务卡。
- 只保留一行范围信息，不再堆 `允许范围 / 当前目录 / 当前选择` 三段摘要。
- `返回` 的语义是回到目录主页面，不是取消整个 target picker。

### 5.6 页面 3B：Git

适用：

- `模式 = 新建工作区`
- `来源 = 从 Git URL`

```text
[Card]
Title: 选择工作区与会话
Stage: 模式/来源/Git

克隆哪个仓库，到哪里？

落地父目录
/workspace/projects
[选择目录]

Git URL
https://github.com/org/repo.git

本地目录名（可选）
repo-work

最终路径
/workspace/projects/repo-work

提示
- 该目录下已有其他内容；导入时会在其中创建新的子目录

[取消]                [上一步]                [克隆并继续]
```

阻塞错误态示例：

```text
提示
- 目标目录已存在：/workspace/projects/repo-work，请更换落地目录或本地目录名
```

设计说明：

- 这页虽然内容多，但仍然只回答一个主问题：“把哪个仓库克隆到哪里”。
- `最终路径` 只做结果预览，不再上升成和输入字段同级的大块摘要。
- `提示` 区只承载本步相关校验。
- `克隆并继续` 只有在 Git URL、落地父目录、最终路径都通过前置校验时才可点击。
- `目标目录已存在` 这类已知必败条件，直接在这页阻塞，不进入克隆执行态。

### 5.7 页面 3B-1：Git 落地目录子页面

适用：

- 从页面 `Git` 点 `选择目录`

```text
[Card]
Title: 选择工作区与会话
Stage: Git/选择目录

选择仓库要落到哪个本地父目录

范围：/workspace

当前位置
/workspace/projects

进入目录
[ .. / repos / projects / scratch v ]

[返回]                              [使用这个目录]
```

设计说明：

- 结构和目录子页面一致，只改主问题。
- 次级说明只保留到一句，不把路径浏览器做成独立产品。

### 5.8 页面 4A：目标处理中

适用：

- 已有工作区 / 会话 confirm 之后

```text
[Card]
Title: 选择工作区与会话
Stage: 目标/处理中

正在切换工作区 / 会话

对象
- 工作区：workspace-a
- 会话：修 CI 的那条会话

当前阶段
- 🔄 切换目标
- ⚪ 完成
```

设计说明：

- 快路径不需要额外展示过多上下文。
- 默认不提供取消按钮。

### 5.9 页面 4B：目录处理中

适用：

- 本地目录 confirm 之后

```text
[Card]
Title: 选择工作区与会话
Stage: 目录/处理中

正在接入工作区

对象
- 目录：/workspace/projects/my-project

当前阶段
- ✅ 校验目录
- 🔄 接入工作区
- ⚪ 准备新会话
```

### 5.10 页面 4C：Git 处理中

适用：

- Git confirm 之后

```text
[Card]
Title: 选择工作区与会话
Stage: Git/处理中

正在导入 Git 工作区

对象
- 仓库：https://github.com/org/repo.git
- 目标：/workspace/projects/repo-work

当前阶段
- ✅ 校验参数
- 🔄 克隆仓库
- ⚪ 接入工作区
- ⚪ 准备新会话

最近输出
- 正在执行 git clone ...

普通输入已暂停。你可以等待完成，或取消本次导入。

[取消导入]
```

设计说明：

- 这里只有最小必要的对象摘要、阶段进度、最近输出。
- 继续使用同一张 owner card，不再额外弹“命令已提交”或旁路 notice 卡。

### 5.11 页面 5A：成功终态

适用：

- 目标切换成功
- 本地目录接入成功
- Git 导入成功

```text
[Card]
Title: 选择工作区与会话
Stage: 已完成

已进入新会话待命

结果
- 工作区：workspace-a
- 会话：新会话

下一步
- 直接发送下一条文本即可开始工作
```

已有会话切换成功变体：

```text
[Card]
Title: 选择工作区与会话
Stage: 已完成

已切换会话

结果
- 工作区：workspace-a
- 会话：修 CI 的那条会话

下一步
- 现在发送的文本会进入这个会话
```

### 5.12 页面 5B：失败终态

适用：

- 任一分支执行失败

```text
[Card]
Title: 选择工作区与会话
Stage: 失败

导入失败

停在阶段
- 克隆仓库

原因
- 目标目录已存在：/workspace/projects/repo-work

下一步
- 修改落地目录或本地目录名后重试
```

设计说明：

- 失败终态必须告诉用户“卡在哪一步”和“现在该怎么做”。
- 不再只给一段泛化失败文案。

### 5.13 页面 5C：取消终态

适用：

- 用户主动取消整个 target picker

```text
[Card]
Title: 选择工作区与会话
Stage: 已取消

已取消

结果
- 当前选择流程已结束
- 工作目标保持不变
```

设计说明：

- `取消` 不回菜单，不保留可继续编辑的半死表单卡。
- 直接在同一卡上封住生命周期。

## 6. 对当前渲染实现的直接改造结论

如果按上面的卡图落地，当前 target picker 投影层需要直接改掉这些点：

1. editing 页默认去掉 `targetPickerSummaryMarkdown(...)`
2. `ShowModeSwitch` 只留在 `模式` 页面，不再在目标页、来源页、目录页、Git 页同时显示
3. `ShowWorkspaceSelect` / `ShowSessionSelect` 只留在 `目标` 页面
4. `ShowSourceSelect` 只留在 `来源` 页面
5. `AddModeSummary` / `AddModeDetail` 不再作为长期常驻块，改成页面主问题文案
6. `Hint` 不再作为全局兜底说明块，而是缩到每一步各自的提示区
7. path picker 顶部摘要从三段压缩为：
   - 一行范围
   - 一行当前位置
8. 编辑页引入前置阻塞校验，已知必败条件直接拦在当前页：
   - Git 目标目录已存在
   - 本地目录不存在 / 不是目录 / 不可访问
   - 目录正被其他飞书会话接管
9. processing 页的阶段进度统一改成 emoji 风格：
   - `✅` 已完成
   - `🔄` 进行中
   - `⚪` 未开始
10. processing / terminal 的阶段命名统一改成：
   - `目标/处理中`
   - `目录/处理中`
   - `Git/处理中`
   - `已完成`
   - `失败`
   - `已取消`

## 7. 这版图的直接收益

1. 一张卡一次只问一个阶段问题，用户不会在同页里同时做模式判断和字段填写
2. 首屏不再先被“当前上下文”挤掉
3. 模式 / 来源恢复成真正的单题页面
4. 子页面更像“这一步里的选择器”，而不是跳出原流程的工具卡
5. 已知必败条件会在前面被挡住，减少“点了继续才知道不能做”的挫败感
6. Git 执行态、取消、成功、失败的生命周期更清楚
7. 执行态进度比原来的文字括号态更直观
8. 后续实现时，页面结构与 owner-flow 语义是一一对应的，更不容易再回到“流程改了、卡还没改顺”的状态
