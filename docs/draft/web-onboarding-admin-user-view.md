# Web Setup And Admin User View

> Type: `draft`
> Updated: `2026-04-11`
> Summary: 细化 setup 中 bot 接入阶段的用户可见步骤，明确每一页应该展示什么、用户会怎么继续。

## 1. 文档定位

这不是技术 PRD。

这份文档只回答一个问题：

- 如果我是普通用户，我最终应该在 `web setup` 和 `web admin` 里看到什么？

它不讨论：

- 后端状态模型怎么设计
- 哪些字段来自 daemon / runtime / gateway
- 哪些能力靠检测，哪些能力靠手工确认

如果要看技术拆解，请回到：

- [web-onboarding-admin-workflow-prd.md](./web-onboarding-admin-workflow-prd.md)

## 2. 用户应该感受到的整体变化

新的体验应该让用户感受到这几件事：

1. `setup` 是一次性的开通向导，不是一个复杂控制台。
2. `admin` 是日常管理页，不是“继续做 setup”的中转站。
3. 页面会先告诉用户“现在已经能不能用”，而不是先让用户到处打卡。
4. 如果某个 bot 已经在别处配好了，页面应该识别出来，并直接告诉用户“这个已经能用”。
5. 如果某些增强项还没补齐，页面应该明确告诉用户“现在已经能用，但还有这些可以稍后再补”，而不是一直说“你还没完成首次配置”。

## 3. 用户进入 setup 时，应该先看到什么

用户一进 `setup`，第一眼不应该看到一长串步骤，也不应该看到大量技术字段。

用户第一眼应该看到的是一张很清楚的结果页，回答四个问题：

1. 这台机器现在能不能开始用？
2. 系统有没有检测到已经存在的 bot？
3. 当前检测到的 bot 是正常可用、部分可用，还是根本不可用？
4. 我接下来最应该点哪个按钮？

页面上应该只有少量明确动作：

- `开始接入 bot`
- `继续使用当前 bot`
- `修复当前 bot`

而不是一上来就让用户自己判断去权限、事件、菜单、发布、VS Code、自动启动里先点哪一个。

## 4. 三种最重要的用户场景

### 4.1 我是第一次安装，什么都没有

我进入 setup 后，应该看到：

- 这台机器还没有可用 bot
- 下一步需要先接入一个 bot

然后页面让我直接选择：

- `新建飞书应用`
- `接入已有飞书应用`

这时页面的目标很简单：

- 先把一个 bot 接上

而不是让我先理解整套系统结构。

### 4.2 我已经在别处把 bot 配好了

我进入 setup 后，应该看到：

- 系统已经检测到一个可用 bot
- 这个 bot 现在已经可以开始用

如果还有一些没补齐的增强项，页面应该写得很直白：

- `现在已经能开始使用`
- `还有一些增强项可以稍后补齐`

而不应该继续把我塞回“你还没完成首次配置”的旧流程里。

### 4.3 这个 bot 是外部接管的，不是本机来管理

我进入 setup 或 admin 后，应该看到：

- 这个 bot 当前已经被别的地方接管
- 它现在是只读状态
- 我能看当前状态，能知道它是否可用
- 但我不能在这里修改它的飞书平台配置

页面要把这件事说得很明确，避免用户反复点进一个根本改不了的本地向导。

## 5. setup 最终应该是什么流程

从用户视角看，setup 最终应该只有这五段。

### 5.1 第一步：先看结果

页面先告诉我：

- 有没有 bot
- bot 能不能用
- 机器环境有没有明显阻断项

这一步的重点是“判断现状”，不是“开始填表”。

### 5.2 第二步：决定我要用哪一个 bot

如果没有 bot，就让我选：

- 新建一个
- 接入已有的

如果已经有 bot，就明确告诉我当前目标是谁。

如果有多个 bot，页面必须直接显示：

- 当前选中的是哪一个
- 我能不能切换

不能静默帮我选一个，然后让我后面自己猜。

#### 5.2.1 用户先看到的是“选择 bot 来源”页

这一页的目标非常单纯：

- 让我决定，这个 bot 是怎么来的

页面上应该是几张很清楚的卡片，而不是一个技术表单。

用户应该看到：

- `新建飞书应用`
- `接入已有飞书应用`
- 如果系统已经检测到 bot，再加一个：`使用当前已检测到的 bot`

如果当前已经检测到 bot，这一页还应该直接写清楚：

- 当前检测到的是哪个 bot
- 它现在是可用、部分可用，还是不可用
- 它是不是只读 / 外部接管

用户在这一页只需要做一件事：

- 选一条路径，然后点 `下一步`

#### 5.2.2 如果选择“新建飞书应用”，下一页应该是扫码页

用户到了这一步，页面不应该再出现手填 App ID / Secret。

这一页应该非常明确地告诉用户：

- `请用飞书扫码创建应用`
- `系统会自动拿回这个应用的凭据`
- `完成后会自动继续验证连接`

用户在这一步应该看到：

- 一个清楚的二维码
- 一句很短的话，说明扫码是干什么的
- 一个等待中的状态提示，比如：`正在等待扫码完成`
- 两个辅助动作：
  - `刷新二维码`
  - `返回重新选择`

这一步不应该要求用户自己去猜：

- 扫完码以后是不是还要回来填表
- 系统到底有没有拿到凭据
- 下一步到底会发生什么

#### 5.2.3 扫码成功后，应该有一张“已经创建好”的说明页

扫码成功后，不应该直接把用户悄悄扔到别的地方。

页面应该明确告诉用户：

- `已经为你创建并接入飞书应用`
- 这个应用现在叫什么名字
- 如果暂时拿不到名字，至少显示 App ID

同时要把“已经自动做完了什么”写清楚，例如：

- `应用已经创建`
- `凭据已经带回`
- `当前 bot 已经接入本机`

如果还有后续动作，页面要说人话：

- `下一步我们会继续检查这个 bot 现在能不能直接开始使用`

这一页的主按钮应该只有一个：

- `继续`

#### 5.2.4 如果选择“接入已有飞书应用”，下一页应该是手动接入页

这一页的目标是：

- 把一个已经存在的飞书应用接到这台机器上

页面应该只出现用户真的需要填写的内容：

- `App ID`
- `App Secret`
- 可选的 `备注名称`

页面上要有一句用户能听懂的话：

- `如果你已经在飞书后台配好了这个应用，这一步只需要把它接进来。`

主按钮应该很明确：

- `保存并验证`

这一步做完以后，系统应该立刻给用户一个明确结果，而不是只提示“保存成功”。

#### 5.2.5 如果选择“使用当前已检测到的 bot”，下一页应该是只读确认页

这一页的重点不是编辑，而是确认。

用户应该看到：

- 当前 bot 的名字
- 当前 bot 的状态
- 它是不是外部接管 / 只读
- 它为什么会出现在这里

如果它是只读的，页面要直接说清楚：

- `这个 bot 已经存在，但不在这里修改飞书平台配置`

这一页可用的动作应该很少：

- `验证当前 bot`
- `换一个 bot`

而不应该又出现一套可编辑表单，把用户带回本地创建路径。

### 5.3 第三步：先把 bot 接通

这一步只做一件事：

- 确认当前 bot 已经真正接上了

用户能理解的完成信号应该非常简单：

- `连接成功，可以继续`

而不是让用户自己推测“凭证大概保存了，但好像还没完全通”。

#### 5.3.1 验证通过时，页面应该怎么说

验证通过以后，页面要明确给用户一个“已经接通”的结果。

最自然的表达应该类似：

- `连接成功，可以继续`
- `这个 bot 已经接到当前机器`

如果这是一个已存在的 bot，也应该直说：

- `当前检测到的 bot 可以继续使用`

不要只写成抽象的技术成功提示。

#### 5.3.2 验证失败时，页面应该怎么说

验证失败时，页面也要尽量说人话。

用户应该立刻知道两件事：

1. 这一步还没有完成
2. 失败的大概方向是什么

例如页面应该像这样表达：

- `连接还没有成功`
- `请检查 App ID / App Secret 是否正确`
- `请确认这个应用已经添加了机器人能力`

主按钮可以是：

- `重新验证`

辅助动作可以是：

- `返回修改`
- `换一个 bot`

而不是把一大段原始错误直接丢给用户自己理解。

### 5.4 第四步：告诉我哪些是必须修，哪些可以稍后

这是新体验最关键的一步。

页面应该把所有检查项分成两组：

- `必须先修好，否则现在没法正常用`
- `可以稍后再补，不影响现在开始使用`

例如用户应该看到类似这种表达：

- `基础消息链路：通过`
- `运行环境检查：通过`
- `机器人菜单：还没确认，可稍后处理`
- `卡片回调：还没确认，可稍后处理`

重点不是列很多技术名词，而是让用户立刻知道：

- 我现在卡在哪里
- 哪些现在必须做
- 哪些以后再做也没问题

### 5.5 第五步：处理这台机器自己的选择

bot 接通以后，页面再问机器级选项，比如：

- 自动启动
- VS Code 集成

而且这些选项都应该允许用户明确表达：

- `现在处理`
- `稍后再说`
- `去别的机器处理`

用户一旦做过决定，刷新后页面应该记得，不应该反复再问一遍。

## 6. 什么情况下应该允许完成 setup

从用户视角，完成 setup 的标准应该非常直白：

- 我已经有至少一个 bot 可以开始用
- 这台机器本身没有明显阻断问题
- 对机器级可选项，我已经做了决定

这就够了。

以下情况不应该继续阻止我完成 setup：

- 我没有在当前页面重新点过权限确认
- 菜单还没补
- 卡片回调还没补
- Drive 预览类增强能力还没补
- VS Code 决定以后再弄
- 自动启动决定稍后再开

这些都应该显示成：

- `现在已经能用，但还有待补全项`

而不是：

- `首次配置还没完成`

## 7. 完成页应该长什么样

完成页不应该像一个大总结报告。

完成页只需要清楚告诉用户三件事：

1. 现在已经可以开始使用
2. 如果有待补全项，它们是什么
3. 下一步去哪里

完成页最自然的内容应该类似：

- `已经可以开始使用 Codex Remote`
- `当前可用 bot：xxx`
- `待补全项：机器人菜单、卡片回调`
- `下一步：进入管理页`

## 8. admin 最终应该让用户看到什么

`admin` 的角色应该非常明确：

- 这里是日常管理页

用户进入 admin 后，应该先看到的是稳态信息，而不是重新进入 setup 向导。

最重要的内容应该是：

### 8.1 当前整体状态

例如：

- 这台机器现在是否可用
- 当前有哪些 bot
- 哪些 bot 正常可用
- 哪些 bot 还有待补全项
- 哪些 bot 已损坏或需要修复

### 8.2 每个 bot 的状态和下一步

每个 bot 都应该能让用户一眼看懂：

- `不可用`
- `可用`
- `可用，但还有待补全项`
- `外部接管`

同时要给出很直接的推荐动作，比如：

- `查看待补全项`
- `重新验证连接`
- `修复配置`
- `查看只读状态说明`

### 8.3 新增 bot 是 admin 里的一个独立入口

当用户想再接一个 bot 时，应该在 admin 里看到一个清楚的入口：

- `新增 bot`

点进去以后，进入的是 admin 内部的一个子流程，而不是被踢回 setup 再走一遍首次安装。

## 9. 用户不应该再看到的体验

从用户视角，下面这些体验都应该消失：

- 明明已经能用，但页面仍长期显示“未完成首次配置”
- 因为没有在当前页面点过确认，就被判定成“没完成”
- 外部接管 bot 被引导进入本地可编辑向导
- admin 里不断提示“回到 setup 继续完成”
- 机器级选项每刷新一次就重新问一遍
- 同一个页面同时堆满 setup、admin、多 App、技术状态和未来步骤

## 10. 用一句话概括最终体验

最终用户应该感受到的是：

- 系统先告诉我“现在能不能用”，再告诉我“还可以补什么”，最后才让我进入日常管理。
