# Web Setup And Admin User View

> Type: `draft`
> Updated: `2026-04-11`
> Summary: 为用户视角流程补齐各页面的示例文案，明确标题、副标题、状态提示与主要按钮应该怎么写。

## 1. 文档定位

这不是技术 PRD。

这份文档只回答一个问题：

- 如果我是普通用户，我从打开 `setup` 到以后日常进入 `admin`，整个产品流程应该看到什么？

它不讨论：

- 后端状态模型怎么设计
- 哪些字段来自 daemon / runtime / gateway
- 哪些能力是自动检测、哪些能力是手工补齐

本文约定：

- 对用户展示时，一律称为 `飞书应用`
- 内部实现里如果仍出现 `bot` 这个词，它在产品上对应的也是同一个“飞书应用”

如果要看技术拆解，请回到：

- [web-onboarding-admin-workflow-prd.md](./web-onboarding-admin-workflow-prd.md)

## 2. 用户应该怎么理解这个产品

从用户视角，整个产品其实只有两部分：

### 2.1 `setup`

`setup` 是一次性的开通向导。

它只负责一件事：

- 把这台机器带到“已经可以开始使用”的状态

### 2.2 `admin`

`admin` 是后续的日常管理页。

它负责：

- 看当前机器和飞书应用的状态
- 补做还没做完的增强项
- 修复有问题的飞书应用
- 再新增一个飞书应用

用户不应该再感受到：

- `setup` 是一个长期使用的大控制台
- `admin` 只是一个“回去继续做 setup”的中转页

## 3. 用户打开 setup 后，完整应该经历什么

从用户角度，完整主线应该是 8 段。

### 3.1 第一步：先检查这台机器能不能开始

这是整个流程最前面的硬门槛。

在这一步，页面不应该先让我设置飞书应用。

页面应该先回答：

1. 这台机器有没有最基本的运行条件？
2. 系统能不能检测到 `codex` 这类基础依赖？
3. 现在是不是连最基础的会话都跑不起来？

用户在这一步应该看到的是一个很直接的结果页，而不是技术细节列表。

页面应该给出三种清楚状态：

- `可以继续`
- `有提醒，但不阻断`
- `当前不能继续`

#### 3.1.1 如果基础环境没过，页面应该怎么表现

如果像下面这些基础条件没满足：

- 没检测到 `codex`
- 当前机器的基础运行条件不满足
- 关键路径明显无法工作

那么页面就应该直接停在这里。

这时不应该继续出现：

- `新建飞书应用`
- `接入已有飞书应用`
- 任何飞书应用设置流程

因为这些都还没有意义。

用户在这时应该看到：

- `这台机器还不能开始配置飞书应用`
- `请先修复基础环境问题`

同时页面应该明确列出阻断项，例如：

- `没有检测到 Codex`
- `当前运行环境检查没有通过`

每个阻断项下都应该有很简单的下一步说明，比如：

- `先安装或配置 Codex`
- `修复当前机器的运行环境后，再回来继续`

主按钮应该类似：

- `重新检查环境`

辅助动作可以是：

- `查看修复说明`

但不能让用户直接跳去飞书应用设置。

建议页面文案可以直接写成：

```text
标题：先把这台机器准备好
说明：在开始配置飞书应用之前，我们先确认这台机器已经满足最基本的运行要求。

状态卡标题：当前还不能继续
状态卡正文：这台机器还有基础环境问题没有解决。请先修复下面这些问题，再回来继续配置飞书应用。

阻断项 1：没有检测到 Codex
阻断项说明：系统暂时找不到 Codex。先把 Codex 安装好，或确认当前环境变量已经配置正确。

阻断项 2：运行环境检查没有通过
阻断项说明：这台机器当前还不能稳定运行会话。请先修复运行环境问题，再回来继续。

主按钮：重新检查环境
次按钮：查看修复说明
```

#### 3.1.2 如果基础环境过了，页面应该怎么说

如果基础环境已经满足，页面应该明确告诉用户：

- `这台机器已经满足基础要求，可以继续配置飞书应用`

这一步给用户的感觉应该是：

- 先确认地基没问题
- 之后再去处理飞书应用

而不是一开始就把飞书应用和机器问题混在一起。

建议页面文案可以直接写成：

```text
标题：这台机器已经准备好了
说明：基础环境检查已经通过。接下来可以继续配置飞书应用。

状态卡标题：可以继续
状态卡正文：这台机器已经满足基础要求。下面我们会先看看当前有没有已经可用的飞书应用。

主按钮：继续
```

### 3.2 第二步：看系统现在已经有什么

只有基础环境通过以后，才应该进入这一页。

这一页的目标不是配置，而是告诉用户当前现状。

页面应该回答：

1. 系统现在有没有检测到飞书应用？
2. 如果有，是几个？
3. 它们现在分别是什么状态？
4. 当前最建议我处理哪一个？

用户在这一页应该看到很直白的信息：

- `当前没有检测到可用飞书应用`
- `已经检测到 1 个可用飞书应用`
- `已经检测到 1 个飞书应用，但它现在有问题`
- `已经检测到一个外部接管的飞书应用`

如果有多个飞书应用，页面必须直接显示：

- 名字
- 当前状态
- 哪一个是当前推荐处理对象

而不能静默默认选中一个。

这一页的主动作应该很少，通常只有：

- `开始接入飞书应用`
- `继续使用当前飞书应用`
- `修复当前飞书应用`

建议页面文案可以直接写成：

```text
标题：先看一下当前状态
说明：我们先帮你看一眼，这台机器上现在有没有已经可用的飞书应用。

情况 A：当前没有检测到可用飞书应用
说明：你还没有可以直接使用的飞书应用。下一步需要先接入一个。
主按钮：开始接入飞书应用

情况 B：已经检测到 1 个可用飞书应用
说明：系统已经检测到一个可用飞书应用。你可以直接继续使用它，也可以先看看还有没有待补全项。
主按钮：继续使用当前飞书应用

情况 C：已经检测到 1 个飞书应用，但它现在有问题
说明：系统已经找到一个飞书应用，但它现在还不能正常使用。下一步建议先修复它。
主按钮：修复当前飞书应用
```

### 3.3 第三步：决定当前要处理哪一个飞书应用

到了这一步，用户才真正进入飞书应用接入流程。

这一步的目标是：

- 明确当前这次 setup，到底是处理哪个飞书应用

用户应该先看到一页“选择飞书应用来源”的页面。

页面上应该是几张卡片，而不是一堆字段。

用户应该看到：

- `新建飞书应用`
- `接入已有飞书应用`
- 如果已经检测到飞书应用，再加一个：`使用当前已检测到的飞书应用`

如果当前已经检测到飞书应用，这一页还应该直接写清楚：

- 当前检测到的是哪个飞书应用
- 它现在是可用、部分可用，还是不可用
- 它是不是只读 / 外部接管

用户在这一页只需要做一件事：

- 选一条路径，然后点 `下一步`

建议页面文案可以直接写成：

```text
标题：这次要处理哪个飞书应用？
说明：先选一条最适合你的路径。我们会只处理当前这个飞书应用，不会把多个飞书应用混在一起。

卡片 1：新建飞书应用
卡片说明：适合第一次接入。系统会引导你扫码创建，并自动带回凭据。

卡片 2：接入已有飞书应用
卡片说明：适合你已经在飞书后台建好应用，只需要把它接到这台机器上的情况。

卡片 3：使用当前已检测到的飞书应用
卡片说明：适合系统已经发现一个飞书应用，而且你想直接继续用它的情况。

主按钮：下一步
次按钮：返回上一步
```

### 3.4 第四步：把飞书应用接进来

这一步只做一件事：

- 让当前飞书应用真正接到这台机器上

不同路径下，页面应该长得不一样。

#### 3.4.1 路径 A：新建飞书应用

如果用户选择的是 `新建飞书应用`，接下来应该经历两页。

##### 第一页：扫码创建页

这一页应该非常明确地告诉用户：

- `请用飞书扫码创建应用`
- `系统会自动拿回这个应用的凭据`
- `完成后会自动继续验证连接`

用户在这一步应该看到：

- 一个清楚的二维码
- 一句很短的话，说明扫码是干什么的
- 一个等待中的状态提示，比如：`正在等待扫码完成`
- 两个辅助动作：
  - `刷新二维码`
  - `返回重新选择`

这一步不应该要求用户自己去猜：

- 扫完码以后是不是还要回来填表
- 系统到底有没有拿到凭据
- 下一步到底会发生什么

建议页面文案可以直接写成：

```text
标题：用飞书扫码创建应用
说明：请用飞书扫码。系统会自动创建应用、带回凭据，并在完成后继续验证连接。

辅助说明：如果二维码过期了，可以刷新后重新扫码。

状态提示：正在等待扫码完成

主按钮：我已扫码，继续等待
次按钮：刷新二维码
次按钮：返回重新选择
```

##### 第二页：扫码成功说明页

扫码成功后，不应该直接把用户悄悄扔到别的地方。

页面应该明确告诉用户：

- `已经为你创建并接入飞书应用`
- 这个应用现在叫什么名字
- 如果暂时拿不到名字，至少显示 App ID

同时要把“已经自动做完了什么”写清楚，例如：

- `应用已经创建`
- `凭据已经带回`
- `当前飞书应用已经接入本机`

如果还有后续动作，页面要说人话：

- `下一步我们会继续检查这个飞书应用现在能不能直接开始使用`

这一页的主按钮应该只有一个：

- `继续`

建议页面文案可以直接写成：

```text
标题：飞书应用已经创建好了
说明：我们已经帮你把这个应用接进来了。下一步会继续检查它现在能不能直接开始使用。

信息卡标题：当前应用
信息卡正文：应用名：开发者小助手
信息卡补充：如果暂时还拿不到名字，这里至少显示 App ID。

已完成：
- 应用已经创建
- 凭据已经带回
- 当前飞书应用已经接入本机

主按钮：继续
```

#### 3.4.2 路径 B：接入已有飞书应用

如果用户选择的是 `接入已有飞书应用`，下一页应该是手动接入页。

这一页的目标是：

- 把一个已经存在的飞书应用接到这台机器上

页面应该只出现用户真的需要填写的内容：

- `App ID`
- `App Secret`
- 可选的 `备注名称`

页面上要有一句用户能听懂的话：

- `如果你已经在飞书后台配好了这个应用，这一步只需要把它接进来。`

主按钮应该很明确：

- `保存并验证`

这一步做完以后，系统应该立刻给用户一个明确结果，而不是只提示“保存成功”。

建议页面文案可以直接写成：

```text
标题：接入已有飞书应用
说明：如果你已经在飞书后台配好了这个应用，这一步只需要把它接进来。

字段：App ID
字段说明：填写这个飞书应用的 App ID

字段：App Secret
字段说明：填写这个飞书应用的 App Secret

字段：备注名称（可选）
字段说明：方便你以后在管理页里认出这个应用

主按钮：保存并验证
次按钮：返回重新选择
```

#### 3.4.3 路径 C：使用当前已检测到的飞书应用

如果用户选择的是 `使用当前已检测到的飞书应用`，下一页应该是只读确认页。

这一页的重点不是编辑，而是确认。

用户应该看到：

- 当前飞书应用的名字
- 当前飞书应用的状态
- 它是不是外部接管 / 只读
- 它为什么会出现在这里

如果它是只读的，页面要直接说清楚：

- `这个飞书应用已经存在，但不在这里修改飞书平台配置`

这一页可用的动作应该很少：

- `验证当前飞书应用`
- `换一个飞书应用`

而不应该又出现一套可编辑表单，把用户带回本地创建路径。

建议页面文案可以直接写成：

```text
标题：继续使用当前已检测到的飞书应用
说明：系统已经发现了一个飞书应用。你可以先验证它现在是否可用。

信息卡标题：当前飞书应用
信息卡正文：名称：团队助手
信息卡补充：状态：可用，但还有待补全项

只读提示：这个飞书应用已经存在。这里不会修改它在飞书平台上的原始配置。

主按钮：验证当前飞书应用
次按钮：换一个飞书应用
```

### 3.5 第五步：确认这个飞书应用现在是不是真的能用

飞书应用接进来以后，系统应该立刻给出一个明确的连接结果。

这一页的目标不是“保存成功”，而是：

- 告诉用户这个飞书应用现在到底能不能继续往下走

#### 3.5.1 验证通过时，页面应该怎么说

验证通过以后，页面要明确给用户一个“已经接通”的结果。

最自然的表达应该类似：

- `连接成功，可以继续`
- `这个飞书应用已经接到当前机器`

如果这是一个已存在的飞书应用，也应该直说：

- `当前检测到的飞书应用可以继续使用`

不要只写成抽象的技术成功提示。

建议页面文案可以直接写成：

```text
标题：连接成功
说明：这个飞书应用已经接到当前机器，可以继续下一步。

状态卡标题：现在可以继续
状态卡正文：我们已经确认这个飞书应用可以正常接入。接下来会继续检查它现在是不是已经能开始使用。

主按钮：继续
```

#### 3.5.2 验证失败时，页面应该怎么说

验证失败时，页面也要尽量说人话。

用户应该立刻知道两件事：

1. 这一步还没有完成
2. 失败的大概方向是什么

例如页面应该像这样表达：

- `连接还没有成功`
- `请检查 App ID / App Secret 是否正确`
- `请确认这个飞书应用已经添加了收发消息所需能力`

主按钮可以是：

- `重新验证`

辅助动作可以是：

- `返回修改`
- `换一个飞书应用`

而不是把一大段原始错误直接丢给用户自己理解。

建议页面文案可以直接写成：

```text
标题：连接还没有成功
说明：这一步还没有完成。请先检查下面这些常见问题，再重新验证。

问题提示 1：请检查 App ID / App Secret 是否填写正确
问题提示 2：请确认这个飞书应用已经添加了收发消息所需能力

主按钮：重新验证
次按钮：返回修改
次按钮：换一个飞书应用
```

### 3.6 第六步：按功能组告诉用户哪些能力已经有了，哪些必须先开，哪些可以以后再补

这是整个 setup 里最关键的一页。

这一页不应该再是旧的“权限 / 事件 / 菜单 / 发布”逐页打卡流。

它应该是一张统一的“能力检查页”，而且第一层展示应该按用户能理解的功能组来组织，而不是直接把 raw scopes JSON 丢给用户。

页面的核心目标是：

- 告诉用户，这个飞书应用现在到底能不能开始用
- 告诉用户，哪些现在必须处理
- 告诉用户，哪些可以以后再补

#### 3.6.1 这一页第一眼应该先看到三组能力

参考 `#137`，首版最合适的分组不是按飞书底层权限名，而是按用户能感知到的系统能力：

1. `基础对话与交互`
2. `单聊状态提醒`
3. `Markdown 预览`

用户在页面上看到的应该是这三张能力卡，而不是一大段 scope 列表。

每张卡都应该直接告诉用户三件事：

- 这组能力是干什么的
- 不开会失去什么
- 它现在是已经可用、需要处理，还是可以稍后再说

#### 3.6.2 `基础对话与交互` 这张卡，应该怎么写

这是核心必需组。

页面应该用人话解释它的作用，例如：

- `让这个飞书应用能正常收发消息`
- `让它能在单聊和群聊里工作`
- `让它能处理图片和被引用的内容`
- `让它能显示基础聊天过程反馈`

这组如果没开，页面就要明确告诉用户：

- `不开通这一组，这个飞书应用基本无法正常对话`

所以这张卡应该放在“必须先修好”的区域里。

用户在这张卡上，应该一眼能看到：

- 当前状态
- 简短用途
- 如果没开通会怎样
- 去处理的入口

建议卡片文案可以直接写成：

```text
卡片标题：基础对话与交互
卡片说明：让这个飞书应用能正常收发消息，在单聊和群聊里工作，处理图片与引用内容，并显示基础聊天过程反馈。

状态：必须先开通
不开通影响：这个飞书应用基本无法正常对话。

主按钮：去处理这一组
```

#### 3.6.3 `单聊状态提醒` 这张卡，应该怎么写

这是可选增强组。

页面应该这样解释，而不是用内部术语：

- `当这个飞书应用在单聊里等你继续输入时，可以在会话列表里显眼提醒你回来处理`

如果这一组不开，页面应该明确说：

- `不影响主对话，只是少了等待输入时的显眼提醒`

所以这张卡应该放在“可以稍后再补”的区域里。

建议卡片文案可以直接写成：

```text
卡片标题：单聊状态提醒
卡片说明：当这个飞书应用在单聊里等待你继续输入时，可以在会话列表里显眼提醒你回来处理。

状态：可稍后处理
不开通影响：不影响主对话，只是少了等待输入时的显眼提醒。

主按钮：稍后再说
辅助按钮：现在去处理
```

#### 3.6.4 `Markdown 预览` 这张卡，应该怎么写

这也是可选增强组。

页面应该这样解释：

- `把回复里的本地 .md 链接自动变成飞书文档预览链接`

如果这一组不开，页面应该明确说：

- `主对话仍然可用，只是本地 Markdown 链接不会自动变成飞书预览`

这张卡同样应该放在“可以稍后再补”的区域里。

建议卡片文案可以直接写成：

```text
卡片标题：Markdown 预览
卡片说明：把回复里的本地 .md 链接自动变成飞书文档预览链接。

状态：可稍后处理
不开通影响：主对话仍然可用，只是本地 Markdown 链接不会自动变成飞书预览。

主按钮：稍后再说
辅助按钮：现在去处理
```

#### 3.6.5 这一页不能只谈 scope，还要把相关事件 / 回调一起讲清楚

这里有一个用户视角上很重要的点：

- `菜单快捷入口`
- `卡片按钮交互`
- `消息撤回处理`

这些用户能感知到的能力，不只是 scope 问题，也依赖事件和回调订阅。

所以这一页不应该把它们拆成底层术语清单丢给用户，而应该继续归在“能力组”下面解释：

- 这组能力现在有没有准备好
- 如果没准备好，用户会失去什么体验

如果确实要给高级用户看原始 scope / event / callback，也应该退到：

- `查看技术详情`

而不是默认展开。

#### 3.6.6 这一页应该怎么分成“必须先修”和“可以稍后”

页面应该把所有检查项分成两组：

- `必须先修好，否则现在没法正常用`
- `可以稍后再补，不影响现在开始使用`

用户应该看到类似这种表达：

- `基础对话与交互：通过`
- `当前飞书应用已进入系统：通过`
- `单聊状态提醒：还没开通，可稍后处理`
- `Markdown 预览：还没开通，可稍后处理`
- `卡片按钮交互：还没准备好，可稍后处理`

重点不是列很多技术名词，而是让用户立刻知道：

- 我现在卡在哪里
- 哪些现在必须做
- 哪些以后再做也没问题

这一页顶部的总文案建议直接写成：

```text
标题：再确认一下，现在已经能不能开始用
说明：下面这些能力里，基础项需要先准备好；增强项如果现在还没补齐，也可以以后再处理。
```

#### 3.6.7 如果当前已经能用，但还有待补全项

这时页面必须明确写成：

- `现在已经可以开始使用`
- `还有一些增强项可以稍后补齐`

而不是：

- `首次配置还没完成`

这是整个新体验里非常重要的一点。

这时页面顶部的结果提示建议直接写成：

```text
结果提示：现在已经可以开始使用
补充说明：还有一些增强项可以稍后补齐，不影响你现在开始对话。
```

#### 3.6.8 如果当前还不能用

如果还有关键阻断项，页面也必须说得很直接：

- `现在还不能开始使用`
- `请先修复这些阻断项`

同时每个阻断项要给出简单明确的动作，例如：

- `重新验证连接`
- `重新检查消息链路`
- `先开通基础对话与交互`
- `返回修改飞书应用`

这时页面顶部的结果提示建议直接写成：

```text
结果提示：现在还不能开始使用
补充说明：请先修复下面这些阻断项，修好以后再继续。
```

### 3.7 第七步：处理这台机器自己的选择

飞书应用本身已经接通，或者已经确认“现在能用”以后，页面才应该来问机器级选项。

这一页只处理机器自己的事情，比如：

- 自动启动
- VS Code 集成

而且这些选项都应该允许用户明确表达：

- `现在处理`
- `稍后再说`
- `去别的机器处理`

用户一旦做过决定，刷新后页面应该记得。

不应该每次回来都重新问一遍。

#### 3.7.1 自动启动页，用户应该看到什么

用户应该看到：

- 现在有没有开启自动启动
- 开启以后意味着什么
- 如果现在不想开，也可以以后再处理

主动作应该很简单：

- `开启自动启动`
- `稍后再说`

建议页面文案可以直接写成：

```text
标题：要不要顺手打开自动启动？
说明：打开后，这台机器登录后会自动拉起 Codex Remote。你也可以以后再回来处理。

当前状态：尚未开启自动启动

主按钮：开启自动启动
次按钮：稍后再说
```

#### 3.7.2 VS Code 页，用户应该看到什么

用户应该看到：

- 当前机器是不是适合现在就处理 VS Code
- 如果这是远程机器，是否应该在这里处理，还是去目标机器处理
- 现在处理和以后处理的区别是什么

主动作应该类似：

- `现在处理 VS Code`
- `稍后处理`
- `去别的机器处理`

建议页面文案可以直接写成：

```text
标题：要不要现在处理 VS Code？
说明：如果你准备在这台机器上配合 VS Code 使用，现在就可以继续；如果不是，也可以以后再处理。

主按钮：现在处理 VS Code
次按钮：稍后处理
次按钮：去别的机器处理
```

### 3.8 第八步：完成页

完成页不应该像一个大总结报告。

完成页只需要清楚告诉用户三件事：

1. 现在已经可以开始使用
2. 如果有待补全项，它们是什么
3. 下一步去哪里

完成页最自然的内容应该类似：

- `已经可以开始使用 Codex Remote`
- `当前可用飞书应用：xxx`
- `待补全项：飞书应用菜单、卡片按钮交互`
- `下一步：进入管理页`

建议页面文案可以直接写成：

```text
标题：已经可以开始使用了
说明：这台机器已经完成基本开通。接下来你可以先开始对话，剩下的增强项以后再补。

信息卡：当前可用飞书应用：开发者小助手
信息卡：待补全项：单聊状态提醒、Markdown 预览

主按钮：进入管理页
辅助提示：如果你只是想先试一下，现在就可以开始第一次对话。
```

## 4. 什么情况下应该允许完成 setup

从用户视角，完成 setup 的标准应该非常直白：

- 这台机器的基础环境已经通过
- 我已经有至少一个飞书应用可以开始用
- 对机器级可选项，我已经做了决定

这就够了。

以下情况不应该继续阻止我完成 setup：

- 我没有在当前页面重新点过某条原始 scope
- `单聊状态提醒` 还没开
- `Markdown 预览` 还没开
- 菜单或卡片交互这类增强能力还没补齐
- VS Code 决定以后再弄
- 自动启动决定稍后再开

这些都应该显示成：

- `现在已经能用，但还有待补全项`

而不是：

- `首次配置还没完成`

## 5. 用户完成 setup 后，进入 admin 应该看到什么

`admin` 的角色应该非常明确：

- 这里是日常管理页

用户进入 admin 后，应该先看到稳态信息，而不是重新进入 setup 向导。

### 5.1 admin 首页

首页应该先回答三件事：

1. 这台机器现在整体可不可用？
2. 当前有哪些飞书应用？
3. 哪些飞书应用正常可用，哪些还有待补全项，哪些已经坏了？

用户在首页应该看到：

- 当前机器状态
- 飞书应用列表
- 待补全项总览
- 明确的下一步动作

建议页面文案可以直接写成：

```text
标题：本地管理页
说明：这里会显示这台机器和各个飞书应用的当前状态。你也可以在这里继续补全增强项，或者再接入一个新的飞书应用。

状态总览：这台机器当前可用
待补全项：还有 2 个增强项可以稍后处理

主按钮：新增飞书应用
辅助按钮：查看待补全项
```

### 5.2 每个飞书应用卡片上，应该直接看懂什么

每个飞书应用都应该能让用户一眼看懂：

- `不可用`
- `可用`
- `可用，但还有待补全项`
- `外部接管`

同时要给出很直接的推荐动作，比如：

- `查看待补全项`
- `重新验证连接`
- `修复配置`
- `查看只读状态说明`

建议飞书应用卡片文案可以直接写成：

```text
卡片标题：开发者小助手
卡片状态：可用，但还有待补全项
卡片说明：现在已经可以正常使用。你也可以稍后回来补全单聊状态提醒和 Markdown 预览。

主按钮：查看待补全项
辅助按钮：查看详情
```

### 5.3 点进飞书应用详情后，用户应该看到什么

飞书应用详情页不应该是底层数据面板。

它应该像一张状态卡，直接回答：

1. 这个飞书应用现在能不能用？
2. 如果还不能用，卡在哪里？
3. 如果已经能用，还有哪些可补项？
4. 如果它是只读 / 外部接管，我在这里能做什么、不能做什么？

建议详情页顶部文案可以直接写成：

```text
标题：开发者小助手
说明：这里会显示这个飞书应用现在能不能用、还缺什么，以及你接下来最适合做什么。

状态提示：当前可用
补充说明：还有 2 个增强项可以稍后补齐，不影响现在开始使用。
```

### 5.4 用户想再接一个飞书应用时，应该怎么走

当用户想再接一个飞书应用时，应该在 admin 里看到一个清楚的入口：

- `新增飞书应用`

点进去以后，进入的是 admin 内部的一个独立子流程，而不是被踢回 setup 再走一遍首次安装。

这个子流程本质上应该复用 setup 里的飞书应用接入主线：

- 选择来源
- 接入
- 验证
- 检查能力

只是它发生在 admin 里，而不是 setup 里。

这个入口的按钮文案建议直接写成：

```text
按钮：新增飞书应用
按钮说明：再接入一个新的飞书应用，或继续使用当前已经检测到的飞书应用。
```

### 5.5 用户想补做增强项时，应该怎么走

如果用户已经能正常使用，只是还没补飞书应用菜单、卡片按钮交互、Drive 预览权限之类的增强项，那么 admin 应该给出很自然的入口：

- `继续补全`
- `查看待补全项`

而不是继续强调：

- `你还没完成首次配置`

### 5.6 用户想改机器设置时，应该怎么走

机器设置应该在 admin 里是一个独立区域，例如：

- 自动启动
- VS Code 集成

用户在这里应该能清楚地看到：

- 现在是什么状态
- 以前做过什么决定
- 现在要不要改

## 6. 四种典型用户路径，最终应该是什么感觉

### 6.1 情况 A：第一次安装，基础环境也还没好

用户进入 setup 后，第一步就被停在环境检测页。

页面明确告诉他：

- 现在还不能配置飞书应用
- 先把 `codex` 或基础环境修好

用户修好以后再回来，才继续飞书应用流程。

### 6.2 情况 B：第一次安装，环境没问题，但还没有飞书应用

用户进入 setup 后，会经历：

1. 环境通过
2. 当前没有飞书应用
3. 选择新建或接入已有
4. 接通并验证飞书应用
5. 看能力检查结果
6. 做机器级决定
7. 完成并进入 admin

这是最标准的一条首次安装路径。

### 6.3 情况 C：飞书应用其实早就在别处配好了

用户进入 setup 后，会看到：

- 环境通过
- 系统已经检测到一个可用飞书应用
- 这个飞书应用已经能开始使用

如果还有增强项没补，页面会写：

- `现在已经可以使用`
- `还有这些可以稍后补齐`

用户不应该再被拉回旧的“未完成首次配置”流程。

### 6.4 情况 D：飞书应用是外部接管的

用户进入 setup 或 admin 后，会看到：

- 当前飞书应用是只读 / 外部接管
- 它现在是不是可用
- 我能看状态，但不能在这里改飞书平台配置

页面应该给用户稳定、明确的预期，而不是让他反复撞进一个改不了的本地表单。

## 7. 用户不应该再看到的体验

从用户视角，下面这些体验都应该消失：

- 基础环境还没过，就先让我去设置飞书应用
- 明明已经能用，但页面仍长期显示“未完成首次配置”
- 因为没有在当前页面点过确认，就被判定成“没完成”
- 外部接管飞书应用被引导进入本地可编辑向导
- admin 里不断提示“回到 setup 继续完成”
- 机器级选项每刷新一次就重新问一遍
- 同一个页面同时堆满 setup、admin、多 App、技术状态和未来步骤

## 8. 用一句话概括最终体验

最终用户应该感受到的是：

- 先确认这台机器真的能跑，再接通一个飞书应用，再告诉我现在能不能用、还有什么以后再补，最后把我带进日常管理页。
