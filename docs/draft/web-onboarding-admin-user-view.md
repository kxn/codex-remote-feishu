# Web Setup And Admin User View

> Type: `draft`
> Updated: `2026-04-11`
> Summary: 重整为普通用户视角的完整产品流程说明，明确从基础环境检测到 bot 接入、能力检查、完成页和 admin 的全链路体验。

## 1. 文档定位

这不是技术 PRD。

这份文档只回答一个问题：

- 如果我是普通用户，我从打开 `setup` 到以后日常进入 `admin`，整个产品流程应该看到什么？

它不讨论：

- 后端状态模型怎么设计
- 哪些字段来自 daemon / runtime / gateway
- 哪些能力是自动检测、哪些能力是手工补齐

如果要看技术拆解，请回到：

- [web-onboarding-admin-workflow-prd.md](./web-onboarding-admin-workflow-prd.md)

## 2. 用户应该怎么理解这个产品

从用户视角，整个产品其实只有两部分：

### 2.1 `setup`

`setup` 是一次性的开通向导。

它只负责一件事：

- 把这台机器带到“已经可以开始使用”的状态

### 2.2 `admin`

`admin` 是后续的日常管理页。

它负责：

- 看当前机器和 bot 的状态
- 补做还没做完的增强项
- 修复有问题的 bot
- 再新增一个 bot

用户不应该再感受到：

- `setup` 是一个长期使用的大控制台
- `admin` 只是一个“回去继续做 setup”的中转页

## 3. 用户打开 setup 后，完整应该经历什么

从用户角度，完整主线应该是 8 段。

### 3.1 第一步：先检查这台机器能不能开始

这是整个流程最前面的硬门槛。

在这一步，页面不应该先让我设置 bot。

页面应该先回答：

1. 这台机器有没有最基本的运行条件？
2. 系统能不能检测到 `codex` 这类基础依赖？
3. 现在是不是连最基础的会话都跑不起来？

用户在这一步应该看到的是一个很直接的结果页，而不是技术细节列表。

页面应该给出三种清楚状态：

- `可以继续`
- `有提醒，但不阻断`
- `当前不能继续`

#### 3.1.1 如果基础环境没过，页面应该怎么表现

如果像下面这些基础条件没满足：

- 没检测到 `codex`
- 当前机器的基础运行条件不满足
- 关键路径明显无法工作

那么页面就应该直接停在这里。

这时不应该继续出现：

- `新建飞书应用`
- `接入已有飞书应用`
- 任何 bot 设置流程

因为这些都还没有意义。

用户在这时应该看到：

- `这台机器还不能开始配置 bot`
- `请先修复基础环境问题`

同时页面应该明确列出阻断项，例如：

- `没有检测到 Codex`
- `当前运行环境检查没有通过`

每个阻断项下都应该有很简单的下一步说明，比如：

- `先安装或配置 Codex`
- `修复当前机器的运行环境后，再回来继续`

主按钮应该类似：

- `重新检查环境`

辅助动作可以是：

- `查看修复说明`

但不能让用户直接跳去 bot 设置。

#### 3.1.2 如果基础环境过了，页面应该怎么说

如果基础环境已经满足，页面应该明确告诉用户：

- `这台机器已经满足基础要求，可以继续配置 bot`

这一步给用户的感觉应该是：

- 先确认地基没问题
- 之后再去处理 bot

而不是一开始就把 bot 和机器问题混在一起。

### 3.2 第二步：看系统现在已经有什么

只有基础环境通过以后，才应该进入这一页。

这一页的目标不是配置，而是告诉用户当前现状。

页面应该回答：

1. 系统现在有没有检测到 bot？
2. 如果有，是几个？
3. 它们现在分别是什么状态？
4. 当前最建议我处理哪一个？

用户在这一页应该看到很直白的信息：

- `当前没有检测到可用 bot`
- `已经检测到 1 个可用 bot`
- `已经检测到 1 个 bot，但它现在有问题`
- `已经检测到一个外部接管的 bot`

如果有多个 bot，页面必须直接显示：

- 名字
- 当前状态
- 哪一个是当前推荐处理对象

而不能静默默认选中一个。

这一页的主动作应该很少，通常只有：

- `开始接入 bot`
- `继续使用当前 bot`
- `修复当前 bot`

### 3.3 第三步：决定当前要处理哪一个 bot

到了这一步，用户才真正进入 bot onboarding。

这一步的目标是：

- 明确当前这次 setup，到底是处理哪个 bot

用户应该先看到一页“选择 bot 来源”的页面。

页面上应该是几张卡片，而不是一堆字段。

用户应该看到：

- `新建飞书应用`
- `接入已有飞书应用`
- 如果已经检测到 bot，再加一个：`使用当前已检测到的 bot`

如果当前已经检测到 bot，这一页还应该直接写清楚：

- 当前检测到的是哪个 bot
- 它现在是可用、部分可用，还是不可用
- 它是不是只读 / 外部接管

用户在这一页只需要做一件事：

- 选一条路径，然后点 `下一步`

### 3.4 第四步：把 bot 接进来

这一步只做一件事：

- 让当前 bot 真正接到这台机器上

不同路径下，页面应该长得不一样。

#### 3.4.1 路径 A：新建飞书应用

如果用户选择的是 `新建飞书应用`，接下来应该经历两页。

##### 第一页：扫码创建页

这一页应该非常明确地告诉用户：

- `请用飞书扫码创建应用`
- `系统会自动拿回这个应用的凭据`
- `完成后会自动继续验证连接`

用户在这一步应该看到：

- 一个清楚的二维码
- 一句很短的话，说明扫码是干什么的
- 一个等待中的状态提示，比如：`正在等待扫码完成`
- 两个辅助动作：
  - `刷新二维码`
  - `返回重新选择`

这一步不应该要求用户自己去猜：

- 扫完码以后是不是还要回来填表
- 系统到底有没有拿到凭据
- 下一步到底会发生什么

##### 第二页：扫码成功说明页

扫码成功后，不应该直接把用户悄悄扔到别的地方。

页面应该明确告诉用户：

- `已经为你创建并接入飞书应用`
- 这个应用现在叫什么名字
- 如果暂时拿不到名字，至少显示 App ID

同时要把“已经自动做完了什么”写清楚，例如：

- `应用已经创建`
- `凭据已经带回`
- `当前 bot 已经接入本机`

如果还有后续动作，页面要说人话：

- `下一步我们会继续检查这个 bot 现在能不能直接开始使用`

这一页的主按钮应该只有一个：

- `继续`

#### 3.4.2 路径 B：接入已有飞书应用

如果用户选择的是 `接入已有飞书应用`，下一页应该是手动接入页。

这一页的目标是：

- 把一个已经存在的飞书应用接到这台机器上

页面应该只出现用户真的需要填写的内容：

- `App ID`
- `App Secret`
- 可选的 `备注名称`

页面上要有一句用户能听懂的话：

- `如果你已经在飞书后台配好了这个应用，这一步只需要把它接进来。`

主按钮应该很明确：

- `保存并验证`

这一步做完以后，系统应该立刻给用户一个明确结果，而不是只提示“保存成功”。

#### 3.4.3 路径 C：使用当前已检测到的 bot

如果用户选择的是 `使用当前已检测到的 bot`，下一页应该是只读确认页。

这一页的重点不是编辑，而是确认。

用户应该看到：

- 当前 bot 的名字
- 当前 bot 的状态
- 它是不是外部接管 / 只读
- 它为什么会出现在这里

如果它是只读的，页面要直接说清楚：

- `这个 bot 已经存在，但不在这里修改飞书平台配置`

这一页可用的动作应该很少：

- `验证当前 bot`
- `换一个 bot`

而不应该又出现一套可编辑表单，把用户带回本地创建路径。

### 3.5 第五步：确认这个 bot 现在是不是真的能用

bot 接进来以后，系统应该立刻给出一个明确的连接结果。

这一页的目标不是“保存成功”，而是：

- 告诉用户这个 bot 现在到底能不能继续往下走

#### 3.5.1 验证通过时，页面应该怎么说

验证通过以后，页面要明确给用户一个“已经接通”的结果。

最自然的表达应该类似：

- `连接成功，可以继续`
- `这个 bot 已经接到当前机器`

如果这是一个已存在的 bot，也应该直说：

- `当前检测到的 bot 可以继续使用`

不要只写成抽象的技术成功提示。

#### 3.5.2 验证失败时，页面应该怎么说

验证失败时，页面也要尽量说人话。

用户应该立刻知道两件事：

1. 这一步还没有完成
2. 失败的大概方向是什么

例如页面应该像这样表达：

- `连接还没有成功`
- `请检查 App ID / App Secret 是否正确`
- `请确认这个应用已经添加了机器人能力`

主按钮可以是：

- `重新验证`

辅助动作可以是：

- `返回修改`
- `换一个 bot`

而不是把一大段原始错误直接丢给用户自己理解。

### 3.6 第六步：告诉用户哪些问题必须先修，哪些可以以后再补

这是整个 setup 里最关键的一页。

这一页不应该再是旧的“权限 / 事件 / 菜单 / 发布”逐页打卡流。

它应该是一张统一的“能力检查页”。

页面的核心目标是：

- 告诉用户，这个 bot 现在到底能不能开始用
- 告诉用户，哪些现在必须处理
- 告诉用户，哪些可以以后再补

#### 3.6.1 这一页应该怎么分组

页面应该把所有检查项分成两组：

- `必须先修好，否则现在没法正常用`
- `可以稍后再补，不影响现在开始使用`

用户应该看到类似这种表达：

- `基础消息链路：通过`
- `当前 bot 已进入系统：通过`
- `机器人菜单：还没确认，可稍后处理`
- `卡片回调：还没确认，可稍后处理`
- `Drive 预览权限：还没开通，可稍后处理`

重点不是列很多技术名词，而是让用户立刻知道：

- 我现在卡在哪里
- 哪些现在必须做
- 哪些以后再做也没问题

#### 3.6.2 如果当前已经能用，但还有待补全项

这时页面必须明确写成：

- `现在已经可以开始使用`
- `还有一些增强项可以稍后补齐`

而不是：

- `首次配置还没完成`

这是整个新体验里非常重要的一点。

#### 3.6.3 如果当前还不能用

如果还有关键阻断项，页面也必须说得很直接：

- `现在还不能开始使用`
- `请先修复这些阻断项`

同时每个阻断项要给出简单明确的动作，例如：

- `重新验证连接`
- `重新检查消息链路`
- `返回修改 bot`

### 3.7 第七步：处理这台机器自己的选择

bot 本身已经接通，或者已经确认“现在能用”以后，页面才应该来问机器级选项。

这一页只处理机器自己的事情，比如：

- 自动启动
- VS Code 集成

而且这些选项都应该允许用户明确表达：

- `现在处理`
- `稍后再说`
- `去别的机器处理`

用户一旦做过决定，刷新后页面应该记得。

不应该每次回来都重新问一遍。

#### 3.7.1 自动启动页，用户应该看到什么

用户应该看到：

- 现在有没有开启自动启动
- 开启以后意味着什么
- 如果现在不想开，也可以以后再处理

主动作应该很简单：

- `开启自动启动`
- `稍后再说`

#### 3.7.2 VS Code 页，用户应该看到什么

用户应该看到：

- 当前机器是不是适合现在就处理 VS Code
- 如果这是远程机器，是否应该在这里处理，还是去目标机器处理
- 现在处理和以后处理的区别是什么

主动作应该类似：

- `现在处理 VS Code`
- `稍后处理`
- `去别的机器处理`

### 3.8 第八步：完成页

完成页不应该像一个大总结报告。

完成页只需要清楚告诉用户三件事：

1. 现在已经可以开始使用
2. 如果有待补全项，它们是什么
3. 下一步去哪里

完成页最自然的内容应该类似：

- `已经可以开始使用 Codex Remote`
- `当前可用 bot：xxx`
- `待补全项：机器人菜单、卡片回调`
- `下一步：进入管理页`

## 4. 什么情况下应该允许完成 setup

从用户视角，完成 setup 的标准应该非常直白：

- 这台机器的基础环境已经通过
- 我已经有至少一个 bot 可以开始用
- 对机器级可选项，我已经做了决定

这就够了。

以下情况不应该继续阻止我完成 setup：

- 我没有在当前页面重新点过权限确认
- 菜单还没补
- 卡片回调还没补
- Drive 预览类增强能力还没补
- VS Code 决定以后再弄
- 自动启动决定稍后再开

这些都应该显示成：

- `现在已经能用，但还有待补全项`

而不是：

- `首次配置还没完成`

## 5. 用户完成 setup 后，进入 admin 应该看到什么

`admin` 的角色应该非常明确：

- 这里是日常管理页

用户进入 admin 后，应该先看到稳态信息，而不是重新进入 setup 向导。

### 5.1 admin 首页

首页应该先回答三件事：

1. 这台机器现在整体可不可用？
2. 当前有哪些 bot？
3. 哪些 bot 正常可用，哪些还有待补全项，哪些已经坏了？

用户在首页应该看到：

- 当前机器状态
- bot 列表
- 待补全项总览
- 明确的下一步动作

### 5.2 每个 bot 卡片上，应该直接看懂什么

每个 bot 都应该能让用户一眼看懂：

- `不可用`
- `可用`
- `可用，但还有待补全项`
- `外部接管`

同时要给出很直接的推荐动作，比如：

- `查看待补全项`
- `重新验证连接`
- `修复配置`
- `查看只读状态说明`

### 5.3 点进 bot 详情后，用户应该看到什么

bot 详情页不应该是底层数据面板。

它应该像一张状态卡，直接回答：

1. 这个 bot 现在能不能用？
2. 如果还不能用，卡在哪里？
3. 如果已经能用，还有哪些可补项？
4. 如果它是只读 / 外部接管，我在这里能做什么、不能做什么？

### 5.4 用户想再接一个 bot 时，应该怎么走

当用户想再接一个 bot 时，应该在 admin 里看到一个清楚的入口：

- `新增 bot`

点进去以后，进入的是 admin 内部的一个独立子流程，而不是被踢回 setup 再走一遍首次安装。

这个子流程本质上应该复用 setup 里的 bot onboarding 主线：

- 选择来源
- 接入
- 验证
- 检查能力

只是它发生在 admin 里，而不是 setup 里。

### 5.5 用户想补做增强项时，应该怎么走

如果用户已经能正常使用，只是还没补机器人菜单、卡片回调、Drive 预览权限之类的增强项，那么 admin 应该给出很自然的入口：

- `继续补全`
- `查看待补全项`

而不是继续强调：

- `你还没完成首次配置`

### 5.6 用户想改机器设置时，应该怎么走

机器设置应该在 admin 里是一个独立区域，例如：

- 自动启动
- VS Code 集成

用户在这里应该能清楚地看到：

- 现在是什么状态
- 以前做过什么决定
- 现在要不要改

## 6. 四种典型用户路径，最终应该是什么感觉

### 6.1 情况 A：第一次安装，基础环境也还没好

用户进入 setup 后，第一步就被停在环境检测页。

页面明确告诉他：

- 现在还不能配置 bot
- 先把 `codex` 或基础环境修好

用户修好以后再回来，才继续 bot 流程。

### 6.2 情况 B：第一次安装，环境没问题，但还没有 bot

用户进入 setup 后，会经历：

1. 环境通过
2. 当前没有 bot
3. 选择新建或接入已有
4. 接通并验证 bot
5. 看能力检查结果
6. 做机器级决定
7. 完成并进入 admin

这是最标准的一条首次安装路径。

### 6.3 情况 C：bot 其实早就在别处配好了

用户进入 setup 后，会看到：

- 环境通过
- 系统已经检测到一个可用 bot
- 这个 bot 已经能开始使用

如果还有增强项没补，页面会写：

- `现在已经可以使用`
- `还有这些可以稍后补齐`

用户不应该再被拉回旧的“未完成首次配置”流程。

### 6.4 情况 D：bot 是外部接管的

用户进入 setup 或 admin 后，会看到：

- 当前 bot 是只读 / 外部接管
- 它现在是不是可用
- 我能看状态，但不能在这里改飞书平台配置

页面应该给用户稳定、明确的预期，而不是让他反复撞进一个改不了的本地表单。

## 7. 用户不应该再看到的体验

从用户视角，下面这些体验都应该消失：

- 基础环境还没过，就先让我去设置 bot
- 明明已经能用，但页面仍长期显示“未完成首次配置”
- 因为没有在当前页面点过确认，就被判定成“没完成”
- 外部接管 bot 被引导进入本地可编辑向导
- admin 里不断提示“回到 setup 继续完成”
- 机器级选项每刷新一次就重新问一遍
- 同一个页面同时堆满 setup、admin、多 App、技术状态和未来步骤

## 8. 用一句话概括最终体验

最终用户应该感受到的是：

- 先确认这台机器真的能跑，再接通一个 bot，再告诉我现在能不能用、还有什么以后再补，最后把我带进日常管理页。
