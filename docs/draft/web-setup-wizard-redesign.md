# Web Setup 向导流程重设计

> Type: `draft`
> Updated: `2026-04-09`
> Summary: 同步第 8 步 VS Code 场景化引导草案，继续保持 setup 向导与 admin 管理页的职责分离。

## 1. 文档定位

这份文档只讨论 `web setup` 的交互流程重设计。

它不讨论：

- daemon / config / gateway 的底层结构
- admin 页面如何组织
- 多 App 在线的后端实现细节

它只回答三件事：

- setup 向导应该按什么步骤带用户完成首次配置
- 每一步页面上应该显示什么，不应该显示什么
- 用户在每一步会看到什么、要做什么、完成条件是什么

相关背景文档：

- [web-admin-ui-redesign.md](./web-admin-ui-redesign.md)
- [web-install-admin-prerequisites-design.md](./web-install-admin-prerequisites-design.md)

## 2. 背景收敛

基于当前反馈，现有 setup 页面的核心问题不是“功能不够”，而是“暴露方式错误”和“当前步骤信息不足”同时存在：

- setup 现在更像一个大控制台页面，而不是向导
- 用户一进入页面，就能往下滚看到大量还没到的步骤
- “当前状态”这类总览信息对普通用户价值很低，反而会制造噪音
- 多 App、manifest、VS Code、完成态都在第一页同时出现，认知负担过大

因此这一版设计的核心不是简单减少信息，而是：

- 对未来步骤严格收口
- 对当前步骤充分解释
- 让普通用户知道“现在要去哪里、做什么、为什么”

## 3. 核心设计原则

### 3.1 单步曝光

- setup 是向导，不是长页面知识库
- 当前只显示“当前步骤”的内容
- 已完成步骤只保留简短摘要，不展开细节
- 未开始步骤只显示步骤名，不显示任何正文内容

### 3.2 面向普通用户

- setup 只服务“先完成一个可工作的安装”
- 多 App 管理能力保留在后端和 admin，但不作为 setup 首屏主结构
- 复杂系统状态、runtime 术语、底层路径不默认展示
- 技术细节只在出错时按需展开
- 当前步骤必须把用户真正需要的操作信息讲清楚
- 不能因为怕信息多，就把关键指引省掉

### 3.3 无长滚动泄露

- 页面可以仍然是单页 SPA
- 但用户在视觉上不应该通过下滚看到后续步骤内容
- 后续步骤在到达前不能展开，不能预览，不能滚动到

### 3.4 每步只做一件事

- 每一步只围绕一个目标
- 页面只保留一个主要 CTA
- 需要解释时，解释必须服务于本步操作，不提前讲后面的事
- 当前步骤允许有足够的说明、链接、截图位、操作提示
- 但这些信息必须只服务当前步骤

### 3.5 setup 不等于 admin

- setup 不是“把 admin 页面裁一半”
- setup 的目标是完成首次接入
- setup 完成后，复杂管理工作再进入 admin

## 4. 信息显隐规则

这是本轮重设计最重要的硬规则。

### 4.1 左侧步骤栏

左侧可以始终显示完整步骤名，但只允许显示三种状态：

- `当前`
- `已完成`
- `未开始 / 已锁定`

左侧不显示：

- 后续步骤的字段内容
- 后续步骤的 checklist 详情
- 后续步骤的实时状态
- 后续步骤的操作按钮

### 4.2 中央内容区

中央内容区永远只渲染一个步骤：

- 当前步骤

已完成步骤不在正文区保留展开内容。

未开始步骤不在正文区出现占位卡片，不出现折叠区，不出现长滚动空白。

### 4.3 技术信息

默认隐藏这些内容：

- config path
- relay URL
- install state path
- gateway runtime 状态
- bundle / shim 检测细节
- preview / storage / instance 等 admin 语义

只有两种情况才展示：

- 本步骤确实需要用户处理这个信息
- 当前步骤失败，需要给出可执行的诊断提示

### 4.4 必做与可选

- 飞书接入主链路步骤不提供 `跳过`、`稍后再说`
- 只有真正不影响可用性的步骤才允许 `跳过`
- 当前确认可选的只有：
  - VS Code 集成

### 4.5 完成态

完成页允许给摘要，但也只保留：

- 已完成
- 进入下一步建议
- admin 入口

不重新把整个 setup 全部内容摊开。

## 5. 整体页面骨架

## 5.1 桌面布局

推荐布局：

```text
+------------------------------------------------------------------+
| Header: Setup 向导 / 步骤 2/9 / 当前要做：创建并连接飞书应用     |
+------------------------+-----------------------------------------+
| 左侧步骤栏             | 当前步骤正文                            |
| 1. 开始                | 标题                                    |
| 2. 创建并连接飞书应用 <-当前 | 简短说明                           |
| 3. 配置应用权限        | 主操作区                                |
| 4. 配置事件订阅        | 辅助说明 / 帮助卡 / 错误入口            |
| 5. 配置回调订阅方式    |                                         |
| 6. 配置机器人菜单      |                                         |
| 7. 发布应用            |                                         |
| 8. VS Code（可选）     |                                         |
| 9. 完成                |                                         |
+------------------------+-----------------------------------------+
| Footer: 上一步 / 当前步骤主按钮 / 当前步骤辅助按钮（按需出现）  |
+------------------------------------------------------------------+
```

关键点：

- 左侧步骤栏固定，但只显示名字和状态
- 右侧一次只显示一个步骤
- 底部操作栏固定，避免用户在长页里寻找按钮
- 必做步骤不显示 `跳过`
- 只有第 8 步 `VS Code（可选）` 才允许出现 `稍后在管理页处理`

## 5.2 移动端布局

移动端改为：

- 顶部横向步骤条
- 下方单步骤卡片
- 底部固定操作区

移动端同样不能出现“往下滚动看到未来步骤”的情况。

## 5.3 顶部信息

顶部只保留三类轻量信息：

- 当前处于 setup 向导
- 当前第几步 / 共几步
- 一句当前步骤目的说明

删除当前实现里的“当前状态总览”大卡片。

## 6. setup 流程重设计

本版推荐把 setup 收敛成 9 步。

其中：

- `1-7` 是飞书接入主线，全部必做
- `8` 是可选的 VS Code 集成
- `9` 是完成出口

## 6.0 飞书平台页面链接规则

为避免用户每一步都自己在后台左侧菜单里找入口，setup 必须给出明确的后台直达链接。

推荐规则：

- 创建应用入口固定使用：`https://open.feishu.cn/app?lang=zh-CN`
- 第 `3-7` 步都根据当前 `App ID` 生成“打开当前应用后台”的链接
- 如果飞书控制台支持稳定 deep-link，就直接落到对应二级页
- 如果控制台 deep-link 不稳定，至少落到当前应用详情首页，例如：`https://open.feishu.cn/app/<AppID>` 或控制台当时的等价路由
- 同时在按钮下方补一句明确提示：
  - `打开后点击左侧「权限管理」`
  - `打开后点击左侧「事件与回调」`
  - `打开后点击左侧「机器人」`
  - `打开后点击左侧「版本管理与发布」`

也就是说：

- 页面上既要有“能点开的链接”
- 也要有“打开之后点哪里”的人话说明

## 6.1 第 1 步：开始

### 目标

让用户知道：

- 这是安装向导
- 接下来要完成什么
- 整个流程会分步进行

### 用户看到什么

- 标题：`开始设置 Codex Remote`
- 一句说明：`我们会一步一步完成飞书接入和本机集成。`
- 三条短说明：
  - `先创建并连接飞书应用`
  - `再完成权限、事件、回调长连接、菜单和发布`
  - `最后可按需配置 VS Code`

### 页面布局

- 中央一张欢迎卡
- 下方一个主要按钮：`开始`

### 不应该看到什么

- manifest JSON
- VS Code 检测详情
- 完成页内容
- 多 App 列表
- 系统状态总览

### 用户要做什么

- 点击 `开始`

### 完成条件

- 进入第 2 步

## 6.2 第 2 步：创建并连接飞书应用

### 目标

让用户完成：

- 去飞书开发者后台创建应用
- 确认该应用已经添加机器人能力
- 回到 setup 页面录入 `App ID / App Secret`
- 当场完成连通性测试

### 用户看到什么

- 标题：`创建并连接飞书应用`
- 一句说明：`先在飞书开发者后台创建一个企业自建应用，并给它添加机器人能力。`
- 一个显眼的后台入口区：
  - `打开飞书开发者后台：https://open.feishu.cn/app?lang=zh-CN`
  - `创建企业自建应用：进入后点击「创建企业自建应用」`
- 一段明确提示：
  - `必须给该应用添加机器人能力，否则后续消息、菜单和事件都不会生效`
  - `添加路径要直接写明：左侧「应用能力」或「添加应用能力」 -> 「机器人」 -> 「添加」`
- 一个“在哪里找到这些值”的帮助块：
  - `App ID：进入应用后，打开左侧「凭证与基础信息」，在「应用凭证」区域复制`
  - `App Secret：同一页、同一块区域复制`
- 字段：
  - `显示名称`
  - `App ID`
  - `App Secret`

### 页面布局

推荐双列：

- 左列：表单
- 右列：后台操作帮助卡

帮助卡至少包括：

- `打开开发者后台`
- `打开创建应用页`
- `如何添加机器人能力`
- `App ID / App Secret 在哪里查看`
- `如果刚创建完应用，回到这里继续填写`
- `没有机器人能力会导致什么后果`

### 不应该看到什么

- 其他 App 的列表
- runtime gateway 状态
- preview/storage/admin 信息
- 后续平台 checklist 详情

### 用户要做什么

- 去飞书后台创建应用
- 给应用添加机器人能力
- 填写 `显示名称 / App ID / App Secret`
- 点击 `测试并继续`

### 主操作规则

- 主按钮文案固定为：`测试并继续`
- 点击后立即执行连接测试
- 如果测试成功：
  - 保存配置
  - 自动进入下一步
- 如果测试失败：
  - 不跳页
  - 停留当前页面
  - 弹出强确认错误模态

### 完成条件

- 凭证保存成功
- 连通性测试成功

## 6.3 第 3 步：导入平台权限

### 目标

让用户完成飞书平台里最容易标准化的一部分：

- scopes / 权限导入

### 用户看到什么

- 标题：`配置应用权限`
- 一句说明：`先把当前应用需要的权限导入到飞书后台。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的权限管理页面`
- 一句辅助说明：
  - `如果没有直接进入权限页，就在打开后的左侧菜单点击「权限管理」`
- 一段说明：
  - `复制下方 JSON`
  - `去飞书后台打开“批量导入/导出权限”`
  - `粘贴 JSON 并点击“保存并申请开通”`
  - `保存后回到这里继续`
- 一份只读 JSON 导出区
- 一个主按钮：`复制权限配置`

### 页面布局

允许展示 JSON，但页面核心不应该只是“扔一段 JSON”。

推荐布局：

- 上方：做什么的说明
- 中间：只读代码框，显示足够高度方便复制
- 下方：`复制权限配置`
- 最下方：复选确认 `我已经在飞书后台完成权限导入`

### 不应该看到什么

- 事件列表
- 回调说明
- 菜单说明
- 发布说明

这些都属于后续步骤。

### 用户要做什么

- 复制权限配置
- 点击动态后台入口进入当前应用的权限设置页
- 去飞书后台导入
- 勾选确认
- 点击 `继续`

### 完成条件

- 用户确认已完成权限导入

## 6.4 第 4 步：配置事件订阅

### 目标

让用户单独完成事件订阅，不和回调、菜单、发布混在一起。

### 用户看到什么

- 标题：`配置事件订阅`
- 一句说明：`这一页只处理事件订阅。先保存订阅方式为长连接，再去下一页配置回调订阅方式。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的事件订阅设置页`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「事件与回调」`
- 一段分步骤说明：
  - `进入事件订阅页`
  - `点击「订阅方式」，默认就是长连接，点击保存`
  - `根据下方列表勾选事件`
  - `保存配置后回到这里`
- 一份事件列表卡片：
  - `im.message.receive_v1`
  - `im.message.recalled_v1`
  - `im.message.reaction.created_v1`
  - `application.bot.menu_v6`
  - `card.action.trigger`
  - 每个事件为什么需要

### 页面布局

- 顶部说明
- 后台入口按钮
- 事件列表说明卡
- 底部确认项：`我已经完成事件订阅`

### 不应该看到什么

- 回调长连接说明
- 菜单配置说明
- 发布说明

### 用户要做什么

- 打开动态后台入口
- 按列表勾选事件
- 保存
- 回来确认并继续

### 完成条件

- 用户确认已完成事件订阅

## 6.5 第 5 步：配置回调订阅方式

### 目标

明确告诉用户：

- 当前版本使用飞书长连接处理卡片交互
- 不需要填写 HTTP 回调地址
- 需要在飞书平台的“回调配置”里把回调订阅方式保存为长连接

### 用户看到什么

- 标题：`配置回调订阅方式`
- 一句说明：`当前版本通过飞书长连接接收卡片点击和交互事件，这一步只需要把回调订阅方式保存为长连接。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的事件与回调页面`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「事件与回调」`
- 一段较详细说明：
  - `找到「回调配置」`
  - `点击「回调订阅方式」`
  - `选择「长连接」并点击保存`
  - `确认事件列表里已经包含 card.action.trigger`
  - `保存后，卡片点击、approval request 等交互会直接通过长连接进入当前服务`
  - `保存后回到这里继续`

### 页面布局

- 顶部说明
- 后台入口按钮
- 回调长连接说明卡
- 事件要求说明卡
- 确认项：`我已经完成回调长连接配置`

### 不应该看到什么

- 菜单说明
- 发布说明

### 用户要做什么

- 打开动态后台入口
- 确认当前应用的回调订阅方式使用长连接
- 确认 `card.action.trigger` 已订阅
- 在飞书后台保存
- 回来确认并继续

### 完成条件

- 用户确认已完成回调长连接配置

## 6.6 第 6 步：配置机器人菜单

### 目标

让用户明确配置机器人菜单，并知道哪些按钮事件会被接收和生效。

### 用户看到什么

- 标题：`配置机器人菜单`
- 一句说明：`把需要的机器人菜单按钮加到飞书后台。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的机器人菜单配置页`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「机器人」并进入自定义菜单区域`
- 一份菜单说明区：
  - `列出实例（list）`
  - `当前状态（status）`
  - `切换会话（threads）`
  - `停止当前执行（stop）`
  - `推理 Low / Medium / High / XHigh（reason_low / reason_medium / reason_high / reason_xhigh）`
  - `执行权限 Full / Confirm（access_full / access_confirm）`
  - 每个按钮点击后会触发什么事件
  - 这些按钮里哪些是当前实现真正会处理的
  - 明确强调 `key` 必须与页面展示一致，否则点击后不会生效

### 页面布局

- 顶部说明
- 后台入口按钮
- 菜单定义列表
- 确认项：`我已经完成菜单配置`

### 不应该看到什么

- 事件订阅说明
- 发布说明
- 复杂系统状态

### 用户要做什么

- 打开后台入口
- 按说明创建菜单按钮
- 确认菜单 key 与页面展示一致
- 回来确认并继续

### 完成条件

- 用户确认已完成菜单配置

## 6.7 第 7 步：发布应用并验收

### 目标

要求用户真正发版，并在点击下一步时执行一次“是否生效”的验收检查。

### 用户看到什么

- 标题：`发布应用`
- 一句说明：`前面的权限、事件、回调长连接、菜单都配置完后，还需要在飞书后台发版，变更才会真正生效。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的版本管理 / 发布页面`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「版本管理与发布」`
- 一段明确说明：
  - `请在飞书后台完成发版`
  - `发版后点击“检查并继续”`
- 一个主按钮：`检查并继续`

### 页面布局

- 顶部说明
- 发布页入口按钮
- 简短操作指引
- 主按钮 `检查并继续`

### 用户要做什么

- 去飞书后台发版
- 回来点击 `检查并继续`

### 下一步校验规则

点击 `检查并继续` 时，后端要执行一次验收，不通过就拒绝进入下一步。

验收目标至少包括：

- 当前应用权限已生效
- 事件订阅已可见
- 回调长连接配置已完成
- 菜单配置已可见

如果校验失败：

- 停留在本页
- 弹出强确认错误模态
- 明确告诉用户“哪一项还没生效”

### 完成条件

- 验收通过

## 6.8 第 8 步：VS Code 集成（可选）

### 目标

把 VS Code 集成作为可选收尾步骤，而不是飞书 setup 主链路的一部分。

### 用户看到什么

- 标题：`配置 VS Code`
- 一句说明：`如果你需要在本机或远程 VS Code 里使用，可以现在完成；也可以稍后再配置。`

当前不再一上来展示内部推荐模式，而是先按实际使用场景引导：

- 如果当前是远程 SSH 机器：
  - 只显示“在这台远程机器上启用 VS Code”
  - 说明这一步只接管扩展入口，不会去写 host 机器的 `settings.json`
- 如果当前不是 SSH 机器：
  - 先问“你以后主要怎么使用 VS Code 里的 Codex”
  - 选项固定为：
    - `只在这台机器本地使用`
    - `主要去别的 SSH 机器上使用`
    - `这台机器本地要用，也会 SSH 到别的机器`
  - 用户选中后，再显示该场景唯一合理的推荐卡和主按钮

### 页面布局

页面主体：

- 当前环境提示
- 使用场景选择卡或推荐卡
- 一个主要按钮
- 一个次级按钮：
  - `稍后在管理页处理`

技术细节例如：

- settings path
- latest bundle
- recorded bundle
- shim reinstall

默认折叠到 `查看技术详情` 里，不默认显示。

### 不应该看到什么

- 一上来就展示所有 detect 字段
- 一上来把 `editor_settings` / `managed_shim` 当成用户主决策

### 用户要做什么

- 先确认自己属于哪种 VS Code 使用场景
- 再执行对应的主路径
- 或点击 `稍后在管理页处理`

### 完成条件

满足其一即可：

- 用户成功完成当前场景对应的接入
- 用户明确选择跳过或改在目标 SSH 机器上处理

## 6.9 第 9 步：完成

### 目标

让 setup 在结束前给出最后一次明确引导，然后直接进入本地管理页。

### 用户看到什么

- 标题：`完成并进入本地管理页`
- 简短摘要：
  - `飞书应用已连接`
  - `平台配置已完成`
  - `VS Code 已在当前机器接入`
  - 或 `VS Code 当前机器未接入，将在目标 SSH 机器处理`
  - 或 `VS Code 已留到管理页稍后处理`
- 一个“首次对话提示”说明卡：
  - `推荐先在飞书里打开“开发者小助手”`
  - `找到刚完成发布或审批通过的应用`
  - `点击“打开应用”后，先给机器人发一条测试消息完成第一次私聊`
  - `如果你的工作台已经能看到该应用，也可以直接从工作台进入`

- 一个主要按钮：`完成并进入本地管理页`

### 页面布局

- 完成摘要卡
- 首次对话提示卡

### 不应该看到什么

- 重新摊开所有 setup 内容
- 技术状态大面板
- 多模块 admin 内容预览

### 用户要做什么

- 完成 setup
- 页面自动跳到本地管理页

### 完成条件

- setup access 关闭
- 已进入本地管理页

## 7. 页面级交互规则

## 7.1 固定底部操作栏

每一步底部固定：

- `上一步`
- `主要操作`
- `次级操作`

避免用户滚动到页面底部找按钮。

补充规则：

- 必做步骤只出现与当前步骤相关的主按钮
- 不出现全局统一的 `跳过`
- `稍后在管理页处理` 只允许出现在第 8 步 `VS Code（可选）`

## 7.2 已完成步骤如何回看

允许用户回到已完成步骤，但默认只通过：

- 点击左侧已完成步骤
- 或 `返回上一步`

进入后仍然一次只看一个步骤。

## 7.3 统一报错交互

setup 不使用 toast 作为主错误承载方式。

统一规则：

- 当前步骤执行失败时，使用模态弹窗
- 模态必须要求用户明确点击确认
- 模态正文必须包含：
  - 一句人话错误说明
  - 当前失败的是哪一步
  - 建议用户下一步做什么
- 技术详情默认折叠，但允许展开

toast 只允许用于：

- 复制成功
- 已保存
- 这种不需要用户决策的轻提示

### 推荐错误模态结构

- 标题：`这一步还没有完成`
- 正文第一段：人话说明
- 正文第二段：建议动作
- 折叠区：技术详情
- 按钮：
  - `我知道了`

## 7.4 恢复上次进度

如果 setup 中断再回来：

- 直接定位到“第一个未完成步骤”
- 左侧把之前完成的步骤标成已完成
- 仍然不展开后续内容

## 8. 推荐文案风格

setup 页文案统一要求：

- 面向动作
- 短句
- 少术语
- 先告诉用户要做什么，再解释为什么

例如：

- 用 `先填写飞书应用信息`
- 不用 `当前 runtime 还没有 configured app`

- 用 `开始验证`
- 不用 `执行网关连通性检测`

- 用 `复制权限配置`
- 不用 `导出 scopes JSON`

- 用 `请先在飞书后台发版，再回来检查`
- 不用 `当前 publish 状态未就绪`

## 9. 与当前实现的主要差异

如果按这份设计重做，和当前 setup 最大的不同是：

1. 删除“当前状态”作为 setup 主页面块
2. setup 不再是长滚动单页内容集合
3. 左侧只保留步骤名，不保留未来步骤正文
4. 当前步骤信息会明显增强，包括后台入口、字段定位说明、操作指引
5. 凭证输入与连接测试合并为一步，改为 `测试并继续`
6. 平台权限、事件订阅、回调长连接配置、菜单配置、发布验收拆成独立页面
7. 发布不是“用户自己点个勾”，而是点击下一步时由系统做验收
8. VS Code 从首页主信息块降级为后置可选步骤
9. setup 错误交互改成强确认模态，而不是 toast

## 10. 推荐实施顺序

如果这份设计确认可行，推荐下一轮实现不要一次性重写全部页面，而是分三段：

1. 先改 setup 页面骨架：
   - 去掉当前状态块
   - 改成“左侧步骤 + 当前步骤正文”
   - 屏蔽未来步骤正文
2. 再重排步骤顺序与内容分组：
   - 创建应用 + 测试
   - 权限
   - 事件
   - 回调长连接配置
   - 菜单
   - 发布验收
   - VS Code
3. 最后统一补：
   - 动态后台入口
   - 错误模态
   - 文案与技术详情折叠

这样能先把最影响体验的“信息泄露”和“长滚动”问题消掉。
