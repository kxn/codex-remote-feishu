# Web Setup 向导流程重设计

> Type: `draft`
> Updated: `2026-04-29`
> Summary: 当前 setup 的用户可见合同以 `v1.7.0` 页面形态为基线：左侧步骤栏、右侧单步正文、事件与回调仍留在 setup 内，不再叠加共享 onboarding 总览或额外全局说明。

## 1. 文档定位

本文记录当前接受的 `web setup` 页面合同。

它只回答三件事：

1. setup 页面长什么样。
2. setup 里保留哪些步骤。
3. 哪些后来加出来的全局叙事不应再回到页面里。

## 2. 页面合同

### 2.1 最终用户

- 第一次在本机完成安装和接入的用户

### 2.2 当前页面只服务什么任务

- 把这台机器带到“已经可以开始使用”的状态
- 接入并验证一个可用的飞书应用
- 完成 setup 内保留的关键步骤

### 2.3 允许出现的信息类型

- 当前步骤名
- 当前步骤说明
- 当前步骤结果
- 当前步骤按钮
- 当前步骤需要的后台入口和复制信息

### 2.4 不允许默认出现的信息类型

- 全局 workflow 概览
- “当前推荐处理”“剩余建议项”之类的全局说明
- 共享 onboarding 总结卡
- 与当前步骤无关的 admin 语义
- 额外的大段解释文字

## 3. 页面骨架

setup 继续采用 `v1.7.0` 的两栏结构：

- 左侧：`设置流程`
- 右侧：当前步骤正文

左侧步骤栏只显示：

- 步骤名
- `进行中 / 已完成 / 待处理`

右侧正文一次只显示一个步骤。

顶部继续只保留：

- 页面标题：`<产品名> 安装程序`
- 全局反馈槽位

不再在顶部追加额外的总览、推荐处理或完成总结。

## 4. 当前保留步骤

setup 当前固定保留以下步骤：

1. `环境检查`
2. `飞书连接`
3. `权限检查`
4. `事件订阅`
5. `回调配置`
6. `菜单确认`
7. `自动启动`
8. `VS Code 集成`
9. `完成`

其中：

- `事件订阅` 和 `回调配置` 仍然保留在 setup 内，不从 setup 主流程移走。
- `自动启动` 和 `VS Code 集成` 仍然是向导后半段。
- `完成` 页只保留欢迎文案和进入 admin 的入口。

## 5. 各步骤的可见边界

### 5.1 环境检查

- 自动检查当前机器是否具备基础运行条件
- 失败时列出失败项
- 成功时自动进入下一步

### 5.2 飞书连接

- 保留 `扫码创建` 和 `手动输入`
- 连接验证通过后继续
- 不额外增加共享 onboarding 的摘要区

### 5.3 权限检查

- 保留缺失权限提示
- 保留缺失 scope 列表
- 保留一次性 JSON 复制区
- 保留飞书后台入口

### 5.4 事件订阅 / 回调配置 / 菜单确认

- 继续在 setup 中显示
- 继续以当前步骤自己的说明、提示和 `下一步` 按钮收口
- 不额外套一层全局能力检查页

### 5.5 自动启动 / VS Code 集成

- 保持 `v1.7.0` 的简单决策形态
- 当前页只处理当前动作，不补充全局机器决策说明

## 6. 明确不回归的内容

以下内容不应再回到 setup 主页面：

- 共享 onboarding 组件驱动的整页工作流
- 全局完成概览
- “当前推荐处理”
- “剩余建议项”
- setup 内的 admin 级分区
- 不直接服务当前步骤的额外说明文字

## 7. 维护约束

- 后续调整 setup 时，以 `v1.7.0` 的页面形态作为唯一基线。
- 如果流程逻辑后来变化，优先在当前步骤内部消化，不新增全局解释区。
- 若确需改变 setup 的默认结构，必须先同步本文档和 mock，再改产品页面。
