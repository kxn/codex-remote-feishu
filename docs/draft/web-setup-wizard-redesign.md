# Web Setup 向导流程重设计

> Type: `draft`
> Updated: `2026-04-11`
> Summary: 同步“新建 / 已有”飞书应用分流、二维码完成说明页、扫码路径快进 `runtimeRequirements` 与 admin/setup 共用移动端 shell 的草案，继续保持 setup 向导与 admin 管理页的职责分离。

## 1. 文档定位

这份文档只讨论 `web setup` 的交互流程重设计。

它不讨论：

- daemon / config / gateway 的底层结构
- admin 页面如何组织
- 多 App 在线的后端实现细节

它只回答三件事：

- setup 向导应该按什么步骤带用户完成首次配置
- 每一步页面上应该显示什么，不应该显示什么
- 用户在每一步会看到什么、要做什么、完成条件是什么

相关背景文档：

- [web-admin-ui-redesign.md](./web-admin-ui-redesign.md)
- [web-install-admin-prerequisites-design.md](./web-install-admin-prerequisites-design.md)

## 2. 背景收敛

基于当前反馈，现有 setup 页面的核心问题不是“功能不够”，而是“暴露方式错误”和“当前步骤信息不足”同时存在：

- setup 现在更像一个大控制台页面，而不是向导
- 用户一进入页面，就能往下滚看到大量还没到的步骤
- “当前状态”这类总览信息对普通用户价值很低，反而会制造噪音
- 多 App、manifest、VS Code、完成态都在第一页同时出现，认知负担过大

因此这一版设计的核心不是简单减少信息，而是：

- 对未来步骤严格收口
- 对当前步骤充分解释
- 让普通用户知道“现在要去哪里、做什么、为什么”

## 3. 核心设计原则

### 3.1 单步曝光

- setup 是向导，不是长页面知识库
- 当前只显示“当前步骤”的内容
- 已完成步骤只保留简短摘要，不展开细节
- 未开始步骤只显示步骤名，不显示任何正文内容

### 3.2 面向普通用户

- setup 只服务“先完成一个可工作的安装”
- 多 App 管理能力保留在后端和 admin，但不作为 setup 首屏主结构
- 复杂系统状态、runtime 术语、底层路径不默认展示
- 技术细节只在出错时按需展开
- 当前步骤必须把用户真正需要的操作信息讲清楚
- 不能因为怕信息多，就把关键指引省掉

### 3.3 无长滚动泄露

- 页面可以仍然是单页 SPA
- 但用户在视觉上不应该通过下滚看到后续步骤内容
- 后续步骤在到达前不能展开，不能预览，不能滚动到

### 3.4 每步只做一件事

- 每一步只围绕一个目标
- 页面只保留一个主要 CTA
- 需要解释时，解释必须服务于本步操作，不提前讲后面的事
- 当前步骤允许有足够的说明、链接、截图位、操作提示
- 但这些信息必须只服务当前步骤

### 3.5 setup 不等于 admin

- setup 不是“把 admin 页面裁一半”
- setup 的目标是完成首次接入
- setup 完成后，复杂管理工作再进入 admin

### 3.6 扫码新建必须是一等路径

- “扫码创建新飞书应用”不能只是手填路径旁边的附属按钮
- 如果产品推荐走扫码新建，这条路径就必须有完整的开始页、等待页、完成说明页和明确的下一步
- 扫码成功后不能直接把用户悄悄扔回旧编辑态或旧的人工飞书平台步骤
- 当扫码路径已经自动完成了大部分飞书平台配置时，setup 应该明确快进到新的下一步，而不是把用户重新带回手工权限/事件/菜单/发布页

## 4. 信息显隐规则

这是本轮重设计最重要的硬规则。

### 4.1 左侧步骤栏

左侧可以始终显示完整步骤名，但只允许显示三种状态：

- `当前`
- `已完成`
- `未开始 / 已锁定`

左侧不显示：

- 后续步骤的字段内容
- 后续步骤的 checklist 详情
- 后续步骤的实时状态
- 后续步骤的操作按钮

### 4.2 中央内容区

中央内容区永远只渲染一个步骤：

- 当前步骤

已完成步骤不在正文区保留展开内容。

未开始步骤不在正文区出现占位卡片，不出现折叠区，不出现长滚动空白。

### 4.3 技术信息

默认隐藏这些内容：

- config path
- relay URL
- install state path
- gateway runtime 状态
- bundle / shim 检测细节
- preview / storage / instance 等 admin 语义

只有两种情况才展示：

- 本步骤确实需要用户处理这个信息
- 当前步骤失败，需要给出可执行的诊断提示

### 4.4 必做与可选

- 飞书接入主链路步骤不提供 `跳过`、`稍后再说`
- 只有真正不影响可用性的步骤才允许 `跳过`
- 当前确认可选的只有：
  - VS Code 集成

### 4.5 完成态

完成页允许给摘要，但也只保留：

- 已完成
- 进入下一步建议
- admin 入口

不重新把整个 setup 全部内容摊开。

## 5. 整体页面骨架

## 5.1 桌面布局

推荐布局：

```text
+------------------------------------------------------------------+
| Header: Setup 向导 / 步骤 2/10 / 当前要做：创建并连接飞书应用    |
+------------------------+-----------------------------------------+
| 左侧步骤栏             | 当前步骤正文                            |
| 1. 开始                | 标题                                    |
| 2. 创建并连接飞书应用 <-当前 | 简短说明                           |
| 3. 配置应用权限        | 主操作区                                |
| 4. 配置事件订阅        | 辅助说明 / 帮助卡 / 错误入口            |
| 5. 配置回调订阅方式    |                                         |
| 6. 配置机器人菜单      |                                         |
| 7. 发布应用            |                                         |
| 8. 运行环境检查        |                                         |
| 9. VS Code（可选）     |                                         |
| 10. 完成               |                                         |
+------------------------+-----------------------------------------+
| Footer: 上一步 / 当前步骤主按钮 / 当前步骤辅助按钮（按需出现）  |
+------------------------------------------------------------------+
```

关键点：

- 左侧步骤栏固定，但只显示名字和状态
- 右侧一次只显示一个步骤
- 底部操作栏固定，避免用户在长页里寻找按钮
- 必做步骤不显示 `跳过`
- 只有第 9 步 `VS Code（可选）` 才允许出现 `稍后在管理页处理`

## 5.2 移动端布局

移动端改为：

- 顶部横向步骤条
- 下方单步骤卡片
- 底部固定操作区

移动端同样不能出现“往下滚动看到未来步骤”的情况。

另外，setup 与 admin 当前应共用一套移动端 shell：

- 桌面端保留侧栏信息架构
- 窄屏下统一收起为顶部可展开导航
- 点击步骤或分区后自动收起，避免导航长期挤占正文区

## 5.3 顶部信息

顶部只保留三类轻量信息：

- 当前处于 setup 向导
- 当前第几步 / 共几步
- 一句当前步骤目的说明

删除当前实现里的“当前状态总览”大卡片。

## 6. setup 流程重设计

本版推荐把 setup 收敛成 10 步。

其中：

- `1-2` 是飞书应用接入入口
- 若用户选择“接入已有应用”，则继续走 `3-7` 的人工飞书平台配置主线
- 若用户选择“新建飞书应用”并扫码成功，则在第 2 步的说明页后直接快进到 `8`
- `8` 是基础运行条件检查，仍属于必做
- `9` 是可选的 VS Code 集成
- `10` 是完成出口

## 6.0 飞书平台页面链接规则

为避免用户每一步都自己在后台左侧菜单里找入口，setup 必须给出明确的后台直达链接。

推荐规则：

- 创建应用入口固定使用：`https://open.feishu.cn/app?lang=zh-CN`
- 第 `3-7` 步都根据当前 `App ID` 生成“打开当前应用后台”的链接
- 如果飞书控制台支持稳定 deep-link，就直接落到对应二级页
- 如果控制台 deep-link 不稳定，至少落到当前应用详情首页，例如：`https://open.feishu.cn/app/<AppID>` 或控制台当时的等价路由
- 同时在按钮下方补一句明确提示：
  - `打开后点击左侧「权限管理」`
  - `打开后点击左侧「事件与回调」`
  - `打开后点击左侧「机器人」`
  - `打开后点击左侧「版本管理与发布」`

也就是说：

- 页面上既要有“能点开的链接”
- 也要有“打开之后点哪里”的人话说明

## 6.1 第 1 步：开始

### 目标

让用户知道：

- 这是安装向导
- 接下来要完成什么
- 整个流程会分步进行

### 用户看到什么

- 标题：`开始设置 Codex Remote`
- 一句说明：`我们会一步一步完成飞书接入和本机集成。`
- 三条短说明：
  - `先选择是新建飞书应用，还是接入已有应用`
  - `如果你接入已有应用，再完成权限、事件、回调长连接、菜单和发布`
  - `发布后先做运行环境检查，再按需配置自动启动和 VS Code`

### 页面布局

- 中央一张欢迎卡
- 下方一个主要按钮：`开始`

### 不应该看到什么

- manifest JSON
- VS Code 检测详情
- 完成页内容
- 多 App 列表
- 系统状态总览

### 用户要做什么

- 点击 `开始`

### 完成条件

- 进入第 2 步

## 6.2 第 2 步：创建并连接飞书应用

### 目标

先把“怎么接入飞书应用”这件事一次讲清楚，并让用户走上一条完整、可收口的路径。

这一页不是单一表单，而是一个带分流的正式步骤：

- 先选 `新建飞书应用` 或 `接入已有应用`
- 选完后，再进入对应的第二屏
- 如果是扫码新建，还要有单独的“扫码完成说明页”

### 用户看到什么

- 标题：`创建并连接飞书应用`
- 第一屏：
  - 一个标题块：`你想怎么接入飞书应用？`
  - 两张 radio 卡片：
    - `新建飞书应用`
    - `接入已有应用`
  - 一个主按钮：`下一步`

选择 `新建飞书应用` 后，进入二维码等待屏：

- 一张大二维码卡片
- 一句明确提示：
  - `请使用飞书手机客户端扫码完成应用创建与授权`
  - `拿到凭据后，页面会自动继续连接测试`
- 自动轮询状态说明：
  - `系统会每隔几秒自动检查一次扫码结果`
  - `如果等待较久，你也可以手动刷新`
- 二维码过期时间
- 一个浏览器兜底入口：
  - `无法扫码时，在浏览器打开`
- 辅助按钮：
  - `刷新二维码状态`
  - `重新开始扫码`
  - `改为接入已有应用`

当二维码已经拿到凭据、正在完成连接测试时：

- 明确展示：
  - 当前应用名 / App ID
  - `凭据已经拿到，正在完成连接测试`
  - 若自动测试失败，则提示 `飞书应用已经创建好了，你可以直接重试连接测试，或者改成手动填写`

扫码并验证成功后，进入单独说明页：

- 左侧展示：
  - 新应用名称
  - `App ID`
  - `扫码创建已经完成` 或后端返回的自动配置摘要
- 右侧展示：
  - `还剩哪些可选动作`
  - 默认至少包括 `如果需要把 Markdown 预览上传到飞书云盘，还需要额外申请 drive:drive`
  - 并明确提示这个权限通常需要管理员审批，不需要预览时可以先跳过
- 主按钮：`下一步`

选择 `接入已有应用` 后，进入手动接入屏：

- 左侧表单字段：
  - `App ID`
  - `App Secret`
- 支持直接粘贴 `cli_xxx:secret_xxx`，页面自动拆分
- 一句规则说明：
  - `改成另一个 App ID 等于切换到另一个机器人身份，旧飞书会话不会自动迁移`
  - `显示名称会优先尝试从飞书拉取，拿不到时回退到 App ID`
- 右侧帮助块：
  - `打开飞书开发者后台`
  - `进入应用后打开「凭证与基础信息」`
  - `复制 App ID 和 App Secret，然后回来验证并继续`
- 按钮：
  - `返回选择`
  - `验证并继续`

### 页面布局

推荐保持“单步骤、可切屏”的结构：

- 第一屏：单列模式选择
- 二维码等待屏 / 手动接入屏 / 扫码完成说明页：双列
- 双列右侧只承载当前子阶段相关帮助，不提前泄露后续平台步骤

### 不应该看到什么

- 其他 App 的列表
- runtime gateway 状态
- preview/storage/admin 信息
- 后续平台 checklist 详情

### 用户要做什么

- 如果选择 `新建飞书应用`：
  - 先扫码
  - 等页面自动完成凭据带回与连接测试
  - 看完说明页后点击 `下一步`
- 如果选择 `接入已有应用`：
  - 去飞书后台复制 `App ID / App Secret`
  - 回来填写
  - 点击 `验证并继续`

### 主操作规则

- 模式选择页主按钮固定为：`下一步`
- 手动接入路径主按钮固定为：`验证并继续`
- 扫码路径在二维码状态变成 ready 后，应自动进入连接测试；用户不需要先理解“还要自己再点一次刷新”
- 扫码成功后不直接跳下一大步，而是先进入正式说明页
- setup 中，扫码完成说明页点击 `下一步` 后，应该进入 `runtimeRequirements`
- admin 中，同一套说明页点击 `下一步` 后，应该回到机器人详情 / 编辑态

### 完成条件

- 手动接入路径：
  - 凭证保存成功
  - 连通性测试成功
- 扫码新建路径：
  - 飞书应用创建成功
  - 凭据带回成功
  - 连通性测试成功
  - 用户已经看到收口说明页，并能明确进入下一步

### 分支规则

- “接入已有应用”继续进入第 `3-7` 步
- “新建飞书应用”在说明页后直接进入第 `8` 步 `runtimeRequirements`
- 若后端已经明确产出自动完成摘要、剩余动作和推荐下一步，前端优先展示这些 guide 字段，而不是自己硬编码猜测

## 6.3 到 6.7 的适用范围

下面第 `3-7` 步只对“接入已有应用”的手动路径出现。

如果用户在第 `2` 步走的是扫码新建并且验证成功：

- 飞书侧基础接入里程碑会在后端一并持久化
- setup 不应再把用户带回权限 / 事件 / 菜单 / 发布这些人工页面
- 下一步直接进入 `runtimeRequirements`

## 6.4 第 3 步：导入平台权限

### 目标

让用户完成飞书平台里最容易标准化的一部分：

- scopes / 权限导入

### 用户看到什么

- 标题：`配置应用权限`
- 一句说明：`先把当前应用需要的权限导入到飞书后台。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的权限管理页面`
- 一句辅助说明：
  - `如果没有直接进入权限页，就在打开后的左侧菜单点击「权限管理」`
- 一段说明：
  - `复制下方 JSON`
  - `去飞书后台打开“批量导入/导出权限”`
  - `粘贴 JSON 并点击“保存并申请开通”`
  - `保存后回到这里继续`
- 一份只读 JSON 导出区
- 一个主按钮：`复制权限配置`

### 页面布局

允许展示 JSON，但页面核心不应该只是“扔一段 JSON”。

推荐布局：

- 上方：做什么的说明
- 中间：只读代码框，显示足够高度方便复制
- 下方：`复制权限配置`
- 最下方：复选确认 `我已经在飞书后台完成权限导入`

### 不应该看到什么

- 事件列表
- 回调说明
- 菜单说明
- 发布说明

这些都属于后续步骤。

### 用户要做什么

- 复制权限配置
- 点击动态后台入口进入当前应用的权限设置页
- 去飞书后台导入
- 勾选确认
- 点击 `继续`

### 完成条件

- 用户确认已完成权限导入

## 6.5 第 4 步：配置事件订阅

### 目标

让用户单独完成事件订阅，不和回调、菜单、发布混在一起。

### 用户看到什么

- 标题：`配置事件订阅`
- 一句说明：`这一页只处理事件订阅。先保存订阅方式为长连接，再去下一页配置回调订阅方式。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的事件订阅设置页`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「事件与回调」`
- 一段分步骤说明：
  - `进入事件订阅页`
  - `点击「订阅方式」，默认就是长连接，点击保存`
  - `根据下方列表勾选事件`
  - `保存配置后回到这里`
- 一份事件列表卡片：
  - `im.message.receive_v1`
  - `im.message.recalled_v1`
  - `im.message.reaction.created_v1`
  - `application.bot.menu_v6`
  - `card.action.trigger`
  - 每个事件为什么需要

### 页面布局

- 顶部说明
- 后台入口按钮
- 事件列表说明卡
- 底部确认项：`我已经完成事件订阅`

### 不应该看到什么

- 回调长连接说明
- 菜单配置说明
- 发布说明

### 用户要做什么

- 打开动态后台入口
- 按列表勾选事件
- 保存
- 回来确认并继续

### 完成条件

- 用户确认已完成事件订阅

## 6.6 第 5 步：配置回调订阅方式

### 目标

明确告诉用户：

- 当前版本使用飞书长连接处理卡片交互
- 不需要填写 HTTP 回调地址
- 需要在飞书平台的“回调配置”里把回调订阅方式保存为长连接

### 用户看到什么

- 标题：`配置回调订阅方式`
- 一句说明：`当前版本通过飞书长连接接收卡片点击和交互事件，这一步只需要把回调订阅方式保存为长连接。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的事件与回调页面`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「事件与回调」`
- 一段较详细说明：
  - `找到「回调配置」`
  - `点击「回调订阅方式」`
  - `选择「长连接」并点击保存`
  - `确认事件列表里已经包含 card.action.trigger`
  - `保存后，卡片点击、approval request 等交互会直接通过长连接进入当前服务`
  - `保存后回到这里继续`

### 页面布局

- 顶部说明
- 后台入口按钮
- 回调长连接说明卡
- 事件要求说明卡
- 确认项：`我已经完成回调长连接配置`

### 不应该看到什么

- 菜单说明
- 发布说明

### 用户要做什么

- 打开动态后台入口
- 确认当前应用的回调订阅方式使用长连接
- 确认 `card.action.trigger` 已订阅
- 在飞书后台保存
- 回来确认并继续

### 完成条件

- 用户确认已完成回调长连接配置

## 6.7 第 6 步：配置机器人菜单

### 目标

让用户明确配置机器人菜单，并知道哪些按钮事件会被接收和生效。

### 用户看到什么

- 标题：`配置机器人菜单`
- 一句说明：`把需要的机器人菜单按钮加到飞书后台。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的机器人菜单配置页`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「机器人」并进入自定义菜单区域`
- 一份菜单说明区：
  - `列出实例（list）`
  - `当前状态（status）`
  - `切换会话（threads）`
  - `停止当前执行（stop）`
  - `推理 Low / Medium / High / XHigh（reason_low / reason_medium / reason_high / reason_xhigh）`
  - `执行权限 Full / Confirm（access_full / access_confirm）`
  - 每个按钮点击后会触发什么事件
  - 这些按钮里哪些是当前实现真正会处理的
  - 明确强调 `key` 必须与页面展示一致，否则点击后不会生效

### 页面布局

- 顶部说明
- 后台入口按钮
- 菜单定义列表
- 确认项：`我已经完成菜单配置`

### 不应该看到什么

- 事件订阅说明
- 发布说明
- 复杂系统状态

### 用户要做什么

- 打开后台入口
- 按说明创建菜单按钮
- 确认菜单 key 与页面展示一致
- 回来确认并继续

### 完成条件

- 用户确认已完成菜单配置

## 6.8 第 7 步：发布应用并验收

### 目标

要求用户真正发版，并在点击下一步时执行一次“是否生效”的验收检查。

### 用户看到什么

- 标题：`发布应用`
- 一句说明：`前面的权限、事件、回调长连接、菜单都配置完后，还需要在飞书后台发版，变更才会真正生效。`
- 一个根据当前 `App ID` 动态生成的后台入口：
  - `打开该应用的版本管理 / 发布页面`
- 一句辅助说明：
  - `如果没有直接进入目标页，就在左侧菜单点击「版本管理与发布」`
- 一段明确说明：
  - `请在飞书后台完成发版`
  - `发版后点击“检查并继续”`
- 一个主按钮：`检查并继续`

### 页面布局

- 顶部说明
- 发布页入口按钮
- 简短操作指引
- 主按钮 `检查并继续`

### 用户要做什么

- 去飞书后台发版
- 回来点击 `检查并继续`

### 下一步校验规则

点击 `检查并继续` 时，后端要执行一次验收，不通过就拒绝进入下一步。

验收目标至少包括：

- 当前应用权限已生效
- 事件订阅已可见
- 回调长连接配置已完成
- 菜单配置已可见

如果校验失败：

- 停留在本页
- 弹出强确认错误模态
- 明确告诉用户“哪一项还没生效”

### 完成条件

- 验收通过

## 6.9 第 8 步：运行环境检查

### 目标

在进入自动启动和 VS Code 这些后置可选步骤之前，先统一确认“这台机器是否已经满足正常使用 Codex Remote 的基础运行条件”。

### 用户看到什么

- 标题：`运行环境检查`
- 一句结论：
  - `当前机器已满足基础运行条件，可以继续后面的可选配置`
  - 或 `当前机器还不满足基础运行条件，请先修复失败项后再继续`
- 一个检查说明块：
  - 当前 daemon 有没有可用的 headless 启动器
  - wrapper 实际将要启动哪个真实 `codex`
  - 当前服务环境里能不能把它解析成可执行文件
  - 是否存在明显配置风险，例如回指自身或只靠 `PATH` 解析
- 一组结构化检查项：
  - `Headless 启动器`
  - `真实 Codex 二进制`
  - `Wrapper 启动目标`
  - `二进制定位方式`
- 一个主按钮：`检查并继续`

### 页面布局

- 总结 banner
- 当前 `codex-remote` / 真实 `codex` 摘要卡
- 结构化检查项列表
- 当前边界说明

### 不应该看到什么

- 引导用户在页面内安装 Codex CLI
- 引导用户在页面内完成 Codex 登录或 provider 配置
- 把 VS Code 检查和基础运行检查混在一个步骤里

### 用户要做什么

- 先根据检查结果修正本机环境
- 修正完成后点击 `检查并继续`

### 下一步校验规则

点击 `检查并继续` 时，页面必须重新请求服务端检测结果。

如果仍有失败项：

- 停留在本页
- 弹出强确认错误模态
- 明确告诉用户当前还有哪些阻断项

如果只有 warning：

- 允许继续
- 但保留风险提示，例如“当前通过 PATH 解析，后续服务环境变化可能失效”

### 完成条件

- 服务端统一判定基础运行条件已满足

## 6.10 第 9 步：VS Code 集成（可选）

### 目标

把 VS Code 集成作为可选收尾步骤，而不是飞书 setup 主链路的一部分。

### 用户看到什么

- 标题：`配置 VS Code`
- 一句说明：`如果你需要在本机或远程 VS Code 里使用，可以现在完成；也可以稍后再配置。`

当前不再一上来展示内部推荐模式，而是先按实际使用场景引导：

- 如果当前是远程 SSH 机器：
  - 只显示“在这台远程机器上启用 VS Code”
  - 说明这一步只接管扩展入口，不会去写 host 机器的 `settings.json`
- 如果当前不是 SSH 机器：
  - 先问“你以后主要怎么使用 VS Code 里的 Codex”
  - 选项固定为：
    - `只在这台机器本地使用`
    - `主要去别的 SSH 机器上使用`
    - `这台机器本地要用，也会 SSH 到别的机器`
  - 用户选中后，再显示该场景唯一合理的推荐卡和主按钮

### 页面布局

页面主体：

- 当前环境提示
- 使用场景选择卡或推荐卡
- 一个主要按钮
- 一个次级按钮：
  - `稍后在管理页处理`

技术细节例如：

- settings path
- latest bundle
- recorded bundle
- shim reinstall

默认折叠到 `查看技术详情` 里，不默认显示。

### 不应该看到什么

- 一上来就展示所有 detect 字段
- 一上来把 `editor_settings` / `managed_shim` 当成用户主决策

### 用户要做什么

- 先确认自己属于哪种 VS Code 使用场景
- 再执行对应的主路径
- 或点击 `稍后在管理页处理`

### 完成条件

满足其一即可：

- 用户成功完成当前场景对应的接入
- 用户明确选择跳过或改在目标 SSH 机器上处理

## 7. 页面级交互规则

## 7.1 固定底部操作栏

每一步底部固定：

- `上一步`
- `主要操作`
- `次级操作`

避免用户滚动到页面底部找按钮。

补充规则：

- 必做步骤只出现与当前步骤相关的主按钮
- 不出现全局统一的 `跳过`
- `稍后在管理页处理` 只允许出现在第 9 步 `VS Code（可选）`

## 7.2 已完成步骤如何回看

允许用户回到已完成步骤，但默认只通过：

- 点击左侧已完成步骤
- 或 `返回上一步`

进入后仍然一次只看一个步骤。

## 7.3 统一报错交互

setup 不使用 toast 作为主错误承载方式。

统一规则：

- 当前步骤执行失败时，使用模态弹窗
- 模态必须要求用户明确点击确认
- 模态正文必须包含：
  - 一句人话错误说明
  - 当前失败的是哪一步
  - 建议用户下一步做什么
- 技术详情默认折叠，但允许展开

toast 只允许用于：

- 复制成功
- 已保存
- 这种不需要用户决策的轻提示

### 推荐错误模态结构

- 标题：`这一步还没有完成`
- 正文第一段：人话说明
- 正文第二段：建议动作
- 折叠区：技术详情
- 按钮：
  - `我知道了`

## 7.4 恢复上次进度

如果 setup 中断再回来：

- 直接定位到“第一个未完成步骤”
- 左侧把之前完成的步骤标成已完成
- 仍然不展开后续内容

## 8. 推荐文案风格

setup 页文案统一要求：

- 面向动作
- 短句
- 少术语
- 先告诉用户要做什么，再解释为什么

例如：

- 用 `先填写飞书应用信息`
- 不用 `当前 runtime 还没有 configured app`

- 用 `开始验证`
- 不用 `执行网关连通性检测`

- 用 `复制权限配置`
- 不用 `导出 scopes JSON`

- 用 `请先在飞书后台发版，再回来检查`
- 不用 `当前 publish 状态未就绪`

## 9. 与当前实现的主要差异

如果按这份设计重做，和当前 setup 最大的不同是：

1. 删除“当前状态”作为 setup 主页面块
2. setup 不再是长滚动单页内容集合
3. 左侧只保留步骤名，不保留未来步骤正文
4. 当前步骤信息会明显增强，包括后台入口、字段定位说明、操作指引
5. 凭证输入与连接测试合并为一步，改为 `测试并继续`
6. 平台权限、事件订阅、回调长连接配置、菜单配置、发布验收拆成独立页面
7. 发布不是“用户自己点个勾”，而是点击下一步时由系统做验收
8. 发布后新增一页统一的运行环境检查，而不是把判定逻辑散落到自动启动或 VS Code 步骤
9. VS Code 从首页主信息块降级为后置可选步骤
10. setup 错误交互改成强确认模态，而不是 toast

## 10. 推荐实施顺序

如果这份设计确认可行，推荐下一轮实现不要一次性重写全部页面，而是分三段：

1. 先改 setup 页面骨架：
   - 去掉当前状态块
   - 改成“左侧步骤 + 当前步骤正文”
   - 屏蔽未来步骤正文
2. 再重排步骤顺序与内容分组：
   - 创建应用 + 测试
   - 权限
   - 事件
   - 回调长连接配置
   - 菜单
   - 发布验收
   - 运行环境检查
   - VS Code
3. 最后统一补：
   - 动态后台入口
   - 错误模态
   - 文案与技术详情折叠

这样能先把最影响体验的“信息泄露”和“长滚动”问题消掉。
