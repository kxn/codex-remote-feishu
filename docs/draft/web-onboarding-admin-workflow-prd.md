# Web Onboarding And Admin Workflow PRD

> Type: `draft`
> Updated: `2026-04-11`
> Summary: 定义检测驱动的 Web Setup / Admin 工作流，拆分机器 onboarding、bot onboarding 与 admin 运维职责，并重写 setup 完成语义。

## 1. 文档定位

本文是新的 `web setup` / `web admin` 工作流 PRD。

它回答 4 个问题：

- `setup` 到底在解决什么问题
- 什么条件下允许判定 setup 完成
- `admin` 应该承载哪些稳定运维任务
- 当 bot 已在别处配置完成时，页面应该如何基于真实检测结果收口

## 2. 背景

当前体验的主要问题不是功能缺失，而是完成语义和页面职责都混在了一起：

- setup 里“有没有完成”过度依赖本地向导记录，而不是当前真实检测结果
- 用户可能已经在别处把权限、事件、菜单或发布配置好了，但 setup 仍把 bot 视为“未完成”
- 用户如果懒得在当前页面补勾选，也会出现“实际上已经能用，但页面一直说没完成”的状态
- setup 同时承担机器首次开通、首个 bot 接入、机器可选项处理，职责过重
- admin 同时承担日常运维和 bot 首次接入，导致经常从 admin 跳回 setup
- 外部接管的只读 bot 会被误导进入本地可编辑的首次配置流程，形成死流程
- VS Code / autostart 这类可选决策没有持久化，用户刷新后会被反复要求再做一次

结论：

- “有没有在本页面点过继续”不能再作为完成依据
- “至少已有一套凭证”也不能再等价为“setup 完成”
- 真相来源必须改成“当前检测结果 + 当前明确决策”

## 3. 产品目标

新的工作流需要满足：

1. 用户能尽快把“这台机器 + 一个 bot”带到可用状态
2. setup 完成语义以真实检测结果为准，而不是本地打卡记录
3. 系统允许“已可用，但仍有待补全项”的中间稳态
4. admin 不再承担“回到 setup 继续走 bot 首次配置”的职责
5. 外部接管 bot 能以只读但可用的状态正常收口
6. 机器级可选项要能持久化“现在启用 / 稍后处理 / 去别的机器处理”这类决策

## 4. 非目标

本文不要求一次性完成以下事项：

- 一次性把所有平台能力都做到自动检测
- 在第一阶段重写 daemon 全部底层状态结构
- 要求 setup 完成时所有增强能力都已补齐
- 让远程 SSH 场景在 setup 完成后自动转成远程 admin

## 5. 核心定义

### 5.1 机器状态

机器状态只表达“这台机器是否已经完成 onboarding”：

- `blocked`
  - 当前没有任何可用 bot，或机器运行条件不通过
- `usable`
  - 已经存在至少一个可用 bot，可以开始第一次对话
- `usable_with_pending_items`
  - 已经可用，但仍有增强项或建议项待补全
- `completed`
  - 已满足 setup 完成条件，且机器级可选项已经有明确决策

### 5.2 Bot 状态

bot 状态只表达“这个 bot 当前能不能用，以及是否仍有待补全项”：

- `broken`
  - 当前不可用
- `usable`
  - 当前可开始对话
- `usable_with_pending_items`
  - 当前可用，但仍有待补全项
- `fully_verified`
  - 核心能力和当前纳入验证范围的增强能力都已通过
- `external_managed`
  - 来自运行时接管，只读，但当前可用或可检测

### 5.3 检查项状态

每个检查项都必须显式区分：

- `pass`
- `fail`
- `unknown`
- `not_applicable`
- `deferred`

系统判断顺序：

1. 优先相信实时检测结果
2. 其次相信端到端 smoke test
3. 再其次相信当前运行时状态
4. 最后才使用用户手工确认补齐无法自动检测的项目

这意味着：

- 用户没在当前页面点过确认，不等于失败
- 只要系统检测到已配置成功，就应该视为通过

## 6. Setup 完成语义

### 6.1 允许完成 setup 的条件

满足以下条件即可：

1. 至少有一个 bot 处于以下任一状态：
   - `usable`
   - `usable_with_pending_items`
   - `fully_verified`
   - `external_managed`
2. 机器运行环境检查通过
3. 机器级可选项已有明确决策：
   - 现在启用
   - 稍后处理
   - 去别的机器处理

### 6.2 不再阻塞 setup 完成的项目

以下项目如果尚未确认，不应阻塞 setup 完成：

- 用户没有在当前页面补勾选权限导入
- 菜单尚未在本页面确认
- 卡片回调尚未在本页面确认
- Drive 预览权限尚未开通
- VS Code 决定留到管理页或别的机器处理
- autostart 明确选择稍后处理

这些项目应进入：

- `usable_with_pending_items`

而不是继续显示：

- `setup 未完成`

## 7. 新的 Setup 工作流

新的 setup 只负责机器首次 onboarding，收敛成 5 个阶段。

### 7.1 阶段一：环境扫描

页面自动给出当前事实，不让用户先填表：

- 当前是本机还是 SSH 远程机器
- 当前是否检测到可用 bot
- 当前 bot 是否来自外部接管
- 当前机器运行环境是否基本通过

页面动作：

- 没有 bot 时：`开始接入 bot`
- 已有可用 bot 时：`继续使用当前 bot`
- 已有 bot 但不可用时：`修复当前 bot`

### 7.2 阶段二：选择 bot 来源

只保留 3 条清晰路径：

- `新建飞书应用`
- `接入已有飞书应用`
- `使用当前已检测到的 bot`

说明：

- setup 一次只处理一个目标 bot
- 如果当前已有多个 bot，必须显式显示当前目标 bot，而不是静默选第一个
- `?app=<id>` 仍可用作深链，但页面本身必须可见当前目标 bot

### 7.3 阶段三：接入并验证 bot

这一阶段只解决“当前 bot 是否接上了”。

#### 路径 A：新建 bot

- 扫码创建
- 自动拿回 App ID / Secret
- 自动做连接验证

#### 路径 B：接入已有 bot

- 填写 App ID / Secret
- 保存并验证

#### 路径 C：使用当前已检测到的 bot

- 只展示只读信息
- 允许做连接验证
- 不允许进入本地可编辑凭证流程

完成条件：

- 当前 bot 的连接验证通过

### 7.4 阶段四：能力检查

这一步不再拆成固定的“权限 / 事件 / 长连接 / 菜单 / 发布”逐页打卡流。

改为一个统一的能力检查页，展示：

- 当前核心能力是否已经通过
- 当前增强能力是否通过 / 未知 / 待补全
- 每个失败项或未知项对应的补救动作

#### 核心能力

这些能力决定“当前能不能开始用”：

- 凭证连接验证通过
- bot 已进入当前运行时配置并可被实际使用
- 基础消息链路可用
- 机器运行环境检查通过

#### 增强能力

这些能力决定“是不是完整验证通过”，但默认不阻塞 finish：

- 卡片回调
- 机器人菜单
- Drive 预览权限
- 额外事件或审批类能力
- 其他未来接入的扩展能力

能力检查页规则：

- 能自动检测的，系统自动检测
- 不能稳定自动检测的，允许用户显式选择：
  - `我已确认`
  - `稍后处理`

### 7.5 阶段五：机器级选项与完成

这一阶段只处理机器自己的能力，不再混入 bot 平台配置：

- 自动启动
- VS Code 接入

每一项都必须支持持久化的明确决策：

- 现在启用
- 稍后处理
- 当前机器不处理
- 去目标 SSH 机器处理

完成页展示两类结果：

- `已可用`
  - 机器与当前 bot 已可开始使用
- `已可用，但仍有待补全项`
  - 当前可以开始使用，但仍有增强项推荐补齐

## 8. 新的 Admin 工作流

admin 只承担两类工作：

1. 稳态运维
2. 新 bot 的增量接入

### 8.1 Admin 主页结构

建议收敛为：

1. 机器总览
2. bot 列表
3. bot 详情
4. 机器级选项
5. 存储 / 实例 / 技术详情

### 8.2 Admin 中的 bot 状态展示

bot 不再默认显示：

- `首次配置未完成`

改为显示更符合真实状态的标签：

- `可用`
- `可用，但待补全`
- `不可用`
- `外部接管`
- `需关注`

### 8.3 Admin 中的 bot 动作

bot 详情里的主动作改为：

- `测试连接`
- `重新检测能力`
- `补全待处理项`
- `启用 / 停用`
- `删除`

避免继续使用：

- `继续完成首次配置`

因为这个文案会再次把 bot 级问题错误地拉回机器级 setup。

### 8.4 Admin 中新增 bot

在 admin 内直接开启一个独立的 `bot onboarding` 子流程：

1. 选择来源
2. 接入凭证
3. 自动检测能力
4. 返回 bot 详情页

规则：

- 不再跳去 setup
- setup 只负责机器首次 onboarding

## 9. 外部接管 Bot 的专用规则

外部接管 bot 不能再复用本地可编辑 wizard。

### 9.1 页面语义

它的文案应是：

- `该 bot 由外部系统管理`
- `当前页面只能验证和查看状态，不能修改凭证`

而不是：

- `首次配置未完成`

### 9.2 允许的动作

- 测试连接
- 重新检测能力
- 查看待处理项
- 处理机器级选项

### 9.3 不允许的动作

- 修改 App ID / Secret
- 在本地写入 wizard 手工进度
- 进入本地可编辑的首次配置死流程

## 10. 机器级与 Bot 级职责边界

### 10.1 机器级

属于 setup 或 admin 的机器区：

- 运行环境检查
- VS Code
- 自动启动
- 本地存储、实例、诊断

### 10.2 Bot 级

属于 bot onboarding 或 bot 详情页：

- 凭证验证
- 权限 / 事件 / 回调 / 菜单 / 发布相关能力状态
- Drive 预览能力
- 机器人连接状态

规则：

- 机器级完成不等于 bot 完整验证
- bot 可用也不等于所有增强能力都已补齐

## 11. 建议的分阶段落地

### 11.1 第一阶段：状态语义与后端真相源

- 重定义 setup 完成条件
- 明确机器状态与 bot 状态
- 把当前“本地勾选即完成”的字段降级为辅助证据，而不是真相源

### 11.2 第二阶段：Setup 信息架构重写

- 先做环境扫描页
- 把 setup 改成机器 onboarding
- 用能力检查页替代固定手工打卡流

### 11.3 第三阶段：Admin bot 详情与 bot onboarding

- admin 内新增 bot onboarding 子流程
- 去掉“继续完成首次配置 -> 跳回 setup”的主路径
- 在 bot 详情页展示能力矩阵和待处理项

### 11.4 第四阶段：自动检测补强

- 补充更多自动检测项
- 把当前仍依赖手工确认的项目逐步收敛到真实探测
- 为“已在别处配置好”的路径补足可靠证据

## 12. 验收标准

当以下条件全部满足时，可认为本 PRD 已落地：

1. 用户可以在“bot 已在别处配置好”的情况下，仅依赖检测结果完成 setup
2. setup 完成不再要求用户把所有页面逐个点过
3. 外部接管 bot 不会再进入本地可编辑的死流程
4. admin 中新增 bot 不再回跳 setup
5. UI 能区分：
   - `不可用`
   - `可用`
   - `可用，但待补全`
   - `外部接管`
6. VS Code / autostart 等机器级决策刷新后仍然保留
7. 至少一个可用 bot + 机器运行环境通过时，setup 可以正常结束
